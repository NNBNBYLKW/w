# 后续阶段任务大纲（建议版）

## 1. 文档目的

本文档用于基于三类来源，整理一版新的后续阶段任务大纲：

- 当前仓库代码
- `docs/current-project-status-dossier.md`
- 历史产品草案 `docs/windows本地文件管理与素材库产品文档草案.md`

这是一份 **planning-only / 建议版** 文档，不代表当前已实现状态，也不替代后续某个阶段的范围草案或执行版。

本版大纲采用一条更明确的产品推进思路：

- 先优先把 **媒体库 / 游戏库 / 电子书库 / 软件库** 等垂类库功能做出来
- 先形成一个可用于外部验证的 **测试版**
- 稳定性、性能和更完整的收边界工作，放到测试版之后集中推进

同时，本文档仍然以当前代码为起点，不把历史产品草案里的未实现能力写成当前事实。

## 2. 当前状态摘要

当前项目已经从单一通用文件浏览，扩展成一个 local-first 的 indexed-files workbench。当前代码已经验证了以下主链：

`source setup -> scan/index -> search/browse/retrieve -> shared details -> tags/color tags/collections -> open actions`

当前已经成立的主要 surfaces 包括：

- `Home`
- `Onboarding`
- `Search`
- `Files`
- `Media`
- `Books`
- `Software`
- `Recent`
- `Tags`
- `Collections`
- `Settings`

当前也已经存在三个 subset routes：

- `GET /library/media`
- `GET /library/books`
- `GET /library/software`

这说明项目当前已经验证了几条重要方法论：

- 以 `files` 为统一索引对象
- 以 subset surfaces 扩展垂类入口
- 以 shared `DetailsPanelFeature` 复用详情与动作
- 以 tags、color tags、collections、Search 作为跨 surface 的组织与检索链

但如果以历史产品草案作为长线目标来看，当前还没有完成的关键部分仍然很多，尤其是：

- `游戏库` 还未落地
- `媒体库` 仍偏基础 listing / thumbnail surface
- `Books` 与 `Software` 仍是较克制的 subset v1，而不是更完整的“库式体验”
- 收藏 / 状态 / 评分 / 批量整理 / 智能集合 / 最近使用 / 最近标记等能力还未形成草案所描述的完整产品层

因此，后续阶段若要朝“先出一个功能更齐的测试版”推进，最合理的方式不是先做稳定性总收口，而是先把垂类库线与相关横向组织能力补齐到能验证产品价值的程度。

## 3. 规划原则

### 3.1 当前代码是起点，历史草案是长线目标

- 当前代码定义“今天从哪里开始”
- current-state dossier 定义“当前已经成立的方法论”
- 历史产品草案定义“长线目标要覆盖哪些能力族”

这意味着后续阶段的规划应是：

- **从当前代码出发**
- **朝历史产品草案的能力范围推进**
- **但不把草案内容提前写成已实现**

### 3.2 优先做垂类库功能，稳定性后置

本版大纲接受这样一种产品推进顺序：

- 先把用户可感知的垂类库功能做出来
- 先形成可验证的测试版
- 再做更系统的稳定性、性能、硬化与收口

这不是说稳定性不重要，而是把它从“前置门槛”改成“测试版后期与正式版前的关键阶段”。

### 3.3 继续沿用已验证的方法论

即便后续范围会更大，也仍然优先延续当前已验证的方法论：

- indexed files 为底座
- subset surfaces 为垂类入口
- shared details 为共同详情与动作面
- Search / tags / color tags / collections 为复用组织层

不建议为了追求草案中的全部愿景，过早把系统改造成：

- 阅读平台
- 安装管理器 / 启动器平台
- 重度外部 metadata 平台
- 统一重对象模型平台

### 3.4 阶段仍需有边界

即使采用“功能齐全测试版优先”的策略，阶段也必须可收边界：

- 每个阶段只聚焦一个核心主题
- 每个阶段都要说明非目标
- 每个阶段都要说明它为何排在当前顺序上

## 4. 长线目标如何理解

如果以历史产品草案为目标，本项目后续不应只停留在“通用文件管理增强”，而应逐步补齐以下几类能力：

### 4.1 垂类库线

- 素材库（图片 / 视频）
- 游戏库
- 电子书库
- 软件库

### 4.2 横向组织层

- 普通标签
- 颜色标签
- 收藏 / 状态 / 评分
- 智能集合 / 保存搜索条件
- 最近导入 / 最近使用 / 最近标记

### 4.3 库式浏览体验

- 缩略图 / 封面 / 卡片流
- 更好的筛选与排序
- 更贴近内容浏览的首页与入口组织
- shared details 之上的更强信息密度

### 4.4 当前不应直接照搬的草案方向

历史草案中有一些更远的方向，但不建议作为近期阶段主线直接展开：

- 阅读器化
- 启动器平台化
- 下载器 / 商店化
- AI 自动标签 / OCR / 语义识别
- 过重的统一对象模型中台

## 5. 推荐的阶段顺序

本版建议把后续推进拆成六个阶段：

### 阶段一：媒体库增强版

先把 `Media` 从基础 subset surface 拉到更接近“素材库测试版”的程度。

### 阶段二：游戏库首版

补齐历史草案中最关键、当前又完全缺失的垂类入口：`游戏库`。

### 阶段三：电子书库 / 软件库增强版

在已有 `Books` / `Software` 基础上，补足更接近草案目标的库式体验。

### 阶段四：组织层与库体验补齐

把收藏 / 状态 / 评分 / 智能集合 / 批量整理 / 最近链路等横向能力补起来。

### 阶段五：测试版收口与外部验证

把前面阶段的垂类与横向能力整合成一个“功能足够完整、可投入测试市场验证”的测试版。

### 阶段六：稳定性 / 性能 / 剩余草案功能收口

在测试版反馈基础上，再系统推进稳定性、性能与长尾能力补齐，朝历史产品草案中的完整产品形态靠近。

## 6. 每个推荐阶段的任务大纲

### 6.1 阶段一：媒体库增强版

**阶段目标**

把当前 `Media` 从“image/video subset listing”增强为一个真正可用于找素材、看素材、筛素材的媒体库测试版。

**主要任务簇**

- 强化缩略图 / 卡片流 / 视觉浏览体验
- 补强图片 / 视频的预览与基础信息展示
- 让 tags / color tags / filters 在媒体库里成为真正高频入口
- 让最近导入到素材整理的链路更自然
- 提高媒体库与 shared details 的协同强度

**该阶段应优先补的能力**

- 更完整的媒体库卡片流
- 更明显的标签与颜色标签筛选
- 更稳的预览体验
- 更适合内容浏览的排序 / 浏览方式

**明确不做什么**

- 图片编辑器
- 视频编辑器
- 专业 DAM 平台
- AI 自动打标签

**为什么排第一**

- 素材库是历史草案中最核心、也最适合先形成产品差异化的垂类
- 当前代码已经有 `Media` 基础，增强成本低于从零开始新建游戏库
- 它最容易尽快形成“可被市场感知”的测试版亮点

**进入前提**

- 保持当前 `files` 底座不变
- 保持 shared details 与 tags / collections 复用链不变

**建议拆分批次**

- 批次 1：媒体库浏览密度增强
  - 强化卡片流、缩略图优先级、空状态与加载状态
  - 让“看图找素材 / 看视频封面找素材”成为主路径，而不是回退到只看文件名
- 批次 2：媒体筛选与预览增强
  - 提升类型、标签、颜色标签、时间等筛选入口的可见度
  - 补齐媒体卡片与 shared details 之间的预览协同
- 批次 3：媒体整理闭环增强
  - 强化媒体库与最近导入、Tags、Collections 的回流关系
  - 让用户能从“刚扫描进来”快速进入“筛选、标记、再找到”

**本阶段结束时，用户至少应能**

- 在媒体库里主要依靠缩略图 / 卡片而不是文件名找图片和视频
- 在媒体库里直接使用标签、颜色标签与基础筛选缩小范围
- 在媒体库与 shared details 之间完成预览、标记、打开动作的闭环
- 明显感知到“素材库”已经不是通用文件页的轻包装

**建议退出条件**

- `Media` 已经具备清晰的“媒体库测试版”气质
- 媒体相关的浏览、筛选、详情、整理四条链能自然连通
- 不依赖新增重对象模型，也不需要引入专业媒体平台能力

**下一份应补文档**

- `媒体库增强版（范围草案）`
- 在范围草案通过后，再拆对应执行版

### 6.2 阶段二：游戏库首版

**阶段目标**

补齐历史产品草案中的 `游戏库`，形成一个轻量但可验证价值的游戏库首版。

**主要任务簇**

- 新增 `Game Library` surface
- 建立最小识别与入库规则
- 支持封面 / 图标 / 标题优先浏览
- 支持启动入口或打开对应路径
- 支持“在玩 / 通关 / 搁置”等状态型组织方式
- 继续复用 shared details、Search、tags、collections

**该阶段应优先补的能力**

- 统一游戏库页
- 基础封面 / 图标浏览
- 启动或打开入口
- 状态标签体系

**明确不做什么**

- 完整 launcher 平台
- 游玩时长
- 成就系统
- 平台账户同步
- 下载器 / 商店

**为什么排第二**

- 历史草案里游戏库是很重要的垂类，但当前代码中仍完全缺失
- 若先补它，能更快把产品从“通用文件工作台”推进到“多垂类本地资产库”
- 在媒体库之后推进，能复用更多库式浏览与卡片经验

**进入前提**

- 阶段一已把卡片流 / 库式浏览基础做得更稳
- 先明确 `游戏库` 仍然是 indexed-files subset，而不是 launcher 平台

**建议拆分批次**

- 批次 1：游戏库范围定义与识别基线
  - 明确游戏库识别规则、支持文件形态、最小展示字段
  - 明确哪些本地游戏内容先纳入，哪些内容后置
- 批次 2：游戏库页面与库式浏览基线
  - 新增游戏库页
  - 补齐封面 / 图标 / 标题优先浏览
  - 建立与 shared details 的自然联动
- 批次 3：启动与状态整理闭环
  - 建立最小启动入口或打开入口
  - 引入“在玩 / 通关 / 搁置”这类状态型组织方式
  - 与 tags / collections / Search 形成回流闭环

**本阶段结束时，用户至少应能**

- 在统一游戏库页中浏览本地游戏条目
- 依靠封面 / 图标 / 标题，而不是路径记忆，重新定位游戏
- 对游戏进行轻量状态标记并再次找回
- 执行最小打开 / 启动动作，而不依赖复杂 launcher 平台

**建议退出条件**

- 游戏库已经是一个真实可用的入口，而不是占位页
- 用户能完成“入库 -> 浏览 -> 标记 -> 再找到 -> 打开”的基本闭环
- 范围仍停留在轻量本地游戏库，不滑向平台账户、时长、成就体系

**下一份应补文档**

- `游戏库首版（范围草案）`
- 该草案需先定义 subset 识别规则与边界，再进入执行版

### 6.3 阶段三：电子书库 / 软件库增强版

**阶段目标**

把当前已经存在的 `Books` / `Software` 从轻量 subset surface，增强到更接近历史草案中“轻量库式管理”的测试版状态。

#### 3A. 电子书库增强

**主要任务簇**

- 从当前列表式 `Books` 逐步增强为更接近书库的浏览体验
- 补强封面、标题、作者、系列等核心展示字段
- 补强标签化浏览与更自然的检索入口
- 在不转向阅读器的前提下提升书库感

**建议拆分批次**

- 批次 A：书库展示密度增强
  - 从当前书籍子集列表逐步增强到更自然的书库浏览
  - 明确哪些书籍信息可稳定提取，哪些需要保守回退
- 批次 B：书籍组织入口增强
  - 让标签、作者、系列、格式等入口在书库中更自然
  - 让 Books 与 Search / Collections 的关系更清晰
- 批次 C：书库闭环收口
  - 强化详情、打开、归类、再次找到的路径
  - 保持“书库入口”而不是“阅读平台”定位

**本阶段结束时，用户至少应能**

- 更像浏览书库而不是浏览扩展名列表
- 通过封面 / 标题 / 作者 / 标签更自然地定位书籍
- 继续使用 shared details、Search、Collections 完成组织与取回

**明确不做**

- 阅读器
- 阅读进度系统
- 电子书制作或批注系统

#### 3B. 软件库增强

**主要任务簇**

- 从当前列表式 `Software` 逐步增强为更接近软件库的浏览体验
- 补强图标 / 封面 / 标题展示
- 补强快速启动 / 打开目录的工作流
- 支持便携软件的轻量库式管理

**建议拆分批次**

- 批次 A：软件库展示密度增强
  - 让 software-related files 更像“库条目”而不是纯文件记录
  - 明确哪些图标 / 展示字段可稳定使用
- 批次 B：软件动作与组织增强
  - 强化快速启动 / 打开目录 / 再次找回的流程
  - 让状态、标签、collections 在软件库里更自然
- 批次 C：软件库边界收口
  - 明确软件库是便携软件与相关文件的库式入口
  - 保持与安装管理器 / 包管理器的边界

**本阶段结束时，用户至少应能**

- 在软件库中集中浏览便携软件和相关文件
- 使用更自然的标题 / 图标 / 操作入口定位内容
- 完成轻量启动、归类、再找到的闭环

**明确不做**

- 安装管理器
- 卸载流程
- 包管理器
- 平台化软件库存中心

**为什么排第三**

- `Books` / `Software` 已经有 subset v1 起点，适合在媒体库与游戏库之后做“第二层增强”
- 这样能让测试版在垂类覆盖面上更完整
- 同时避免在媒体库和游戏库都未站稳前，把 Books / Software 做得过深

**进入前提**

- 已确认测试版目标是“轻量库式管理”，不是阅读平台或安装平台
- shared details、Search、tags、collections 仍是共同底座

**建议退出条件**

- `Books` 与 `Software` 都明显超出当前 subset v1 水平
- 它们已经形成轻量库式体验，但仍没有转向独立平台
- 用户能感知到“电子书库 / 软件库”与“媒体库 / 游戏库”处于同一产品体系下

**下一份应补文档**

- `电子书库增强版（范围草案）`
- `软件库增强版（范围草案）`
- 两者可独立立项，但建议在同一大阶段中连续推进

### 6.4 阶段四：组织层与库体验补齐

**阶段目标**

把目前已经存在但仍偏薄的组织层，扩展成能够真正支撑多垂类库体验的横向能力层。

**主要任务簇**

- 批量标签 / 批量颜色标签
- 收藏 / 状态 / 评分
- 智能集合 / 保存搜索条件增强
- 最近导入 / 最近使用 / 最近标记链路补齐
- 标签入口、集合入口、最近入口的角色增强
- 跨 surface 的结果回流与整理闭环

**该阶段应优先补的能力**

- 批量整理能力
- 收藏 / 状态 / 评分这一层的轻量元数据
- 更成熟的 collections / smart collections
- 更完整的 recent family

**明确不做什么**

- 复杂自动化规则引擎
- AI 自动标签
- 过重后台化运营系统

**为什么排第四**

- 垂类库线先建立后，这一层的价值会更明显
- 如果过早先做横向层，容易变成组织系统先行但垂类体验仍薄
- 放在这里更符合“先把垂类功能做出来，再补共通层”的测试版策略

**进入前提**

- 媒体库、游戏库、电子书库、软件库已经形成可用雏形
- 团队已接受“当前测试版需要更强的整理闭环”

**建议拆分批次**

- 批次 1：批量整理能力
  - 批量标签
  - 批量颜色标签
  - 与最近导入 / 各库页的连续整理节奏打通
- 批次 2：状态型元数据层
  - 收藏
  - 状态
  - 评分
  - 明确它们在不同库页中的展示与筛选位置
- 批次 3：智能集合与 recent family 增强
  - 保存搜索条件
  - 智能集合增强
  - 最近使用 / 最近标记 / 最近导入关系补齐

**本阶段结束时，用户至少应能**

- 对一批新入库内容进行连续整理，而不是逐个处理
- 用收藏 / 状态 / 评分表达轻量管理语义
- 用智能集合和 recent family 更稳定地重新找到内容
- 在不同库页之间获得更一致的整理逻辑

**建议退出条件**

- 组织层不再只是“有标签功能”，而是成为多垂类库的共通支撑层
- 用户能从任意一个库页回到 tags / collections / recent，再次找到内容
- 没有滑向复杂自动化平台或后台化运营系统

**下一份应补文档**

- `组织层与库体验补齐（范围草案）`

### 6.5 阶段五：测试版收口与外部验证

**阶段目标**

将前四个阶段形成的能力整合成一个“功能足够完整、可投入市场验证”的测试版。

**主要任务簇**

- 首页 / 最近 / 库入口的整合
- 关键页面空状态、加载状态、关键操作路径统一
- 测试版范围说明与验收口径
- 形成可对外演示与验证的版本节奏
- 收集真实用户对库式浏览、标签整理、垂类入口的反馈

**测试版至少应具备的能力**

- 素材库可用
- 游戏库首版可用
- 电子书库与软件库达到轻量库式管理水平
- Search / Files / Tags / Collections 仍可复用
- shared details 与 open actions 流程稳定可感知
- 最近链路与组织动作足以支撑持续整理

**明确不做什么**

- 为了测试版而重写底层架构
- 为了测试版一次性补齐所有长尾能力

**为什么排第五**

- 测试版应在垂类体验与组织闭环都具备基本说服力之后再推出
- 太早推出会得到“方向不清 / 页面多但都偏薄”的反馈

**进入前提**

- 前四个阶段都已形成可用 current-state slice
- 有一套可对外解释的测试版定位

**建议拆分批次**

- 批次 1：测试版范围冻结
  - 明确哪些能力属于测试版必须项
  - 明确哪些能力属于后置项
- 批次 2：测试版体验统一
  - 首页入口、空状态、加载状态、详情侧栏、关键动作路径统一
  - 对外演示路径与内部验收路径统一
- 批次 3：外部验证准备
  - 测试版说明
  - 关键验证问题
  - 反馈收集与优先级归并机制

**本阶段结束时，用户至少应能**

- 在一个相对完整的产品里体验媒体库、游戏库、电子书库、软件库与横向组织层
- 明确感知这不是单一文件页工具，而是本地资产工作台测试版
- 沿着几条核心路径连续完成浏览、整理、再找到与打开动作

**建议退出条件**

- 已有清晰的测试版边界，而不是“什么都想塞进去”
- 已经能拿出去验证“垂类库 + 组织层 + shared details”这套产品组合
- 已经准备好接真实用户反馈，而不是继续只在内部抽象讨论

**下一份应补文档**

- `测试版范围说明与验收草案`
- `测试版验证问题清单`

### 6.6 阶段六：稳定性 / 性能 / 剩余草案功能收口

**阶段目标**

在测试版验证之后，再集中推进稳定性、性能、长尾功能与剩余草案目标的收口。

**主要任务簇**

- scan / task / runtime hardening
- 缩略图 / 预览 / metadata 提取稳定性提升
- 大库规模性能与滚动 / 搜索响应优化
- 根据测试版反馈补齐高价值长尾能力
- 把历史产品草案中仍缺失、但经验证确有价值的功能继续纳入

**该阶段可能覆盖的能力**

- 更完整的元数据提取
- 更成熟的封面 / 预览缓存体系
- 更稳的大规模库浏览性能
- 更完整的 recent / collections / status 细化
- 针对 Books / Software / Games / Media 的收口优化

**明确不做什么**

- 因为收口而突然转向另一种产品
- 在未验证价值时强上高风险平台能力

**为什么放在最后**

- 本版策略明确是“先做出能验证价值的测试版，再系统优化”
- 稳定性和性能在正式进入更广用户面前必须加强，但不再作为本阶段序列的第一阻塞项

**进入前提**

- 阶段五已产出可以收集真实反馈的测试版
- 已能区分哪些问题是高优先级稳定性问题，哪些只是阶段性不完美

**建议拆分批次**

- 批次 1：主链稳定性与运行时加固
  - scan / task / source feedback hardening
  - shared details、open actions、库页关键交互稳定性提升
- 批次 2：性能与大库体验优化
  - 搜索响应
  - 滚动与卡片流性能
  - 缩略图 / 封面 / 预览缓存策略优化
- 批次 3：草案长尾功能收口
  - 根据测试版反馈补齐高价值剩余功能
  - 持续推进仍有必要的元数据、库体验与组织层细化

**本阶段结束时，用户至少应能**

- 在更大规模本地库里稳定使用核心功能
- 感知到搜索、浏览、筛选、预览、打开等路径比测试版明显更稳
- 使用到更多真正被验证过价值的草案能力，而不是抽象愿景

**建议退出条件**

- 高优先级稳定性与性能问题被系统压下
- 测试版中已验证有价值的草案能力得到正式收口
- 当前产品已更接近历史产品草案中的“完整功能实现”状态

**下一份应补文档**

- `稳定性 / 性能 / 正式版收口范围草案`

## 7. 当前不建议插入主线的方向

即便目标是逐步靠近历史产品草案，也不建议在当前主线中提前插入这些方向：

### 7.1 阅读平台化

会把电子书库从“本地书籍库入口”拉成独立阅读平台，范围膨胀过快。

### 7.2 安装管理器 / 启动器平台化

会把软件库和游戏库从“库式入口”拉成系统级管理中心，偏离当前代码已验证的方法论。

### 7.3 重外部 metadata 平台

会让系统从 local-first 走向外部数据依赖，工程复杂度和维护成本都显著提高。

### 7.4 AI 自动标签 / OCR / 语义识别

这是高风险高范围方向，不适合作为当前“先做出测试版”的主线。

### 7.5 统一重对象模型中台

当前代码已经验证的是 `files + subset surfaces + shared details`，而不是 `library_items` 这类更重中台抽象。

## 8. 最终推荐结论

如果目标从“当前 subset workbench 继续小步增强”，切换为“尽快做出一版更接近历史产品草案的功能齐全测试版”，那么后续阶段最合理的主线应是：

1. **先把垂类库功能补齐**
   - 媒体库先增强
   - 再补游戏库
   - 再增强电子书库与软件库
2. **再补横向组织层**
   - 批量整理
   - 收藏 / 状态 / 评分
   - 智能集合
   - 最近链路
3. **形成测试版**
   - 用真实用户反馈验证产品方向
4. **最后系统做稳定性 / 性能 / 草案长尾能力收口**

因此，如果要继续往下立项，最建议的下一步不是先写“稳定性加固执行版”，而是先补一份围绕 **媒体库增强版** 的范围草案文档，并把它作为测试版垂类主线的第一阶段。

## 9. 历史产品草案功能覆盖映射

为了避免后续阶段推进时逐渐偏离历史产品草案，下面给出一版“草案功能 -> 推荐阶段”的覆盖映射。它不是当前事实清单，而是后续阶段应逐步覆盖的目标图。

### 9.1 已有底座

以下能力已经由当前代码提供基础底座：

- source onboarding / scan
- general Search / Files
- tags / color tags
- shared details
- Media / Books / Software subset surfaces
- Collections
- Recent imports baseline
- desktop open actions

### 9.2 需要在阶段一到阶段四重点补齐的草案能力

#### 素材库相关

- 更强的缩略图浏览
- 更自然的卡片流 / 视觉浏览
- 更强的标签与颜色标签筛选
- 更自然的预览与整理闭环

对应阶段：

- 主要在 **阶段一**
- 长尾收口在 **阶段六**

#### 游戏库相关

- 统一游戏库页
- 封面 / 图标 / 标题浏览
- 最小启动入口
- 状态型组织方式

对应阶段：

- 主要在 **阶段二**
- 长尾收口在 **阶段六**

#### 电子书库相关

- 书库浏览体验
- 封面 / 标题 / 作者 / 系列信息
- 标签化浏览

对应阶段：

- 长尾收口在 **阶段六**

#### 软件库相关

- 软件库浏览体验
- 图标 / 标题 / 操作入口

- 主要在 **阶段三**
- 长尾收口在 **阶段六**

### 9.3 测试版与完整目标之间的关系

本大纲中的 **阶段五** 不是草案终点，而是一个关键的中间状态：

- 它的目标是形成“可验证的测试版”
- 不是一次性完成草案中的全部长尾功能

真正朝“历史产品草案中要求的全部功能实现”靠近，仍需要在 **阶段六** 中根据测试版反馈逐步收口。

### 9.4 当前不建议纳入“全部功能实现”主线的草案外延

即便目标是尽量靠近历史产品草案，也仍不建议把这些方向定义成主线必要项：

- 阅读平台化
- 启动器 / 商店平台化
- AI 自动标签 / OCR / 语义识别
- 统一重对象模型平台

原因是这些方向会改变产品方法论，而不是单纯把草案中的库式工作台能力做完整。
