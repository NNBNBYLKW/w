# 稳定性 / 性能 / 正式版收口范围草案 v0.1

## 1. 文档目的

本文档用于定义测试版之后的 **稳定性 / 性能 / 正式版收口阶段** 范围。

本阶段不是重新定义产品，也不是引入新的重平台方向，而是在前序阶段已经完成以下能力的前提下：

- 媒体库增强版
- 游戏库首版
- 电子书库增强版
- 软件库增强版
- 组织层与库体验补齐
- 测试版范围冻结与外部验证

对当前产品进行系统性的硬化、性能优化、交互收边与正式版范围收口。

本阶段的核心目标是：

> 在不改变当前 `files + subset surfaces + shared details + tags/color tags/collections` 方法论的前提下，
> 把测试版推进到更稳、更快、更可预测、更适合正式发布的状态。

---

## 2. 当前前提

进入本阶段前，默认以下前提已经成立：

1. 当前产品已经不是单一文件浏览器，而是本地资产工作台。
2. 主要垂类入口已经存在：Media / Games / Books / Software。
3. 横向组织层已经达到可用水平：tags、color tags、collections、recent family、状态型元数据、批量整理。
4. 测试版已经对外或对目标用户群做过验证，已经收集到一轮可归类反馈。
5. 当前产品主链仍保持为：
   `source setup -> scan/index -> search/browse/retrieve -> shared details -> tags/color tags/collections -> open actions`

本阶段默认不推翻以上前提。

---

## 3. 本阶段定位

本阶段是：

- 稳定性加固阶段
- 性能优化阶段
- 交互与状态收边阶段
- 正式版范围冻结阶段

本阶段不是：

- 新产品方向探索阶段
- 新重功能爆发阶段
- 平台化改造阶段
- AI / OCR / 语义识别主线阶段

换句话说，本阶段重点不是“还能再加什么炫功能”，而是：

- 现有能力是否足够稳
- 现有链路是否足够快
- 现有页面是否足够统一
- 哪些测试版问题必须在正式版前压下去

---

## 4. 本阶段核心目标

### 4.1 主链稳定性加固
确保以下关键链路在真实使用中具备高可用性：

- source 添加 / 编辑 / 删除 / 扫描
- scan / index / delete-sync 主链
- Search / Files / Media / Games / Books / Software 主页面读取
- shared details 详情加载与动作执行
- tags / color tags / collections / recent family 组织链
- open file / open containing folder 桌面动作

### 4.2 大库规模性能优化
针对更大体量本地内容，系统优化：

- 搜索响应
- 列表 / 卡片滚动体验
- 缩略图 / 封面 / 预览加载速度
- 详情切换延迟
- 大量筛选条件下的结果刷新速度

### 4.3 状态与交互收边
统一和收口：

- 空状态
- 无结果状态
- 加载状态
- 错误状态
- 占位与降级状态
- 关键 Toast / ConfirmDialog / Inline error 提示文案

### 4.4 正式版范围冻结
根据测试版反馈，把“必须进入正式版”和“后置能力”明确切开，避免范围继续膨胀。

---

## 5. 本阶段不做什么

本阶段明确不做：

- 阅读平台化
- 游戏/软件启动器平台化
- 重外部 metadata 平台
- AI 自动标签
- OCR / 语义识别 / 相似检索
- 为未来抽象而重写底层对象模型
- 脱离 `files` 主语义的新事实层
- 因为正式版收口而突然引入复杂任务系统、复杂多进程流水线或云同步体系

本阶段允许补长尾能力，但前提是它们必须来自测试版反馈验证，且不会改变当前产品方法论。

---

## 6. 本阶段工作簇

本阶段建议分为 5 个工作簇推进。

---

## 7. 工作簇 A：扫描 / 索引 / 运行时稳定性加固

### 7.1 目标
提升底座可靠性，降低“系统看起来在跑，但链路偶发失效”的问题。

### 7.2 包含内容
- source 管理流程的错误处理与反馈收边
- scan 任务开始、进行、完成、失败状态的稳定表达
- 重扫与增量同步后的结果一致性检查
- deleted / missing file 场景的索引状态一致性增强
- 桌面壳与 backend base URL / runtime readiness 相关问题加固
- 与 task runtime 相关的最小状态回显增强

### 7.3 验收口径
- 常规 source 操作不应出现“成功但页面未同步”的含混状态
- scan 结果状态可预测，失败可解释，重试路径清晰
- 文件删除 / 移动 / 重扫后的索引结果不应表现出明显脏数据感

---

## 8. 工作簇 B：搜索 / 浏览 / 详情性能优化

### 8.1 目标
让用户在真实大库中感知到“比测试版更快、更顺”。

### 8.2 包含内容
- 搜索请求性能优化
- 列表页与卡片页的分页 / 懒加载 / 虚拟滚动优化
- 详情切换性能优化
- 结果刷新时的占位与局部刷新策略优化
- 文件列表、媒体库、游戏库、书库、软件库的滚动稳定性优化

### 8.3 重点关注页面
- `/search`
- `/files`
- `/library/media`
- `/library/games`
- `/library/books`
- `/library/software`

### 8.4 验收口径
- 大量结果下滚动明显比测试版更稳定
- 搜索、筛选、排序切换的响应体感得到改善
- 详情切换不出现明显卡顿和频繁闪烁

---

## 9. 工作簇 C：缩略图 / 封面 / 预览缓存体系优化

### 9.1 目标
提高视觉型库页的一致性与加载质量。

### 9.2 包含内容
- 缩略图缓存命中率优化
- 首屏缩略图加载策略优化
- 失败回退与占位图策略统一
- Books / Games / Software 的封面 / 图标 / 预览回退策略统一
- 详情页预览与列表卡片的协同一致性优化

### 9.3 不包含内容
- 专业媒体转码流水线
- 视频深度预处理
- 大规模外部封面抓取平台

### 9.4 验收口径
- 缩略图 / 封面失败时不再显得像系统坏掉
- 首屏视觉体验稳定
- 不同库页对“有图 / 无图 / 失败回退”的处理风格统一

---

## 10. 工作簇 D：状态、文案与交互统一收边

### 10.1 目标
把测试版中“功能有了但体验还散”的部分收口。

### 10.2 包含内容
- 空状态统一
- 无结果状态统一
- 加载骨架统一
- 错误提示统一
- Toast 文案统一
- ConfirmDialog 风格与触发条件统一
- RightDetailsPanel 在各页面中的展开 / 收起 / 占位 / 加载行为统一
- 关键按钮与入口命名统一

### 10.3 重点关注
- Search / Media / Games / Books / Software / Recent / Tags / Collections
- Settings / Onboarding / Home 中的关键状态提示

### 10.4 验收口径
- 页面之间不再像拼接产品
- 用户在不同页面里对选中、打开、标记、失败、重试的心智一致
- 正式版可演示路径中的文案、状态、动作表达统一

---

## 11. 工作簇 E：正式版范围冻结与高价值长尾补齐

### 11.1 目标
将测试版反馈系统归并后，明确正式版必须项与后置项。

### 11.2 包含内容
- 测试版问题归类：P0 / P1 / P2
- 正式版必须修复项清单
- 正式版必须补齐项清单
- 明确后置项清单
- 对经验证确实高价值的长尾能力做有限补齐

### 11.3 可进入正式版收口的长尾项示例
- 某些高频筛选器缺失
- 某些关键列表字段缺失
- 某个库页的高频操作过深
- 某些 recently / status / collections 回流路径不自然

### 11.4 不应在本工作簇中临时引入的方向
- 新产品线
- 新平台型能力
- 重任务体系
- AI 主线

### 11.5 验收口径
- 正式版边界清晰
- 团队知道“哪些一定做、哪些不进正式版”
- 不会因为收口阶段再次失控加功能

---

- `/library/books`

- SoftwareLibraryFeature

---

## 13. 给 Codex 的执行约束

Codex 在本阶段执行时应遵守：

1. 不重写当前底层方法论。
2. 不新建脱离 `files` 的重对象事实层。
3. 不把稳定性阶段变成新功能膨胀阶段。
4. 优先处理测试版反馈中高频、可复现、影响主链的问题。
5. 优先收口 shared details、搜索、列表/卡片浏览、状态反馈的一致性。
6. 只有经验证高价值的长尾能力，才允许进入本阶段。

---

## 14. 进入条件

进入本阶段前，建议至少满足：

- 测试版能力已经形成一个完整可演示闭环
- 已收集到一轮真实使用反馈
- 已能识别出高优先级稳定性问题、高优先级性能问题与正式版必须项

---

## 15. 退出条件

本阶段结束时，至少应满足：

1. 高优先级主链稳定性问题已显著下降。
2. 搜索、浏览、详情、打开动作在大库场景中体感明显优于测试版。
3. 主要页面的空/加载/错误/无结果状态已统一。
4. 缩略图 / 封面 / 预览相关体验稳定。
5. 正式版必须项与后置项已冻结。
6. 当前产品已具备“正式发布前最后一轮收边”的状态，而不是仍处于方向探索期。

---

## 16. 本阶段后的自然下一步

本阶段完成后，才适合进入：

> 正式版发布准备 / 发布节奏文档

而不是继续无限追加下一轮大功能规划。

---

## 17. 当前待你拍板的 6 个点

1. 正式版前允许补齐的“长尾能力”上限是多少。
2. 稳定性与性能问题的优先级裁决规则由谁拍板。
3. 是否需要单独列一份“测试版高优反馈归并表”。
4. 是否在本阶段加入更明确的性能指标门槛。
5. 缩略图 / 封面缓存优化是否允许调整现有缓存策略。
6. 本阶段是否拆成多个 Codex 子任务，还是整阶段一次性执行。
