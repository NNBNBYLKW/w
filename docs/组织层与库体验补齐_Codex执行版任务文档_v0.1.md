# 组织层与库体验补齐（Codex 执行版任务文档）v0.1

## 1. 文档目的

本文档用于指导 Codex 在当前项目中实施“组织层与库体验补齐”阶段的开发工作。

本阶段不是新增新的垂类库，而是在当前已经存在并逐步增强中的 Media / Games / Books / Software 之上，补齐横向复用的组织能力，使系统从“能浏览、能打开”进一步发展为“能持续整理、能稳定再找回”的本地资产工作台。

本文档是 **执行版任务文档**，默认前提如下：

- 当前系统仍以 `files` 为统一索引对象
- 当前系统仍沿用 subset surfaces 扩展垂类入口
- 当前系统仍以 shared details 为共同详情与动作面
- 当前系统仍以 Search / tags / color tags / collections 为横向组织层基础
- 本阶段不重写底层事实层，不引入平台化对象中台

---

## 2. 本阶段目标

本阶段的目标不是继续做新的库页面，而是把当前偏薄的横向组织能力补到“真实可高频使用”的程度。

本阶段完成后，系统至少应达到以下状态：

1. 用户可以对一批内容进行连续整理，而不必逐个低效处理
2. 用户可以通过收藏 / 状态 / 评分表达更细的轻量管理语义
3. 用户可以保存和复用一组检索条件，而不必每次重建同样筛选
4. 用户可以从 Recent / Tags / Collections / 各库页之间自然回流
5. 不同 surface 的整理体验保持一致，而不是各自发展成独立系统

---

## 3. 本阶段非目标

Codex 在本阶段**不得**主动扩展以下方向：

- 不做复杂自动化规则引擎
- 不做 AI 自动标签 / OCR / 语义识别
- 不做后台化运营系统
- 不做独立的“管理中心”式重页面
- 不引入新的正式主语义对象替代 `files`
- 不把某个垂类库单独发展出私有组织体系
- 不重写 Search / Details / Tagging 的既有方法论

如果某个实现思路会把系统推向上述方向，应主动收缩为更克制的方案。

---

## 4. 本阶段执行原则

### 4.1 继续复用现有骨架

必须继续建立在以下现有能力上：

- shared details
- tags / color tags
- collections
- recent imports
- Search
- 各 subset library pages

本阶段是增强和打通，不是推翻和重建。

### 4.2 横向能力优先于页面堆叠

本阶段应优先补齐：

- 批量标签 / 批量颜色标签
- 收藏 / 状态 / 评分
- 智能集合 / 保存搜索条件
- recent family 增强

而不是优先新加大量页面。

### 4.3 整理闭环优先于管理后台感

本阶段应服务“连续整理节奏”：

- 从 Recent 进入
- 批量或连续标记
- 回到 Tags / Collections / 各库页再找回

而不是先做一个功能堆叠很重但低频的后台式页面。

### 4.4 一致性交互优先

同类动作在不同页面中应尽量保持一致：

- 单击选中 / 双击打开
- 详情侧栏承接标记动作
- 批量选择进入明确模式
- 标签与颜色标签尽量就地生效

---

## 5. 本阶段拆分

本阶段分为三个执行批次：

1. 批次 A：批量整理能力
2. 批次 B：状态型元数据层
3. 批次 C：智能集合与 recent family 增强

建议按顺序执行；除非你在局部实现中明确发现共享底座需要先补极少量前置支持，否则不要打乱顺序。

---

## 6. 批次 A：批量整理能力

## 6.1 目标

建立最小但真实可用的批量整理闭环，让用户不必逐个进入详情侧栏完成重复标记动作。

## 6.2 必做范围

### A-1 多选模式基线

实现一个清晰、克制的多选模式：

- 支持显式进入多选模式
- 支持退出多选模式
- 进入多选后显示已选数量
- 进入多选后显示批量操作条
- 退出后恢复普通浏览模式

多选模式应优先支持以下页面：

- Recent
- Media
- Search
- 必要时可扩展到 Books / Software / Games，但不要为了“一次覆盖全部”把实现做复杂

### A-2 批量标签

实现批量为多个文件添加标签的最小闭环：

- 批量选择多个 item
- 调出统一批量标签入口
- 复用已有 TagPicker 逻辑，不重新发明标签编辑系统
- 提交后批量更新成功
- 当前页面列表 / 卡片能立即回显结果

本阶段默认优先做：

- 批量添加标签

以下可以后置：

- 批量移除标签
- 批量覆盖标签集合

### A-3 批量颜色标签

实现批量设置颜色标签：

- 多选多个 item
- 调出颜色标签选择器
- 应用后统一更新
- 页面结果即时回显

本阶段默认优先做：

- 批量设置颜色标签
- 批量清除颜色标签

### A-4 最近导入连续整理节奏增强

在 Recent 中把批量整理作为最优先落地点：

- Recent 页进入后容易开启多选
- Recent 页批量标签和批量颜色标签路径尽量短
- 处理一批新内容后可以自然回到普通浏览模式

## 6.3 实现要求

- 优先新增 `BatchActionBar` 或等价能力，但必须保持轻量
- 不做复杂批量文件操作（移动、删除、批量打开目录等）
- 详情侧栏与多选模式关系要清晰：
  - 可以在多选时收起详情侧栏
  - 或切到批量摘要态
  - 但不要让两套交互互相打架

## 6.4 验收标准

满足以下条件即可视为批次 A 完成：

1. 用户可在至少一个高频页面进入多选模式
2. 用户可对一批文件统一添加标签
3. 用户可对一批文件统一设置颜色标签
4. 批量操作完成后结果有清晰回显
5. 交互没有把页面变成复杂文件管理器模式

---

## 7. 批次 B：状态型元数据层

## 7.1 目标

在普通标签与颜色标签之外，引入一层更轻量、更直觉的管理语义，使用户可以表达“喜欢、常用、处理状态、主观评分”等信息。

## 7.2 必做范围

### B-1 收藏

新增收藏能力：

- 单文件收藏 / 取消收藏
- 在详情侧栏中可见且可操作
- 在列表 / 卡片中有轻量回显
- 能参与筛选或单独入口聚合

### B-2 状态

新增状态型元数据，但第一版状态集合必须克制。

建议采用统一轻量状态集合，例如：

- `unprocessed`
- `in_progress`
- `archived`
- `favorite` 若已由收藏独立承担，则不要重复

注意：

- 状态是横向能力，不是某个垂类库私有字段
- 不要因为 Games / Books / Software 语义不同就立即做出完全割裂的状态系统
- 如需后续做垂类特化状态，应建立在统一基线之上，而不是本阶段直接分裂

### B-3 评分

新增评分能力，但第一版保持最小化：

- 单文件设置评分
- 单文件清除评分
- 评分展示轻量可见
- 可在必要时参与筛选

评分建议使用简单整数离散值，例如 1–5。

## 7.3 设计要求

- 收藏 / 状态 / 评分的 UI 优先放入 shared details 中
- 页面卡片和列表只做轻量回显
- 不新增重管理页
- 不把评分做成评论系统
- 不把状态做成复杂状态机

## 7.4 数据与接口要求

Codex 可根据现有代码结构决定最小持久化方案，但必须遵守：

- 仍然围绕已索引文件记录附加元数据
- 不建立新的正式事实中心替代 `files`
- 不将收藏 / 状态 / 评分做成与 tags 完全平行但互不兼容的封闭系统

如需新增接口，建议优先保持风格统一，围绕 file-level metadata patch/update 方式演进。

## 7.5 验收标准

满足以下条件即可视为批次 B 完成：

1. 用户可对单个文件设置收藏
2. 用户可对单个文件设置状态
3. 用户可对单个文件设置评分
4. 这些元数据能在至少一个页面中回显
5. 它们不会与 tags / color tags 冲突或重复承担同一语义

---

## 8. 批次 C：智能集合与 recent family 增强

## 8.1 目标

增强“再次找到”的稳定入口，把当前已有 Collections 和 Recent 能力补到更像系统化组织层，而不是零散附属页。

## 8.2 必做范围

### C-1 保存搜索条件 / 智能集合基础增强

在现有 Collections 基础上增强：

- 支持保存一组当前筛选条件
- 允许用户重新进入该组条件
- Collections 应继续理解为 saved retrieval surface，而不是复杂规则平台
- 不做自动规则引擎

### C-2 Recent family 增强

当前阶段建议扩展 recent family 的最小范围：

- `Recent Imports` 增强
- `Recent Marked` 或等价轻量记录入口
- `Recent Opened` 若现有底座易于承接可纳入，否则可作为后置预留

注意：

- recent family 的目标是增强高频回流路径
- 不是做日志中心或行为分析平台

### C-3 跨页回流增强

打通以下路径：

- 从 Recent 标记后，经 Tags / Collections 再次找到
- 从 Search 保存筛选到 Collections
- 从各库页进入收藏/状态/评分后，再通过 Collections / Recent 找回

## 8.3 实现要求

- Collections 仍然保持“克制的保存入口”定位
- 不新增重规则编辑器
- 页面设计上优先增强现有入口角色，而不是新增很多并列导航页
- 允许适度增强 Home / Recent / Collections 的摘要能力，但不要把 Home 变成重 Dashboard

## 8.4 验收标准

满足以下条件即可视为批次 C 完成：

1. 用户可以保存一组筛选条件并再次进入
2. 用户可以通过 recent family 更自然地重新回到最近处理过的内容
3. 不同页面之间的组织回流关系更顺畅
4. Collections 没有被做成复杂规则平台

---

## 9. 前端实施约束

### 9.1 优先复用的 feature / component

Codex 应优先基于现有或既定共享能力推进：

- `DetailsPanelFeature`
- `TaggingFeature`
- `FilterBarFeature`
- `RecentImportsFeature`
- `BatchActionBar`（新增或等价）
- `TagPicker`
- `ColorTagPicker`
- 现有列表 / 卡片容器

### 9.2 不建议做的前端偏航

- 不为每个库页单独做一套批量整理组件
- 不为收藏 / 状态 / 评分单独做后台页
- 不把 Home 迅速膨胀成复杂总控台
- 不过早构造大型全局状态系统

### 9.3 状态管理边界

建议如下：

- 当前页面的多选状态：页面局部管理
- 批量操作工具条显示状态：页面局部或 feature 层管理
- 收藏 / 状态 / 评分的持久化结果：通过数据层刷新回流
- 不要把所有页面的多选、筛选、排序都塞进一个全局 store

---

## 10. 后端实施约束

### 10.1 允许新增的内容

允许为本阶段新增最小必要的：

- file-level metadata 存储
- 批量标签/颜色标签更新接口
- 收藏 / 状态 / 评分的最小持久化与读取支持
- collections / recent family 的轻量增强接口

### 10.2 明确禁止的内容

- 新建替代 `files` 的主对象系统
- 把收藏 / 状态 / 评分做成复杂独立域模型中台
- 引入复杂异步自动化流水线
- 引入 AI 分类或外部 metadata 强依赖

---

## 11. 页面影响范围

本阶段主要影响以下页面或 surface：

- `/recent`
- `/search`
- `/library/media`
- `/library/books`
- `/library/software`
- `/library/games`（如该阶段进入时已存在）
- `/tags`
- `/collections`
- shared details 所在的右侧详情体系

必要时可对 Home 做轻量摘要增强，但 Home 不应在本阶段成为主战场。

---

## 12. Codex 实施顺序建议

建议按下面顺序提交：

### 提交 1：批次 A 最小闭环

- 多选模式
- 批量标签
- 批量颜色标签
- Recent 中最小落地

### 提交 2：批次 B 最小闭环

- 收藏
- 状态
- 评分
- shared details 中的轻量编辑与回显

### 提交 3：批次 C 最小闭环

- 保存筛选条件 / Collections 增强
- recent family 最小增强
- 跨页回流修整

### 提交 4：统一体验收边

- 空状态 / 无结果 / 回显细节统一
- 多选模式与详情侧栏冲突修复
- 卡片 / 列表回显统一
- 小范围命名与交互一致性修整

---

## 13. 验收总标准

整个“组织层与库体验补齐”阶段完成时，应满足以下总标准：

1. 用户可以对一批内容进行连续整理
2. 用户可以使用收藏 / 状态 / 评分表达轻量管理语义
3. 用户可以保存并复用检索条件
4. 用户可以通过 Recent / Tags / Collections / 各库页自然回流
5. 系统仍然保持 indexed-files workbench 的方法论
6. 没有滑向复杂自动化平台、AI 平台或对象中台

---

## 14. 交付物要求

Codex 输出至少应包括：

- 涉及页面与组件的实现代码
- 必要的数据层 / API 改动
- 简短的实现说明
- 明确列出本阶段做了什么、没做什么
- 可验证的手工验收步骤

若改动涉及多个 surface，必须说明复用关系，而不是只罗列页面功能点。

---

## 15. 本阶段后的自然下一步

本阶段收口后，下一步不应继续无限补局部功能，而应进入：

- 测试版范围冻结与体验统一
- 或按整体阶段规划进入测试版收口与外部验证

也就是说，本阶段的意义是让系统具备更完整的“长期整理”能力，为测试版奠定横向组织层基础。
