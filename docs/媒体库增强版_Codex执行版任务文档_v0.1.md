# 媒体库增强版（Codex 执行版任务文档）v0.1

## 1. 文档目的

本文档用于将《媒体库增强版（范围草案）v0.1》压缩为一份可直接交付 Codex 执行的任务文档。

本执行版文档的目标不是重新讨论产品方向，而是把已经冻结的阶段边界转译为：

- 明确的实现目标
- 明确的非目标
- 明确的前后端改动边界
- 明确的批次拆分
- 明确的验收标准

本阶段默认是后续阶段中的**第一实现阶段**，应优先于游戏库、电子书库、软件库与组织层的实际编码推进。

---

## 2. 当前前提

Codex 在开始本任务前，默认接受以下前提，不应在实现过程中擅自改写：

1. 当前项目是 **Windows local-first 本地资产工作台**，不是媒体平台。
2. 当前系统以 `files` 为统一索引对象。
3. `Media` 是基于 indexed files 的 subset surface，而不是独立媒体数据库。
4. shared details 仍然是共同详情与动作中心。
5. tags / color tags / collections / Search 仍然是媒体库可复用的横向组织层。
6. 桌面 open actions 仍由现有薄壳承接。
7. 本阶段不允许引入重对象模型中台。

---

## 3. 阶段目标

本阶段的实现目标是：

> 把当前 `Media` 从“image/video subset listing”增强为一个更接近“素材库测试版”的可用入口。

用户在本阶段结束后，应能明显感知到：

- 媒体库更像素材库，而不是普通文件页轻包装
- 查找图片/视频时主要依靠缩略图、卡片和视觉浏览
- 标签、颜色标签与基础筛选在媒体库中成为高频入口
- 从扫描进入系统，到筛选、标记、再次找到素材的链路更自然

---

## 4. 非目标

本阶段明确不做以下内容，Codex 不得自行扩写：

- 图片编辑器
- 视频编辑器
- 专业 DAM 平台能力
- AI 自动打标签
- OCR / 语义识别 / 内容分析
- 独立媒体对象中心表
- 独立媒体详情体系
- 重异步媒体处理流水线
- 媒体推荐、聚类、相似搜索
- 深播放器体系

如果某项实现明显会把 `Media` 推向媒体平台化、对象中台化或 AI 能力扩张，应视为偏航。

---

## 5. 实现总策略

本阶段应坚持以下实现策略：

1. **优先增强现有页面和现有链路**，而不是重建媒体系统。
2. **优先复用现有 feature / shared components**，而不是单独造媒体页私有体系。
3. **优先做体验增强和路径打顺**，而不是为了“未来扩展”过度抽象。
4. **允许少量后端补强**，但必须服务于现有 `GET /library/media` 与 `GET /files/{id}` 主链。
5. **所有改动应可分批验收**，不要把整个阶段一次性重写完再交付。

---

## 6. 任务拆分原则

本阶段拆为三个批次执行：

- Batch 1：浏览密度增强
- Batch 2：筛选与预览增强
- Batch 3：整理闭环增强

默认要求：

- **一次只实现一个 Batch**
- 每个 Batch 必须可独立运行、可独立验收
- 上一批未通过，不进入下一批

---

# 7. Batch 1：浏览密度增强

## 7.1 目标

让 `/library/media` 首屏明显更像素材库，而不是文件列表变体。

## 7.2 任务范围

### 前端必须完成

1. 优化 `MediaLibraryPage` 的卡片流 / 网格浏览表现
2. 提供更合理的媒体浏览密度
3. 提高缩略图在媒体卡片中的视觉优先级
4. 优化媒体库空状态
5. 优化媒体库加载状态
6. 优化缩略图未就绪时的占位状态
7. 保持媒体卡片的选中态清晰
8. 保持单击选中详情、双击打开文件的统一交互
9. 保持右侧 details 继续作为主详情承接面

### 优先影响的模块

- `MediaLibraryPage`
- `MediaLibraryFeature`
- `MediaCard`
- `FileGrid`（如当前媒体库复用该容器）
- `RightDetailsPanel`
- 空状态 / 骨架态 / 无结果态相关共享组件

## 7.3 明确不改

Batch 1 不应改动以下内容：

- tags / color tags 数据模型
- collections 数据模型
- shared details 的主语义
- backend media route 主体语义
- 文件打开动作语义

## 7.4 允许的实现判断

Codex 可以：

- 调整媒体卡片的信息层级
- 调整网格密度与缩略图比例
- 调整页面布局和状态呈现方式
- 优化选中态、hover 态、骨架态

Codex 不可以：

- 新增独立媒体详情页体系
- 让媒体库脱离 shared details
- 新造一套媒体专属状态系统
- 因为媒体页而修改全局产品定位

## 7.5 Batch 1 验收标准

完成后至少应满足：

1. `/library/media` 首屏明显以视觉浏览为主
2. 用户找图找视频时，不再主要依赖文件名
3. 单击 / 双击 / 右侧详情的既有工作流未被破坏
4. 空状态、加载状态、缩略图占位态达到“可交付测试版”的完成度
5. 前端 build 通过
6. 基础手工 smoke 通过

---

# 8. Batch 2：筛选与预览增强

## 8.1 目标

让用户在媒体库中更自然地缩小范围、确认内容，不必频繁退回 Search / Files 页面。

## 8.2 任务范围

### 前端必须完成

1. 强化媒体库中的类型筛选入口
2. 强化标签筛选入口
3. 强化颜色标签筛选入口
4. 强化排序入口
5. 让当前筛选条件回显更清楚
6. 提升媒体卡片与 shared details 之间的预览协同
7. 提升 details 中媒体基础信息的可读性

### 后端允许补强

如果当前后端字段不足以支持更稳定的媒体展示与筛选，可做小范围补强，但必须满足：

1. 仍围绕 `GET /library/media`
2. 仍围绕 `GET /files/{id}`
3. 不引入新媒体中心对象层

可接受的补强包括：

- 轻量字段补充
- 轻量排序/过滤参数增强
- 现有 metadata/thumbnail 返回稳态增强

## 8.3 明确不改

Batch 2 不应引入：

- OCR
- 图像相似搜索
- 视频内容分析
- AI 自动标签
- 独立媒体 metadata 平台

## 8.4 验收标准

完成后至少应满足：

1. 用户可直接在媒体库中用类型 / 标签 / 颜色标签明显缩小范围
2. 当前筛选状态可被用户清楚理解
3. 详情中的媒体基础信息比当前更自然、可用
4. 用户不需要频繁跳到 Search 才能完成媒体定位
5. 前后端 build 通过
6. 基础手工 smoke 通过

---

# 9. Batch 3：整理闭环增强

## 9.1 目标

把媒体库从“主要看素材的地方”，推进到“可以顺手整理素材的地方”。

## 9.2 任务范围

### 必须完成

1. 强化媒体库与 `Recent` 的回流关系
2. 强化媒体库与 `Tags` 的回流关系
3. 强化媒体库与 `Collections` 的回流关系
4. 优化从“刚扫描进入系统”到“打标签 / 打颜色标签 / 再找到”的链路
5. 保持 shared details 承接标签、颜色标签、打开文件、打开所在目录等动作
6. 优化媒体库内连续处理节奏

## 9.3 当前限制

Batch 3 不要求完成完整“组织层阶段”的内容，因此不应提前扩写：

- 完整批量整理工作台
- 收藏 / 状态 / 评分全量体系
- 智能集合引擎升级
- 最近使用 / 最近标记全量系统

这类能力属于后续“组织层与库体验补齐”阶段。

## 9.4 验收标准

完成后至少应满足：

1. 用户能从 Recent 进入媒体库继续整理
2. 用户能从媒体库打标签后，通过 Tags / Collections 再次找回内容
3. 媒体库已具备“轻量整理入口”的属性
4. shared details、tags、color tags、collections 的复用链没有被破坏
5. 前后端 build 通过
6. 基础手工 smoke 通过

---

## 10. 前端实现要求

Codex 在前端实现时应遵守以下约束：

1. 页面层负责布局，不要把大量业务逻辑堆到单页文件中。
2. 优先复用：
   - `MediaLibraryFeature`
   - `DetailsPanelFeature`
   - `FilterBarFeature`
   - `TaggingFeature`
   - 共享空状态 / 骨架态 / 无结果组件
3. 不要为媒体库单独再造一套 details 管理。
4. 保持单击选中、双击打开的统一交互。
5. 不要为“以后可能会支持更多媒体能力”提前过度抽象。

---

## 11. 后端实现要求

Codex 在后端实现时应遵守以下约束：

1. 默认继续复用：
   - `files` 统一索引对象
   - 现有 `GET /library/media`
   - 现有 `GET /files/{id}`
   - 现有 thumbnail / metadata / tags / color tags / collections 基础能力
2. 如果确需补字段或补参数，应控制在轻量增强范围。
3. 不得引入新的媒体中心事实层。
4. 不得在本阶段引入复杂异步媒体处理管线。
5. 不得因为媒体页需求而扭曲全局数据语义。

---

## 12. 不允许的偏航行为

如果 Codex 输出中出现以下方向，应视为不合格：

- 把 `Media` 改造成独立媒体平台
- 新增 `media_items` 等重中心化主表，并让其替代 `files`
- 把 shared details 拆掉或旁路
- 让 tags / color tags / collections 在媒体库中变成私有系统
- 引入 AI 自动分析 / 自动标签 / OCR / 语义检索
- 一次性把媒体库做成“未来大平台”的预埋架构

---

## 13. 提交要求

每个 Batch 的提交内容至少应包括：

1. 改动摘要
2. 涉及文件列表
3. 前端 build 结果
4. 后端测试 / build 结果（如有）
5. 手工 smoke 路径
6. 已知限制
7. 明确说明本批次未做内容

如果某一批次需要后端补强，应明确区分：

- 哪些是当前批次必要补强
- 哪些属于后续阶段，不应提前实现

---

## 14. 手工验收路径

每个 Batch 交付后，至少按以下路径进行手工验收：

### 路径 A：媒体浏览
- 进入 `/library/media`
- 浏览图片 / 视频内容
- 检查缩略图、卡片密度、选中态、空态、骨架态

### 路径 B：详情联动
- 单击某个媒体项
- 检查右侧 details 是否正确刷新
- 检查打开文件 / 打开所在目录是否可用

### 路径 C：筛选缩小范围（Batch 2 起重点）
- 按类型筛选
- 按标签筛选
- 按颜色标签筛选
- 清除筛选并确认结果恢复正常

### 路径 D：整理闭环（Batch 3 起重点）
- 从 Recent 进入媒体项
- 打标签 / 颜色标签
- 回到 Tags / Collections / Media 再次找回目标内容

---

## 15. 阶段总验收标准

三个 Batch 全部完成后，阶段总验收至少应满足：

1. `Media` 已具备清晰的“媒体库测试版”气质
2. 浏览、筛选、详情、整理四条链能自然连通
3. 用户主要依靠缩略图 / 卡片，而不是文件名找素材
4. 标签、颜色标签与基础筛选在媒体库中成为真实高频入口
5. 不依赖新增重对象模型
6. 系统没有滑向媒体平台、编辑器或 AI 能力扩张方向

## 17. 给 Codex 的最终执行口径

请严格按以下口径执行：

- 这是一个**增强现有媒体库**的任务，不是重建媒体系统
- 这是一个**subset surface 增强**任务，不是平台化任务
- 这是一个**分批可验收**任务，不是一次性大重构任务
- 优先保护现有 shared details、tags、color tags、collections、Search 复用链
- 优先做出“素材库测试版”的产品感，而不是堆叠未来能力
