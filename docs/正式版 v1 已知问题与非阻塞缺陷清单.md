# 正式版 v1 已知问题与非阻塞缺陷清单

## 1. 文档目的

本文件用于冻结当前正式版 v1 的**已知问题边界**，帮助团队在 release 讨论中明确区分：

- 什么是真正的已知问题
- 什么是可以接受的非阻塞缺陷
- 什么是当前版本**明确不纳入**的事项
- 什么情况会真正阻止当前版本被定义为 v1
- 什么内容可以安全延后到 `v1.x` 或后续阶段

本文件不是下一阶段需求池，不承担扩 scope 作用，也不用于把未来愿景反推为“当前应该已经实现”。  
判断口径以**当前仓库代码、当前验收文档、当前状态档案**为主，而不是更早、更宽的产品草案。

---

## 2. 当前判断原则

当前版本的 release 判断采用以下四类口径：

### 2.1 v1 blocker
会破坏当前已验收主链，使当前版本不能再被称为 usable MVP / v1 的问题。

典型特征：

- source onboarding、scan/delete-sync、search、files/media/recent retrieval、shared details、open actions 等主链能力失效
- 当前版本的核心承诺无法成立
- 问题不是“能力偏薄”，而是“核心链路失真或不可用”

### 2.2 non-blocking defect
真实存在的缺陷、不一致、体验缺口或 release-facing 粗糙点，但不破坏当前 v1 的核心承诺。

典型特征：

- 行为不够友好、表达不够清楚、局部体验偏薄
- 不影响主链闭环成立
- 文档化后可以随 v1 发布一起接受

### 2.3 intentional non-scope item
当前文档与代码一致地表明**本来就不做**的能力，不能当作 bug 统计。

典型特征：

- 在 `AGENTS.md`、当前阶段文档和当前 dossier 中都被明确排除
- 用户可能会期待，但当前版本从未承诺

### 2.4 tech debt note
架构、流程、测试、文档、bootstrap 或工程机制上的真实债务，但不构成当前 v1 冻结阻碍。

典型特征：

- 对未来维护成本有影响
- 对当前 release 可信度影响有限
- 更适合纳入 release/polish 或后续 hardening

### 2.5 判断优先级
当前文档的判断优先级固定为：

1. 当前代码与测试
2. `docs/mvp_验收总结_v_1.md`
3. `docs/current-project-status-dossier.md`
4. 当前已落地的 Phase 2A / 2B / 2C / 2D / 2E 文档
5. 更早、更宽的产品愿景文档

因此，旧 `docs/` 中更宽的能力设想，不能被反推为“当前缺失即 bug”。

---

## 3. 当前 v1 阻塞项判断

## 结论

**当前未发现真实 v1 blocker。**

理由如下：

- `docs/mvp_验收总结_v_1.md` 当前结论为 **Accept**
- `docs/current-project-status-dossier.md` 当前明确将系统定义为 **usable MVP**
- 当前 repo 已经形成并收口的主链包括：
  - source onboarding
  - scan / delete-sync
  - search
  - files / media / recent / tags / collections retrieval
  - shared details panel
  - normal tags / color tags
  - desktop open actions
- 当前已落地的 Phase 2A / 2B / 2C / 2D / 2E 都是在既有主链上做 **hardening / enrichment**，而不是留下主链级断裂

### 当前什么情况才会构成 blocker

如果以下任一类问题仍未解决，则会阻止当前版本被称为 v1：

- source scan 主链失效
- Search / Files / Details 主检索链失效
- shared details panel 无法稳定消费已选文件
- desktop open actions 无法在桌面壳中工作
- 核心数据语义（如 delete-sync、active indexed files）失真

但按当前代码、测试和现有文档，**并没有发现这种级别的 unresolved 问题**。  
因此，当前版本的已知事项应被归为：**非阻塞问题、范围外事项或技术债**，而不是 blocker。

## 4. 已知问题（非阻塞）

以下问题是真实存在、值得记录的 release-facing 已知问题，但它们**不会阻止当前版本冻结为 v1**。

---

### A. source / scan runtime

#### 4.1 Source onboarding 仍以手动路径输入为主

- **当前行为**：source 创建仍依赖手输路径；桌面 preload 中的 `selectFolder` 目前返回 `null`，尚未形成真实目录选择流程。
- **为什么不是 v1 blocker**：当前 source onboarding 主链已经成立，用户仍可通过手输有效路径完成 source 创建并执行 scan。
- **用户影响**：桌面端 onboarding 不够友好，路径输入易出错，体验更偏开发者友好而非桌面工具友好。
- **建议后续处理阶段**：`v1.x / release polish`

#### 4.2 Source runtime 反馈仍是 source-local 的薄表达

- **当前行为**：Phase 2D 已补齐 same-source active scan conflict 和 `last_scan_error_message`，但仍没有 scan history、retry、task center 或更完整 runtime surface。
- **为什么不是 v1 blocker**：当前 v1 目标是 inline scan hardening，而不是完整 task platform。失败原因已经可以局部表达，且 same-source 冲突已由 backend 拒绝。
- **用户影响**：用户可以知道“当前失败了、为什么失败”，但无法继续回看历史或执行更强的 runtime 操作。
- **建议后续处理阶段**：`later phase / runtime hardening bucket`

---

### B. search / files retrieval

#### 4.3 Search 仍是最小检索面

- **当前行为**：Search 已支持 `query`、`file_type`、`tag_id`、`color_tag`，但不支持更宽的 source/path filter 或更复杂的 retrieval 表达。
- **为什么不是 v1 blocker**：当前 v1 检索闭环已经成立，且 Search / Files / Tags / Collections 已形成分工明确的最小 retrieval loop。
- **用户影响**：更复杂找回场景仍需要在 Search、Files、Tags、Collections 之间切换，而不是在一个统一超级查询器中完成。
- **建议后续处理阶段**：`v1.x / later retrieval phase`

#### 4.4 Search 与 Files 的组织层过滤仍偏克制

- **当前行为**：Phase 2C 已把 tag / color-tag filter 接到 Search 与 Files，但过滤能力仍严格停留在单值、纯 `AND`、无高级逻辑的范围内。
- **为什么不是 v1 blocker**：这是当前 phase 明确收敛的结果，不影响“find → inspect → tag → refind → browse”主链成立。
- **用户影响**：需要复杂多条件找回的用户会感到检索表达能力仍然偏薄。
- **建议后续处理阶段**：`v1.x / retrieval polish`

> 注：`FilesPage` 不是目录树浏览器这一点属于**当前明确不纳入 v1 的事项**，见第 5 节，不在本节作为 defect 归类。

---

### C. metadata / thumbnail / preview

#### 4.5 Metadata 仍停留在 image-first baseline

- **当前行为**：当前 metadata 主要稳定消费 image `width` / `height`；video/document 相关字段在 detail wire shape 中保留，但仍未真正激活。
- **为什么不是 v1 blocker**：这是 Phase 2A 明确收敛后的结果，不影响核心的 find → inspect → open 主链。
- **用户影响**：非图片文件在详情面上的 metadata 价值仍然偏薄。
- **建议后续处理阶段**：`v1.x enrichment`

#### 4.6 Thumbnail / preview 仍只覆盖 image 最小面

- **当前行为**：image thumbnail / preview 已落地；video poster、document preview、更强的 media surface 尚未实现。
- **为什么不是 v1 blocker**：当前 MediaLibrary 与 DetailsPanel 已有最小可用的 image-first surface，且 richer preview 并不是当前 v1 已承诺能力。
- **用户影响**：媒体浏览效率已有提升，但视频和文档体验仍明显偏弱。
- **建议后续处理阶段**：`later media enrichment phase`

---

### D. tags / color tags / collections

#### 4.7 TagsPage / CollectionsPage 都是 retrieval-first，而非管理中心

- **当前行为**：TagsPage 只做按普通 tag 找回；CollectionsPage 只做最小 list/create/delete/result retrieval，不承担 rename/reorder/grouping 等管理角色。
- **为什么不是 v1 blocker**：这与当前 phase 文档和当前 dossier 一致，说明当前设计是故意收敛，而不是某个已承诺 feature 失效。
- **用户影响**：用户可以用它们来“找回”，但不能把它们当成完整 tag management center 或 saved view management center。
- **建议后续处理阶段**：`v1.x / later organization phase`

#### 4.8 Collections 的 stale reference 语义较克制

- **当前行为**：若 collection 保存的 `source_id` 或 `tag_id` 后续失效，collection 仍然保留，只返回空结果，不进入失效修复或删除流。
- **为什么不是 v1 blocker**：这是当前 baseline 的明确选择，用来避免引入 collection invalidation management 子系统。
- **用户影响**：用户可能遇到“集合还在，但结果为空”的轻微困惑。
- **建议后续处理阶段**：`v1.x polish / collections follow-up`

---

### E. desktop shell / open actions

#### 4.9 桌面壳当前仍是薄 shell，非完整桌面集成面

- **当前行为**：`openFile` / `openContainingFolder` 已经成立，但更强的桌面集成仍不存在，例如 folder picker、右键菜单、Explorer reveal/select、批量打开等。
- **为什么不是 v1 blocker**：当前 v1 对桌面能力的核心承诺是“能从 shared details panel 打开文件与打开所在目录”，这一点已成立。
- **用户影响**：桌面工具感已经具备，但仍然偏薄，不像完整桌面文件工具。
- **建议后续处理阶段**：folder picker 可进入 `v1.x`；其余留给 future phase

---

### F. docs / bootstrap / release-facing consistency

#### 4.10 Sidebar / Home 存在少量 release-facing 文案不一致

- **当前行为**：
  - `AppSidebar` brand 仍显示 `Phase 0`
  - `HomeOverviewFeature` 的 quick links 仍未包含 `Collections`
- **为什么不是 v1 blocker**：这些问题不影响主链、不影响数据正确性、不影响任何核心流程达成。
- **用户影响**：会削弱“当前版本已经进入 v1 freeze”的完成度感，产生轻微 release-facing 粗糙感。
- **建议后续处理阶段**：`release/polish`

#### 4.11 文档与代码存在历史漂移

- **当前行为**：旧 docs 仍讨论 `library_items`、database-backed `thumbnails`、更宽的 preview/library 能力，以及旧阶段顺序等；而当前代码和当前 dossier 已采用更窄、已实现的真实边界。
- **为什么不是 v1 blocker**：当前 dossier 与较新的 phase docs 已经在收口，且核心主链并未因此失效。
- **用户影响**：团队容易把 scope 外能力误判成“已经应该有、但没做出来的 bug”。
- **建议后续处理阶段**：`release/polish` + `tech debt bucket`

---

## 5. 明确不纳入 v1 的事项

以下事项在当前版本中**不是 bug**，而是明确不纳入 v1 的范围边界：

- 完整 task platform、scan history、retry system、runtime center
- Explorer replacement / 目录树型 FilesPage
- richer media preview：
  - video poster extraction
  - document preview
  - hover/play
  - richer media canvas
- advanced query builder、OR / NOT、多条件表达式
- smart rules / automation engine
- game / book / software 垂类库
- AI / semantic / OCR / embeddings
- batch actions
- plugin architecture / cloud sync / auth
- page-specific details/action systems
- Search / Files / Media / Recent 的大改版
- dashboard 化 Home
- preferences 化 Settings
- tag rename / delete / merge
- collections rename / reorder / grouping

这些事项若后续要做，应进入 `v1.x` 或更后续 phase，而不是在当前 v1 缺陷清单中被误记为“未修复 bug”。

---

## 6. 当前技术债但不阻塞 v1 的事项

### 6.1 baseline SQL 继续增长

- **依据**：当前 schema bootstrap 仍依赖反复执行 `0001_initial_core.sql`
- **性质**：数据库 bootstrap 技术债
- **为什么不阻塞 v1**：当前 fresh/existing DB 都还能工作，但长期维护成本会继续上升

### 6.2 migration 机制仍偏轻量

- **依据**：当前 implemented phase docs 多次明确，repo 现实仍是 baseline SQL update，而不是完整 migration runner
- **性质**：工程机制债
- **为什么不阻塞 v1**：不会立即破坏当前 release，但会影响后续 schema 演进清晰度

### 6.3 inline scan model limits

- **依据**：当前 dossier 与 Phase 2D 文档都明确 scan 仍是 inline run
- **性质**：runtime 架构边界
- **为什么不阻塞 v1**：当前 v1 只承诺“可运行 scan 主链”，不承诺完整后台任务平台

### 6.4 image-first enrichment boundary

- **依据**：Phase 2A / 2B 的当前实现和文档都明确 metadata / preview 仍是 image-first
- **性质**：能力边界 + 技术债
- **为什么不阻塞 v1**：当前 enrichment 已能增强现有主链，只是覆盖面仍窄

### 6.5 shared details panel complexity concentration

- **依据**：`DetailsPanelFeature` 当前已承担 details、tags、color tag、metadata、preview、open actions
- **性质**：前端复杂度集中债
- **为什么不阻塞 v1**：当前统一详情/动作面仍是优势，但后续维护压力会增大

### 6.6 前端 / 桌面仍以 build + manual verification 为主

- **依据**：当前 dossier 已明确，backend 有较完整回归基线，但 frontend / desktop 仍更依赖 build-level validation 和手工流验证
- **性质**：测试债
- **为什么不阻塞 v1**：当前不代表不可发布，但意味着回归信心更多来自人工而非自动化 UI / E2E

## 7. 建议的后续处理方式

## 7.1 should be handled in release/polish

- `AppSidebar` 中仍显示 `Phase 0` 的 release-facing 文案
- Home quick links 漏掉 `Collections`
- 当前 docs/code drift 的同步修正
- 只涉及已有页面与已有能力的 wording consistency

## 7.2 can wait until v1.x

- source onboarding 的 folder picker 接入
- 更完整但仍克制的 source/runtime 反馈细节
- image-first metadata / preview 之后的有限 enrichment
- collections 的轻量 follow-up，例如更清楚的 stale-ref 提示
- 更强但仍受控的 retrieval/filter polish

## 7.3 belongs to future phase, not v1

- runtime center / task platform
- directory-tree FilesPage
- richer media system
- advanced query builder / smart rules
- game / book / software vertical systems
- AI / semantic layer
- batch actions / broad shell integration

---

## 8. 结论

当前版本**仍然可以冻结为正式版 v1**。

当前清单中：

- **没有发现真实 v1 blocker**
- 当前存在的是：
  - 非阻塞问题
  - 明确不纳入 v1 的事项
  - 当前技术债

因此，更准确的 release 口径应是：

> **当前版本可以按“Accept with documented follow-up”冻结。**

后续工作应以 `release/polish`、`v1.x` 或未来 phase 的方式处理本文列出的事项，而不是借“已知问题清单”的名义继续扩大当前正式版边界。
