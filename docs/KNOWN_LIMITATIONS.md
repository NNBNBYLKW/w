# Workbench Known Limitations

This document lists the known limitations as of the beta release (May 2026). These are not bugs — they are acknowledged constraints of the current version.

---

## Packaging

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Clean Windows install smoke pending | The NSIS installer has been built and verified structurally, but has not been tested on a fresh Windows machine. | Test on clean VM/Windows before distributing. |
| Default Electron icon | The app uses the default Electron icon, not a custom Workbench icon. | Cosmetic. Will be replaced with a proper icon in a future update. |
| Metadata fields missing | `description` and `author` are not set in the desktop `package.json`. | Cosmetic. Affects installer metadata only. |

---

## Performance

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Scan speed | A 10K-file test scan took ~271 seconds (37 files/sec). A 50K-file library is estimated to take ~22 minutes. The scan works correctly but is slow for large libraries. | Run scans when the machine is idle. The scan can be stopped and resumed. |
| Media grid loading | The Media page loads all 50 visible thumbnails simultaneously, taking ~5.7s. | Thumbnails are cached after first load. Subsequent loads are faster. |
| Database size | 10K files produced a ~72 MB database. Large libraries will consume proportionally more disk space. | Periodically clean up unused sources and stale metadata. |
| Full table scans | Search and Recent queries use full table scans with temp B-tree sorting. Fast for 10K files, may slow down at 50K+. | Pagination limits results to 50 per page. No user-visible impact at current scale. |

## Thumbnails

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Corrupted video files | Videos that cannot be decoded will show "Preview unavailable" instead of a thumbnail. | The app handles this gracefully — no crash, no broken image. |
| Thumbnail generation failure caching | Failed generations are cached for 60 seconds. Rapid thumbnail requests for corrupted files produce brief 404 responses. | No user impact — failures are silent. |
| Video thumbnails require FFmpeg | FFmpeg is bundled with the desktop app. In dev mode, FFmpeg must be installed separately or video thumbnails will not be generated. | Desktop app bundles FFmpeg. Dev mode requires manual setup. |

## Organize / File Operations

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Cross-volume move not atomic | Moving files between different drives uses copy+delete rather than an atomic rename. If the app crashes during this, the source file may still exist. | Organize plans block overwrites. Failed actions are logged. Use the same volume for source and managed root when possible. |
| Operation journal is not full rollback | `operation_journal` and `file_path_history` exist for import / organize / compose / amendment flows, but they do not provide automatic crash rollback. | H2 startup recovery marks stale executing plans as failed. Preflight checks prevent unsafe operations. Rollback remains an explicit draft plan. |
| Rollback is manual | Rollback creates a draft plan only. It does not automatically execute or reverse file moves. | You must explicitly mark ready, preflight, and execute the rollback plan. |
| mkdir and asset.yaml not rollbacked | Rollback only reverses move and rename actions. Directories created during execution and asset.yaml files are not removed. | Minor — directories and metadata files are harmless to leave behind. |
| Managed root path exclusion | System directories (C:\Windows, Program Files), drive roots, and internal app directories are blocked. Other potentially unsafe paths (e.g., user profile root) are not blocked but are discouraged. | Choose a dedicated folder for your managed library. |

## AI / Automation

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Rule-based suggestions only | Organize suggestions are generated by a local rule-based engine. They do not use AI, cloud services, or external APIs. | Suggestions are deterministic and safe. Manual review is always required. |
| No auto-classification | Files are not automatically categorized or organized. Users must manually trigger scans, review candidates, and execute plans. | This is by design — Workbench does not touch your files without explicit confirmation. |
| No cloud / LLM / provider | Workbench is fully local. No data leaves your machine. | No network dependency. Works offline. |

---

## UI / Tests

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Frontend tests are partial | Vitest tests and a Playwright smoke configuration exist, but they do not yet cover every user workflow or run as a full product-quality gate in all environments. | Keep manual smoke tests for release candidates and expand automated coverage around core browse / details / organize flows. |
| Media grid not virtualized | Large libraries with many media files may cause the Media page to render many DOM nodes simultaneously. | Pagination limits visible items. Virtualization is a future optimization. |
| Skeleton components duplicated | Browse features (Books, Games, Media, Software) define identical skeleton components inline. | Cosmetic. Will be consolidated to a shared component in a future cleanup. |

---

## Platform

| Limitation | Detail | Mitigation |
|-----------|--------|-----------|
| Windows only | The app currently only targets Windows. macOS and Linux are not supported. | Windows is the primary target for the local-first asset workbench use case. |
| Python 3.14 compatibility warning | PyInstaller reports a compatibility warning for Pydantic V1 on Python 3.14. Pydantic V2 is used and functional. | Informational only. No runtime impact. |
