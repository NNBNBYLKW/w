# Windows 本地资产管理工作台 PoC 执行记录模板 v1

## 1. 文档目的

本文件用于为第一阶段的 PoC / 风险验证提供统一记录格式。

目标是确保每次实验都能稳定回答以下问题：
- 这次验证的目标是什么
- 测试环境是什么
- 使用了什么数据规模
- 采用了什么方法
- 得到了什么结果
- 是否通过
- 如果不理想，下一步该怎么调

本文件不负责定义“要做哪些 PoC”，那部分已经在《PoC 与风险验证清单 v1》中完成。
本文件负责的是：

> **把每次验证过程变成可复盘、可比较、可决策的记录。**

---

## 2. 使用方式

### 2.1 建议原则
1. 每个 PoC 使用一份独立记录
2. 同一个 PoC 的多次实验，建议按版本或轮次编号
3. 所有结果尽量包含可量化数据，不只写感受
4. 结论要明确标记：通过 / 带约束通过 / 不通过
5. 每次记录都要写出“下一步动作”，否则难以形成真正决策

### 2.2 命名建议
建议文件命名格式：

```text
POC-01_大目录扫描性能_实验记录_v1
POC-03_SQLite搜索性能_实验记录_v2
POC-05_图片缩略图缓存_实验记录_v1
```

## B. 验证目标

### B.1 本次验证要回答的问题
用 1~3 句话明确说明这次实验到底想验证什么。

示例：
- 验证在 5 万文件规模下，首次扫描是否能在可接受时间内完成。
- 验证 SQLite 对“文件名模糊搜索 + 类型筛选”的响应时间是否满足搜索页需求。
- 验证图片缩略图是否适合按需生成并缓存，而不是首次全量生成。

### B.2 通过标准（本次）
明确写出本次实验的通过判断标准。

示例：
- 平均查询耗时小于某阈值
- 首屏可交互时间在某范围内
- 失败率低于某比例
- 大量数据下界面仍保持流畅

## G. 结果观察

### G.1 量化结果总结
用简短段落总结核心数据结果。

示例：
- 首次扫描 5 万文件耗时 3 分 25 秒，平均每秒处理约 240 文件。
- 第二次重扫耗时下降到 48 秒，说明增量同步路径有效。
- 搜索查询在常见场景下平均 120ms，但复杂组合筛选在 5 万规模下上升明显。

### G.2 主观体验观察
记录使用时的主观表现，但要和量化结果分开写。

示例：
- 扫描过程中 UI 未卡死，但状态提示不够明显。
- 素材库滚动整体流畅，但首次进入时部分缩略图闪烁较明显。
- 连续点击详情侧栏时，局部骨架过于频繁，体验略割裂。

### G.3 异常情况
记录特殊问题。

示例：
- 遇到 12 个损坏图片文件，缩略图生成失败但未影响主流程。
- 某类视频格式提取封面失败率偏高。
- 某些长路径目录扫描速度异常下降。

## H. 结论判定

### H.1 判定结果
从以下三种中选一：
- **Pass**：可直接进入正式实现
- **Pass with constraints**：可实现，但需带约束进入
- **Fail**：当前方案不建议继续，需先调整

### H.2 判定理由
说明为什么做出该判定。

示例：
- 5 万文件扫描耗时可接受，且不会阻塞 UI，因此扫描主链成立。
- 视频封面提取可用，但只适合按需生成，不适合首次全量处理。
- 多标签复杂组合查询成本明显偏高，第一阶段应收缩为更简单的过滤策略。

---

## I. 风险与问题

### I.1 当前暴露的问题
按重要程度列出。

示例：
1. 缩略图缓存增长速度偏快
2. 搜索结果页在大网格模式下滚动仍有掉帧
3. 详情侧栏频繁切换时局部刷新不够稳定

### I.2 问题影响范围
说明这些问题影响：
- 哪个页面
- 哪条主链
- 是否阻塞 MVP

---

## J. 调整建议

### J.1 立即调整建议
写这次实验后立刻应该调整什么。

示例：
- 扫描阶段只写基础元数据，图片尺寸和视频时长延后到后台任务处理
- 缩略图改为按需生成而非首次全量生成
- 搜索页首屏只返回最小字段集

### J.2 后续跟进实验建议
写下一轮应该继续验证什么。

示例：
- 继续测试 10 万文件规模
- 对 HDD 场景单独做一次扫描实验
- 对 WebP / HEIC / MKV 等格式做兼容性专项验证

## 5. 按 PoC 类型的记录提示

### 5.1 扫描类 PoC
优先记录：
- 文件数
- 总耗时
- 每秒处理文件数
- CPU / 内存占用
- 异常文件数
- UI 是否阻塞

### 5.2 查询类 PoC
优先记录：
- 数据规模
- 查询 SQL / 查询条件
- 平均耗时
- p95
- 首屏耗时
- 索引策略变化

### 5.3 缩略图类 PoC
优先记录：
- 支持格式
- 单张耗时
- 批量耗时
- 缓存体积
- 失败率
- 降级表现

### 5.4 前端性能类 PoC
优先记录：
- 数据条数
- 渲染模式（列表 / 网格）
- 是否使用虚拟滚动
- 滚动流畅度
- 切换详情延迟
- 缩略图加载影响

---

## 6. 推荐记录方式

### 6.1 推荐做法
- 文档记录结论
- 表格记录指标
- 日志或截图作为附件
- 关键测试结果可保留短视频或屏幕录制

### 6.2 不推荐做法
- 只写“感觉还行”
- 不记录数据规模
- 不记录环境差异
- 不区分“通过”和“带约束通过”

---

## 7. 当前结论

这份模板的核心价值不是让文档看起来更完整，而是确保你后面每次验证扫描、查询、缩略图、视频封面、前端滚动时，都能形成：

> **有上下文、有数据、有结论、有后续动作**

只有这样，PoC 才会真正变成项目决策工具，而不是零散实验记录。

---

## 8. 下一步建议

在这份模板之后，最适合继续推进的方向有两个：
1. **前端工程目录与状态管理草案**
2. **技术实现任务清单（更细粒度）**

如果你准备从“文档阶段”进入“开始搭骨架”，我更推荐下一步先做：

> **前端工程目录与状态管理草案**

因为你现在产品、原型、架构、协议、开发阶段、PoC 模板都已经有了，接下来最自然的是把前端工程真正搭成一个可开始写代码的结构。
