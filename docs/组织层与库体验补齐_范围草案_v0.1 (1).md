# 组织层与库体验补齐（范围草案）v0.1

## 1. 文档目的

本文档用于定义后续阶段中的 **阶段四：组织层与库体验补齐** 范围，作为媒体库增强版、游戏库首版、电子书库增强版、软件库增强版之后的横向能力增强草案。

本阶段不负责重定义产品，也不负责把系统推进为复杂自动化平台，而是在当前已经成立的 `files + subset surfaces + shared details + Search / tags / color tags / collections` 方法论之上，补齐真正支撑多垂类库长期使用的组织层与整理体验。

---

## 2. 当前起点

当前项目已经形成一条明确的工作主链：

`source setup -> scan/index -> search/browse/retrieve -> shared details -> tags/color tags/collections -> open actions`

并且已经具有以下基础：

- `files` 作为统一索引对象
- `Search` / `Files` 通用入口
- `Media` / `Books` / `Software` 等 subset surfaces
- 共享详情链 `DetailsPanelFeature`
- normal tags / color tags / collections
- recent imports baseline
- 桌面 open actions

这意味着，本阶段的工作不是从 0 搭建组织系统，而是在既有基础上把“组织层”从**能用**推进到**足以支撑多库长期整理和再找回**。

---

## 3. 本阶段定位

本阶段的核心定位是：

> 让当前已经存在但仍偏薄的组织层，成为多垂类库共同复用的横向能力层。

它服务的不是单一页面，而是：

- 媒体库
- 游戏库
- 电子书库
- 软件库
- Search
- Recent
- Tags
- Collections

这些 surface 之间的组织与回流关系。

本阶段不应把系统推向：

- 复杂自动化规则引擎
- AI 自动标签平台
- 后台化运营系统
- 脱离 `files` 的新对象中台

---

## 4. 本阶段目标

本阶段目标可以概括为四条：

1. **补齐批量整理能力**
2. **补齐状态型轻量元数据层**
3. **补齐智能集合与 recent family**
4. **让多垂类库共享一致的整理逻辑与回流路径**

本阶段结束后，用户应明显感知到：

- 系统不再只是“能浏览、能单个标记”
- 可以对一批新内容进行连续整理
- 可以用收藏 / 状态 / 评分表达轻量管理语义
- 可以通过集合与 recent family 更稳定地再找到内容
- 不同库页的组织体验开始统一，而不是各自为战

---

## 5. 本阶段不做什么

本阶段明确不做以下方向：

- 复杂自动化规则引擎
- 批量搬运 / 自动命名 / 自动归档平台
- AI 自动标签
- OCR / 语义识别 / 内容理解
- 多层级复杂标签体系
- 企业级后台化运营界面
- 独立对象中台或工作流引擎

本阶段要做的是**克制但高价值的横向增强**，而不是把整理系统做成另一条平台线。

---

## 6. 本阶段应坚持的架构前提

本阶段默认继续坚持以下前提：

1. **`files` 仍是底层统一索引主语义**
2. **subset surfaces 仍是内容入口，而不是独立对象系统**
3. **shared details 仍是跨页面高频整理中心**
4. **Search / tags / color tags / collections 继续作为横向组织层的骨架**
5. **新能力优先在现有页面与复用组件上增强，不平地新造一套组织后台**
6. **不为“未来可扩展”过早引入复杂状态机、工作流引擎或中台抽象**

---

## 7. 用户价值定义

本阶段重点服务以下场景：

### 场景 A：整理一批最近导入的素材或文件
用户刚扫描或导入一批图片、视频、书籍、软件，希望快速打标签、颜色标签、收藏或状态。

### 场景 B：跨库表达轻量管理语义
用户希望用“收藏”“在看/在玩/常用/待整理”“评分”等方式，表达比标签更直观的长期状态。

### 场景 C：固化常用检索入口
用户希望把常用筛选结果保存为集合或智能集合，下次一键进入。

### 场景 D：从最近链路重新进入整理工作
用户不只想看最近导入，还希望看最近打开、最近标记、最近整理，从而继续未完成的工作。

---

## 8. 范围总览

本阶段建议分三批次推进：

### 批次 1：批量整理能力
目标：先解决“整理一批内容”的效率问题。

### 批次 2：状态型元数据层
目标：补齐标签之外的轻量表达层，例如收藏、状态、评分。

### 批次 3：智能集合与 recent family 增强
目标：把“常用入口”和“最近链路”补强成真正高频可用的组织入口。

---

## 9. 批次 1：批量整理能力

### 9.1 目标
让用户能对一批文件或库条目进行连续整理，而不是只能逐个处理。

### 9.2 包含内容
- 批量选择模式
- 批量添加标签
- 批量设置颜色标签
- 批量操作条（BatchActionBar）
- Recent 页面优先承接批量整理节奏
- 必要时向 Search / Media / Books / Software / Games 扩展一致的多选交互

### 9.3 优先级说明
第一版批量能力应优先做“高频且低风险”的操作：

- 批量添加标签
- 批量设置颜色标签

以下能力可后置：

- 批量移除标签
- 批量编辑更复杂字段
- 批量打开/批量路径动作
- 批量移动/删除/归档

### 9.4 页面影响范围
优先影响：

- `RecentImportsPage`
- `MediaLibraryPage`
- `SearchPage`

如范围可控，再扩展到：

- `BooksPage`
- `SoftwarePage`
- `GamesLibraryPage`

### 9.5 交互原则
- 不默认一直处于多选模式
- 用户主动进入多选状态后，页面显示批量操作条
- 退出多选后恢复普通浏览模式
- 详情侧栏在多选态下可切换为摘要态，或临时收起

### 9.6 不包含内容
- 批量文件系统操作工作台
- 批量自动分类规则
- 批量重命名、搬运、同步

### 9.7 验收口径
- 用户能在 Recent 或媒体页中选择一批内容
- 用户能一次性完成标签或颜色标签整理
- 整理完成后列表和卡片回显能及时更新
- 多选体验清晰，不会破坏普通浏览节奏

---

## 10. 批次 2：状态型元数据层

### 10.1 目标
在普通标签和颜色标签之外，补齐更轻量、更直觉的状态表达层。

### 10.2 建议能力范围
本批建议优先引入以下三类能力：

#### 收藏（favorite / starred）
用于表达“常用”“偏好”“希望保留”的直觉语义。

#### 状态（status）
用于表达某条目当前所处阶段，例如：

- 待整理
- 已整理
- 常用
- 在看 / 已读 / 搁置（Books）
- 在玩 / 通关 / 搁置（Games）
- 常用 / 测试 / 归档（Software）

#### 评分（rating）
用于表达主观偏好强弱。首版建议做轻量 3 档或 5 档，不做复杂评分体系。

### 10.3 设计边界
本批次要特别注意两件事：

1. **状态是轻量表达层，不是复杂业务状态机**
2. **不同库页可以有推荐状态集合，但不应在底层语义上完全割裂**

建议采取：

- 底层保存统一 status 字段或统一 status 结构
- 页面层按库类型给出推荐文案或推荐选项

### 10.4 展示与使用位置
- DetailsPanel 中可设置/查看收藏、状态、评分
- 卡片/列表中做轻量回显
- Search、Tags、Collections、Recent 中允许基于状态型元数据筛选

### 10.5 不包含内容
- 复杂多级状态流转
- 自动状态变更规则
- 运营型分析后台
- 依赖外部账号的数据状态联动

### 10.6 验收口径
- 用户能在详情侧栏中完成收藏/状态/评分设置
- 这些元数据可在卡片或列表中回显
- 它们可参与基础筛选或集合保存
- 不同库页虽有差异化状态语义，但整体仍属于同一系统

---

## 11. 批次 3：智能集合与 recent family 增强

### 11.1 目标
把当前的 `Collections` 和 `Recent` 从“存在入口”增强到“高频回流入口”。

### 11.2 包含内容
#### A. 保存搜索条件 / 智能集合增强
- 保存当前筛选条件
- 允许以后从集合入口一键进入
- 集合条目更清晰地表达来源和筛选范围
- 尽量复用现有 collections 语义，而不是另起新系统

#### B. recent family 增强
在现有 recent imports 基础上，补强最近链路，建议范围包括：

- 最近导入（existing baseline）
- 最近打开
- 最近标记
- 最近整理（可选，如范围可控）

### 11.3 目标作用
这一批的核心价值不是“统计”，而是：

- 把常用结果变成固定入口
- 把中断的整理工作变成可继续的入口
- 让首页 / recent / collections 在产品中承担更强的回流职责

### 11.4 页面影响范围
- `CollectionsPage`
- `RecentPage`
- `HomePage`
- 各库页中的“保存当前筛选”“从集合重新进入”动作

### 11.5 不包含内容
- 复杂规则引擎式智能集合
- 条件构建器平台
- 自动刷新任务编排系统
- 大而全的 dashboard 分析中心

### 11.6 验收口径
- 用户能把常用筛选保存为集合
- 用户能从集合一键回到目标结果
- recent family 至少不止“最近导入”这一条线
- 首页 / 集合 / 最近链路之间的关系更自然

---

## 12. 页面与组件影响范围

### 12.1 主要影响页面
- `/recent`
- `/collections`
- `/tags`
- `/search`
- `/library/media`
- `/library/books`
- `/library/software`
- `/library/games`
- `/`（HomePage）

### 12.2 主要影响 feature / 组件
- `DetailsPanelFeature`
- `TaggingFeature`
- `SelectionModeFeature`
- `RecentImportsFeature`
- `CollectionsFeature`
- `FilterBarFeature`
- `BatchActionBar`
- `TagPicker`
- `ColorTagPicker`
- 收藏 / 状态 / 评分相关轻量组件
- 空状态、无结果、骨架态组件

### 12.3 默认不新增
- 独立组织后台系统
- 脱离详情侧栏的重整理工作台（除非后续单独立项）
- 大而全的规则配置中心

---

## 13. 后端与数据层边界

### 13.1 默认复用
- `files` 统一索引对象
- 现有 tags / color tags / collections 基础能力
- 现有 details 主链
- 现有 recent imports 基础

### 13.2 允许的补强类型
- 为批量标签 / 批量颜色标签提供必要的批量接口
- 为 favorite / status / rating 提供轻量持久化字段与读写接口
- 为 collections / recent family 提供轻量增强字段或查询接口
- 为筛选与回显提供必要的列表字段补充

### 13.3 明确禁止
- 引入复杂工作流引擎
- 新建脱离 `files` 的组织中心对象层
- 为了智能集合引入复杂自动任务系统
- 把 recent family 做成事件流分析平台

---

## 14. 交互原则

本阶段继续沿用当前统一交互语言：

- 单击查看详情
- 双击打开文件
- 多选是显式进入的模式，不是默认常驻
- 筛选即时生效
- 高价值整理动作尽量在当前页和详情侧栏完成
- 收藏 / 状态 / 评分属于轻量高频动作，不应埋得过深

同时，应额外补充两条：

1. **批量模式必须清晰，不干扰普通浏览模式**
2. **状态型元数据应尽量低摩擦完成，不增加重表单负担**

---

## 15. 进入前提

进入本阶段前，默认以下前提成立：

- 媒体库、游戏库、电子书库、软件库都已具备基础可用入口
- 当前标签、颜色标签、集合与详情链已稳定可用
- 团队已接受“整理体验增强”会成为测试版前的重要主线

本阶段不要求先重构底层架构，也不要求先完成所有垂类深度增强。

---

## 16. 退出条件

本阶段结束时，至少应满足以下条件：

1. 用户可以对一批新入库内容进行连续整理，而不是逐个处理
2. 收藏 / 状态 / 评分已形成轻量元数据层
3. 智能集合与 recent family 已明显强于当前 baseline
4. 多个库页之间的整理逻辑开始统一
5. 组织层不再只是“有标签功能”，而是真正成为多垂类库的共通支撑层
6. 系统仍未滑向复杂自动化平台或对象中台方向

---

## 17. 给 Codex 的执行约束

在编码执行时，应额外遵守以下约束：

- 不改写项目当前产品定位
- 不把组织层做成独立后台平台
- 优先增强现有页面和现有 shared feature
- 能复用现有组件和状态边界的，优先复用
- 批量整理优先服务“标签 / 颜色标签”这类高频动作
- 收藏 / 状态 / 评分做轻量实现，不引入复杂业务状态机
- 智能集合做“保存当前筛选”方向，不做规则平台
- recent family 做“继续工作入口”，不做分析中心

---

## 18. 本阶段后的自然下一步

本阶段完成并验收后，下一步应进入：

> **《测试版收口与外部验证（范围草案）》**

也就是说，组织层补齐不是终点，而是测试版前的最后一层关键横向增强。

---

## 19. 当前待你拍板的 8 个点

这份草案还需要你最终拍板的点有：

1. 批量整理首版是否只做标签与颜色标签
2. 多选模式首版优先覆盖哪些页面
3. 收藏 / 状态 / 评分是否一起进，还是拆批次进
4. 状态是否允许按库页显示不同推荐选项
5. 评分是 3 档还是 5 档
6. 智能集合首版是否仅限“保存当前筛选”
7. recent family 首版具体包含哪几条线
8. 首页是否在本阶段承担更多 recent / collections 回流入口职责
