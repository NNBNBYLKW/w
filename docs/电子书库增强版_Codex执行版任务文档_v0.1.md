# 电子书库增强版（Codex 执行版任务文档）v0.1
> 用于阶段三-A开发执行、任务拆分、验收与交付

## 1. 文档目的与使用方式

本文档不是范围讨论稿，而是交给 Codex 使用的执行版任务文档。它的职责是把《电子书库增强版（范围草案）》压缩为可实施的任务包、顺序、约束与验收口径。

若本执行版与后续人工冻结结论冲突，以新的人工冻结为准；在此之前，Codex 应按本文默认口径实施。

## 2. 上游固定约束（不得改写）

- 当前项目仍是建立在真实文件系统之上的 indexed-files workbench；`files` 是统一索引对象，Books 只能作为 subset surface 扩展，不得另起独立事实层。
- shared details 仍是跨页面详情与动作中心；电子书库不得分叉出独立的重详情系统。
- Search / tags / color tags / collections 仍是横向复用组织层，Books 必须接入这条复用链而不是脱离它自行闭环。
- 桌面打开动作仍由现有薄壳 open actions 承接；本阶段不新增阅读器、阅读进度系统、批注系统或在线书架系统。

## 3. 本执行版采用的暂定口径（用于推动编码）

由于用户尚未对所有细节逐条拍板，本文采用一组“可执行但保守”的暂定口径，目的是先把阶段三-A推进起来，同时尽量避免后续返工。

默认口径如下：

- Books 首版继续围绕当前 `.pdf` / `.epub` 子集增强，不主动扩到更多格式。
- 书库主视觉优先级采用“封面 → 标题 → 作者/格式 → 文件回退信息”的顺序。
- 作者、标题等字段若无法稳定提取，统一回退到文件名与扩展名，不做激进猜测。
- 详情页仍使用 shared details；书籍特化信息只作为该详情体系里的增强区块出现。
- 后端允许做少量字段增强与排序/筛选补强，但不允许新建独立书籍事实层。

## 4. 执行顺序总览

执行顺序固定为 4 个工作包。除非遇到严重阻塞，不建议跳序并行重叠过多。前一个工作包至少应达到“接口与边界已稳定”的状态，才进入后一个工作包。

- WP-1：字段提取回退规则与后端合同
- WP-2：前端页面与展示基线
- WP-3：详情、筛选与回流闭环
- WP-4：QA、文档、演示路径与验收

## 5. WP-1：字段提取回退规则与后端合同

目标：先把“Books 列表与详情展示哪些字段、提取失败如何回退、接口吐什么字段”冻结成代码与文档都能共同依赖的基础合同。

### 5.1 必做项

- 明确 Books 列表最小展示字段：`id`、`name/title`、`format/extension`、`cover/cover_fallback`、可选 `author`。
- 明确字段回退顺序：
  - 标题：优先使用可稳定提取字段，否则回退到文件名。
  - 作者：提取不到则允许为空，不做模糊猜测。
  - 封面：有封面则显示，无封面则显示统一占位。
- 明确 Books 列表接口在现有 subset route 基础上的最小增强方式。
- 明确 `GET /files/{id}` 中书籍详情增强区块的返回口径。
- 明确哪些筛选/排序是本阶段支持的，哪些后置。

### 5.2 可接受实现方式

- 在现有 Books route 上做轻量字段补强。
- 通过现有详情主链扩展最小书籍展示字段。
- 允许使用轻量 metadata 解析或已有字段映射，但不能引入重解析平台。

### 5.3 禁止项

- 不得创建脱离 `files` 的 `books` 主事实表来替代现有主链。
- 不得把联网元数据抓取写成成立前提。
- 不得为了未来扩展预建复杂作者库、系列库中台。

## 6. WP-2：前端页面与展示基线

目标：把 Books 从“扩展名子集列表”拉到“轻量书库入口”的视觉和交互基线。

### 6.1 必做项

- 增强 Books 页面展示密度，使首屏更像书库而不是文件清单。
- 封面优先展示；封面缺失时保持统一占位与稳定布局。
- 标题、作者、格式信息的层级清晰，文件回退信息不喧宾夺主。
- 统一空状态、无结果态、加载骨架态。
- 保持单击查看详情、双击打开文件的统一交互规则。

### 6.2 优先复用

- 复用现有 shared components、RightDetailsPanel、FilterBar、Tagging 相关组件。
- 若需要书籍特化卡片样式，优先在现有卡片体系中做轻量特化，而不是新造一整套页面系统。

### 6.3 禁止项

- 不得在本阶段引入正文阅读区域、章节导航、目录树、批注侧栏。
- 不得把 Books 页面改造成阅读器工作台。

## 7. WP-3：详情、筛选与回流闭环

目标：让电子书库从“能看”推进到“能轻量归类并再次找回”。

### 7.1 必做项

- 让 Books 页面内的标签、颜色标签、格式筛选、基础排序可自然使用。
- 在 shared details 中展示书籍增强字段，并保持标签/颜色标签/打开动作在当前页就地完成。
- 打通 Books 与 Search、Tags、Collections 的回流关系。
- 至少验证一条“从 Books 归类后，再从 Tags/Collections/Search 找回”的闭环。

### 7.2 暂不纳入

- 阅读状态（未读/在读/已读）全量体系。
- 阅读进度、笔记、摘录、批注。
- 复杂系列化浏览或作者中心页。

### 7.3 结果要求

- 用户不需要频繁回退到 Files 才能完成书籍定位与归类。
- 书库的“浏览 → 详情 → 归类 → 再找回”路径应明显顺滑于当前 subset v1。

## 8. WP-4：QA、文档、演示路径与验收

目标：把阶段三-A从“功能片段”收成可交付、可演示、可继续承接软件库增强版的切片。

### 8.1 必做项

- 补齐 Books 相关页面的关键空状态、无结果态、加载态校验。
- 验证无封面、无作者、标题提取失败等回退场景。
- 验证 shared details、tags、collections、Search 回流是否正常。
- 更新阶段文档与 current-state slice 记录，避免把假设写成既定事实。

## 9. Codex 任务拆分建议

建议把阶段三-A拆成 3 个主任务 + 1 个收尾任务，而不是一次性交给 Codex 一个过大任务。

- Task A：Books 列表/详情字段合同与回退规则
- Task B：Books 页面展示增强与状态组件收边
- Task C：Books 详情、筛选与横向回流闭环
- Task D：QA、文档、演示路径与验收收尾

## 10. 文件与代码层面的默认落位原则

- 后端继续按现有 `router / service / repository` 分层落位；不要让路由层承接字段回退规则。
- 前端继续按页面 + feature + shared components 落位；不要复制一份完全独立的书籍详情系统。
- 若需要书籍特化展示，优先在 Books feature 内局部实现；只有真正跨页面可复用时再抽到 shared components。
- 桌面层只承接最小 open actions；不在本阶段新增阅读相关桌面逻辑。

## 11. 明确禁止项（Codex 不得擅自扩展）

- 不得把 Books 做成阅读器、在线书架、阅读社区、知识管理中心。
- 不得引入阅读进度、批注、摘录、云同步、推荐系统。
- 不得创建脱离 `files` 的独立书籍事实层、重对象中台或复杂异步解析平台。
- 不得为了“未来扩展”先做大而空的抽象层；当前阶段只做轻量书库入口所需抽象。

## 12. 人工验收清单

- 可以从导航进入 Books。
- 首屏展示已明显具备“轻量书库”气质，而不是单纯文件列表。
- 单击能进 shared details；双击或动作按钮能触发最小打开动作。
- 封面、标题、作者、格式在“可提取时”明显改善定位体验；提取失败时回退可接受。
- 与 Search、Tags、Collections 的关系清晰，没有形成孤岛。
- 当前状态文档 / 阶段执行记录已同步，未把假设写成既定事实。

## 13. 当前仍需人工拍板，但不阻塞首轮编码的点

- 首版是否显式区分 PDF / EPUB 两个子范围。
- 封面缺失时的默认占位样式与信息密度。
- 作者 / 标题字段回退的最终显示顺序。
- Route 命名与导航文案是否最终冻结。
- 是否允许在本阶段补少量 format 相关筛选能力。

## 14. 阶段完成后的下一步

阶段三-A完成后，不应直接跳去稳定性总收口，而应继续进入《软件库增强版（Codex 执行版任务文档）》以及后续的“组织层与库体验补齐”阶段，继续沿着“先补垂类库，再补横向组织层”的顺序推进。

注：本执行版基于当前项目文档与上游范围草案整理，默认服务于“推动编码开始”的目标，因此含有少量临时执行口径；后续若有人工作出更严格冻结版，应以新的人工冻结为准。
