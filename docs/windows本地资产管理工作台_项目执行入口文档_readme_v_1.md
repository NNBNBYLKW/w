## 2. 项目一句话定义

> **这是一个面向 Windows 海量本地文件的现代化资产管理工作台，不以替代资源管理器为第一目标，而是优先建立本地索引、标签系统、素材库浏览、统一详情层与整理入口。**

当前主线是：

> **扫描源接入 → 文件索引 → 搜索 → 详情 → 标签 / 颜色标签 → 素材库浏览 → 最近导入整理 → 打开原文件 / 打开目录**

这是整个项目第一阶段唯一必须打透的核心链路。

---

## 3. 当前阶段定位

### 3.1 当前不是在做什么
当前阶段**不是**在做：
- 完整资源管理器替代品
- 云同步平台
- 企业协作型 DAM
- AI 自动标签平台
- 游戏启动器 / 电子书管理器 / 软件中心的完整垂类产品

### 3.2 当前真正要做什么
当前阶段真正要做的是：
- 搭建本地应用服务与前端工作台骨架
- 建立扫描源、文件索引、搜索、标签、素材库的主链
- 验证大文件量场景下是否成立
- 形成一个可演示、可试用、可继续扩展的 MVP

### 3.3 当前判断项目是否成功的标准
只要用户能顺畅完成以下闭环，第一阶段就成立：

1. 添加扫描源
2. 触发扫描
3. 搜索到文件
4. 查看详情
5. 添加标签 / 颜色标签
6. 在素材库或标签页再次找到它
7. 打开原文件或所在目录

1. 更快找到内容
2. 更灵活组织内容
3. 更舒服地浏览内容

## 7. 当前最推荐的执行顺序

如果从现在开始真正推进项目，建议严格按以下顺序：

### Step 1：先做 PoC 验证
优先验证：
1. 扫描性能
2. SQLite 查询性能
3. 图片缩略图缓存
4. 视频封面提取
5. 前端大列表 / 大网格渲染

原因：
这些是本项目最容易翻车的底层点。

### Step 2：同时搭前后端骨架
前端：
- AppShell
- Onboarding / Search / Media / Recent 页面壳
- DetailsPanelFeature
- SearchFeature

后端：
- main
- db session / models
- sources route / service / repository
- scanning service / worker
- search route / querying service

### Step 3：先打通三条最小主链
#### 主链 A
Onboarding → 添加扫描源 → 触发扫描

#### 主链 B
Search → 搜索结果 → 详情侧栏 → 打开文件

#### 主链 C
Media Library → 浏览素材 → 标签 / 颜色标签 → 再找回

### Step 4：补最近导入页与标签页
目的：
把产品从“查找工具”提升为“整理入口”。

### Step 5：统一状态、交互、空状态与页面壳
目的：
把功能集合收口成真正的 MVP 产品。

2. 不做右键菜单 / Shell 深度集成
3. 不做云同步

5. 不做 AI 自动标签
6. 不做 OCR / 视频语义识别
7. 不做复杂批量文件操作
8. 不做重型游戏库 / 电子书库 / 软件库深挖

这些方向并非永远不做，而是当前阶段做了会显著偏航。

## 11. 当前最关键的工程约束

### 11.1 前端约束
- App Shell 必须统一
- DetailsPanelFeature 必须跨页面复用
- 搜索、标签、筛选、空状态组件必须尽早统一
- 全局 store 必须克制
- 页面自己的筛选/排序/分页优先页面局部管理

### 11.2 后端约束
- router 只接参数，不承载业务逻辑
- service 是业务中心
- repository 保持克制，只做数据访问
- scanning / thumbnails / metadata 属于 worker 执行层，不接管业务语义
- 第一阶段任务运行时保持简单，不引入复杂调度系统

### 11.3 数据约束
- FileItem 是统一中心对象
- 标签、颜色标签必须持久化
- 不把 files 表做成巨型宽表
- API 参数风格与响应风格要统一

## 15. 当前项目的“唯一优先句”

如果整个项目只允许记住一句话，应该记住这一句：

> **先把本地文件资产的“找回、标记、再找回、可浏览”做透，不要急着把它做成另一个 Explorer、另一个 Steam、另一个 Calibre、另一个 Everything 的叠加体。**

这句话应该被视为当前阶段的最高方向约束。
