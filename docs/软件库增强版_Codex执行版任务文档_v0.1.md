# 软件库增强版（Codex 执行版任务文档）v0.1
> 用于阶段三-B开发执行、任务拆分、验收与交付

## 1. 文档目的与使用方式

本文档不是范围讨论稿，而是交给 Codex 使用的执行版任务文档。它的职责是把《软件库增强版（范围草案）》压缩为可实施的任务包、顺序、约束与验收口径。

若本执行版与后续人工冻结结论冲突，以新的人工冻结为准；在此之前，Codex 应按本文默认口径实施。

## 2. 上游固定约束（不得改写）

- 当前项目仍是建立在真实文件系统之上的 indexed-files workbench；`files` 是统一索引对象，Software 只能作为 subset surface 扩展，不得另起独立事实层。
- shared details 仍是跨页面详情与动作中心；软件库不得分叉出独立的重详情系统。
- Search / tags / color tags / collections 仍是横向复用组织层，Software 必须接入这条复用链而不是脱离它自行闭环。
- 桌面打开动作仍由现有薄壳 open actions 承接；本阶段不新增安装管理中心、卸载中心、平台化 launcher hub 或软件更新系统。

## 3. 本执行版采用的暂定口径（用于推动编码）

由于用户尚未对所有细节逐条拍板，本文采用一组“可执行但保守”的暂定口径，目的是先把阶段三-B推进起来，同时尽量避免后续返工。

默认口径如下：

- Software 首版继续围绕当前 `.exe` / `.msi` / `.zip` 等 software-related files 子集增强，不主动扩到平台账户、安装状态等更深能力。
- 软件库主视觉优先级采用“图标/标题 → 类型提示 → 路径回退信息”的顺序。
- 可稳定显示时增强图标或展示字段；无法增强时统一回退到文件名与扩展名，不做激进猜测。
- 详情页仍使用 shared details；软件特化信息只作为该详情体系里的增强区块出现。
- 后端允许做少量字段增强与排序/筛选补强，但不允许新建独立软件事实层。

## 4. 执行顺序总览

执行顺序固定为 4 个工作包。除非遇到严重阻塞，不建议跳序并行重叠过多。前一个工作包至少应达到“接口与边界已稳定”的状态，才进入后一个工作包。

- WP-1：识别展示字段与后端合同
- WP-2：前端页面与展示基线
- WP-3：动作、筛选与回流闭环
- WP-4：QA、文档、演示路径与验收

## 5. WP-1：识别展示字段与后端合同

目标：先把“Software 列表与详情展示哪些字段、增强失败如何回退、接口吐什么字段”冻结成代码与文档都能共同依赖的基础合同。

### 5.1 必做项

- 明确 Software 列表最小展示字段：`id`、`name/title`、`extension/type`、`icon_or_fallback`、可选轻量分类提示。
- 明确字段回退顺序：
  - 标题：优先使用更自然的显示名；无法稳定获得时回退到文件名。
  - 图标：能拿到则显示，拿不到则统一占位。
  - 类型提示：优先按当前 subset 识别结果回显，不做复杂安装状态判断。
- 明确 Software 列表接口在现有 subset route 基础上的最小增强方式。
- 明确 `GET /files/{id}` 中软件详情增强区块的返回口径。
- 明确哪些筛选/排序是本阶段支持的，哪些后置。

### 5.2 可接受实现方式

- 在现有 Software route 上做轻量字段补强。
- 通过现有详情主链扩展最小软件展示字段。
- 允许使用本地可获得的轻量图标/展示字段映射，但不能引入复杂平台识别系统。

### 5.3 禁止项

- 不得创建脱离 `files` 的 `software` 主事实表来替代现有主链。
- 不得把联网元数据抓取、安装状态全识别写成成立前提。
- 不得为了未来扩展预建复杂软件资产中台。

## 6. WP-2：前端页面与展示基线

目标：把 Software 从“扩展名子集列表”拉到“个人工具库入口”的视觉和交互基线。

### 6.1 必做项

- 增强 Software 页面展示密度，使首屏更像工具库而不是筛选结果。
- 图标、标题、类型提示的层级清晰，路径回退信息不喧宾夺主。
- 统一空状态、无结果态、加载骨架态。
- 保持单击查看详情、双击打开文件的统一交互规则。

### 6.2 优先复用

- 复用现有 shared components、RightDetailsPanel、FilterBar、Tagging 相关组件。
- 若需要软件特化卡片样式，优先在现有卡片体系中做轻量特化，而不是新造一整套页面系统。

### 6.3 禁止项

- 不得在本阶段引入安装器管理面板、卸载流程、运行参数中心。
- 不得把 Software 页面改造成平台化 launcher hub。

## 7. WP-3：动作、筛选与回流闭环

目标：让软件库从“能看”推进到“能轻量使用并再次找回”。

### 7.1 必做项

- 让 Software 页面内的标签、颜色标签、类型筛选、基础排序可自然使用。
- 在 shared details 中展示软件增强字段，并保持标签/颜色标签/打开动作在当前页就地完成。
- 强化 `Open file` 与 `Open containing folder` 在软件场景中的可达性。
- 打通 Software 与 Search、Tags、Collections、Recent 的回流关系。
- 至少验证一条“从 Software 归类后，再从 Tags/Collections/Search 找回”的闭环。

### 7.2 暂不纳入

- 收藏 / 状态 / 评分全量体系。
- 安装状态、版本管理、更新检测。
- 平台账户、运行统计、多启动项复杂配置。

### 7.3 结果要求

- 用户不需要频繁回退到 Files 才能完成工具定位与归类。
- 软件库的“浏览 → 详情 → 打开/归类 → 再找回”路径应明显顺滑于当前 subset v1。

## 8. WP-4：QA、文档、演示路径与验收

目标：把阶段三-B从“功能片段”收成可交付、可演示、可继续承接组织层补齐阶段的切片。

### 8.1 必做项

- 补齐 Software 相关页面的关键空状态、无结果态、加载态校验。
- 验证无图标、标题增强失败、只剩路径回退等场景。
- 验证 shared details、tags、collections、Recent、Search 回流是否正常。
- 更新阶段文档与 current-state slice 记录，避免把假设写成既定事实。

## 9. Codex 任务拆分建议

建议把阶段三-B拆成 3 个主任务 + 1 个收尾任务，而不是一次性交给 Codex 一个过大任务。

- Task A：Software 列表/详情字段合同与回退规则
- Task B：Software 页面展示增强与状态组件收边
- Task C：Software 动作、筛选与横向回流闭环
- Task D：QA、文档、演示路径与验收收尾

## 10. 文件与代码层面的默认落位原则

- 后端继续按现有 `router / service / repository` 分层落位；不要让路由层承接展示字段回退规则。
- 前端继续按页面 + feature + shared components 落位；不要复制一份完全独立的软件详情系统。
- 若需要软件特化展示，优先在 Software feature 内局部实现；只有真正跨页面可复用时再抽到 shared components。
- 桌面层只承接最小 open actions；不在本阶段新增软件管理类桌面逻辑。

## 11. 明确禁止项（Codex 不得擅自扩展）

- 不得把 Software 做成安装管理器、软件中心、平台化启动器、商店或更新中心。
- 不得引入卸载、更新、运行统计、平台账户绑定、复杂启动配置。
- 不得创建脱离 `files` 的独立软件事实层、重对象中台或复杂异步识别平台。
- 不得为了“未来扩展”先做大而空的抽象层；当前阶段只做轻量软件库入口所需抽象。

## 12. 人工验收清单

- 可以从导航进入 Software。
- 首屏展示已明显具备“个人工具库”气质，而不是单纯文件列表。
- 单击能进 shared details；双击或动作按钮能触发最小打开动作。
- 图标、标题、类型提示在“可增强时”明显改善定位体验；增强失败时回退可接受。
- 与 Search、Tags、Collections、Recent 的关系清晰，没有形成孤岛。
- 当前状态文档 / 阶段执行记录已同步，未把假设写成既定事实。

## 13. 当前仍需人工拍板，但不阻塞首轮编码的点

- 首版纳入的 software-related files 边界是否继续严格限定在 `.exe / .msi / .zip`。
- 无图标时的默认占位样式与信息密度。
- 标题增强失败时的最终显示顺序。
- Route 命名与导航文案是否最终冻结。
- 是否允许在本阶段补少量类型相关筛选能力。

## 14. 阶段完成后的下一步

阶段三-B完成后，不应直接跳去稳定性总收口，而应继续进入《组织层与库体验补齐（Codex 执行版任务文档）》以及后续的测试版收口阶段，继续沿着“先补垂类库，再补横向组织层”的顺序推进。

注：本执行版基于当前项目文档与上游范围草案整理，默认服务于“推动编码开始”的目标，因此含有少量临时执行口径；后续若有人工作出更严格冻结版，应以新的人工冻结为准。
