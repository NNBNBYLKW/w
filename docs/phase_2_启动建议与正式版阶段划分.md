# Phase 2 启动建议与正式版阶段划分

## 1. 目的

本文档用于回答三个问题：

1. 什么时候可以开始 Phase 2
2. Phase 2 应该实现哪些功能
3. 从当前状态到正式版，建议按几个大阶段推进

本文档基于当前仓库已完成的 MVP 主链状态，以及现有产品文档中的宏观路线图与功能优先级口径，给出一个更适合当前项目推进的阶段建议。

---

## 2. 当前阶段判断

当前仓库已经完成并验收通过的主链包括：

- 扫描源与索引
- 首扫 / 重扫 delete-sync
- 搜索页
- 详情侧栏
- 普通标签与颜色标签
- 全部文件页
- source / exact-directory browse
- 素材库页
- 最近导入页
- 桌面打开动作
- Home / Settings / Tags 三个主入口收口

从产品推进视角看，当前版本已经不再是“脚手架 + 零散功能点”，而是：

> **Phase 1 收口版 / MVP v1 基线**

这意味着：
- 当前可以先做一次 Phase 1 freeze
- 然后再进入下一层产品能力扩展
- 不建议继续无限拉长 Phase 1 范围

---

## 3. 什么时候可以开始 Phase 2

## 结论

> **现在就可以开始准备 Phase 2，但建议先完成一次 Phase 1 冻结。**

### 建议的启动前条件

进入 Phase 2 前，建议确认以下事项已经完成：

1. 当前 MVP 主链已完成完整回归
2. 当前版本已形成验收结论文档
3. 当前版本已形成已知问题与范围外能力清单
4. 当前版本已形成下一阶段路线建议
5. 当前版本的“Phase 1 收口版”边界已冻结

### 不建议直接开始 Phase 2 的情况

如果以下任意情况仍未稳定，建议不要立即进入 Phase 2：

- Search / Files / Media / Recent 仍存在主链级 bug
- Home / Settings / Tags 仍未真正可用
- 桌面打开动作仍不稳定
- 当前文档与代码状态严重不一致

### 当前建议

按目前仓库进度，建议将：

> **当前版本定义为：Phase 1 收口版 / MVP 正式试用基线**

然后开始进入：

> **Phase 2：统一库体验增强**

---

## 4. Phase 2 应该实现什么

## Phase 2 的核心目标

Phase 2 的任务不是重写当前 MVP，而是把当前已经成立的：

- 文件层
- 元数据层
- 素材库层

继续往“统一库体验”推进。

### Phase 2 的产品主轴

建议明确为：

> **从“现代文件库 + 素材库”扩展到“统一本地资产工作台”。**

也就是在现有图片/视频素材库之外，开始让：

- 游戏
- 电子书
- 软件

也进入同一套本地元数据与库式浏览体系。

---

## 5. 建议的 Phase 2 功能范围

### 5.1 必须进入 Phase 2 的能力

#### A. 游戏库（轻量版）
目标：
- 把本地游戏纳入统一库视图

建议第一批能力：
- 游戏库页
- 基础卡片展示
- 启动入口
- 状态字段（如：在玩 / 通关 / 搁置）
- 标签接入

当前不建议第一批就做：
- 复杂平台识别
- 游玩时长
- 成就
- 联网抓取大量元数据

---

#### B. 电子书库（轻量版）
目标：
- 让 PDF / EPUB / MOBI / AZW3 等文件进入可浏览书库

建议第一批能力：
- 电子书库页
- 封面或占位封面展示
- 标题 / 作者基础字段（能识别则展示）
- 阅读状态（未读 / 在读 / 已读）
- 打开外部阅读器

当前不建议第一批就做：
- 深度书籍元数据生态
- 复杂目录提取
- 云端书籍资料抓取

---

#### C. 软件库（轻量版）
目标：
- 把免安装软件 / 工具集纳入统一库入口

建议第一批能力：
- 软件库页
- 图标 / 卡片展示
- 启动入口
- 常用标记 / 分类管理
- 标签接入

当前不建议第一批就做：
- 复杂版本管理
- 安装器生态
- 软件商店式能力

---

#### D. 智能集合 / 保存搜索条件（基础版）
目标：
- 让用户把常用筛选组合固化为长期入口

建议第一批能力：
- 保存一个筛选条件组合
- 在导航中以集合入口再次进入
- 至少支持基于标签 / 类型 / 颜色标签 / 路径或来源的组合

当前不建议第一批就做：
- 复杂自动化规则引擎
- 高级布尔查询构建器

---

## 6. 不建议在 Phase 2 一开始就做的内容

以下方向建议明确延后：

- AI / semantic search
- OCR / 视频内容识别
- 重在线元数据抓取依赖
- Explorer 替代级目录树系统
- 批量操作系统大扩张
- 大型 Dashboard 改版
- 插件系统
- 多端同步 / 云同步

原因：
- 会显著抬高复杂度
- 会稀释当前“统一库体验增强”的主轴
- 会让项目再次进入范围膨胀

---

## 7. 从当前状态到正式版，建议按几个大阶段推进

## 建议采用 4 个大阶段口径

### Phase 1：可用的现代文件库
目标：
- 扫描目录
- 建立索引
- 搜索
- 标签
- 图片/视频库基础浏览
- 打开文件与打开所在目录
- MVP 主入口成立

当前判断：
> **已基本完成，可视为当前收口阶段。**

---

### Phase 2：统一库体验增强
目标：
- 游戏库
- 电子书库
- 软件库
- 封面、状态、详情页
- 智能集合基础版

当前判断：
> **下一阶段应进入的主轴。**

---

### Phase 3：高级组织能力
目标：
- 保存搜索条件增强
- 批量整理
- 自动规则
- 更强元数据层组织能力
- 跨库的整理与找回效率增强

当前判断：
> **在统一库体验成立后再进入。**

---

### Phase 4：智能增强（后期）
目标：
- AI 自动标签
- OCR / 字幕 / 元数据抽取
- 内容相似检索
- 更智能的内容识别与辅助组织

当前判断：
> **后期阶段，不应提前拉进 MVP 或 Phase 2。**

---

## 8. 当前更适合怎样定义“正式版”

这里建议分成两个口径：

### 8.1 MVP 正式试用版
条件：
- Phase 1 收口完成
- 当前主链稳定可演示
- 用户能完成：
  - 添加扫描源
  - 扫描
  - 搜索
  - 标签标记
  - 再找回
  - 素材浏览
  - 打开文件/目录

当前建议：
> **当前阶段已经接近这个口径。**

---

### 8.2 更完整的正式版 v1
条件：
- Phase 2 完成
- 不仅有“现代文件库”，也有“统一库体验”
- 游戏 / 电子书 / 软件至少有轻量版可用页
- 智能集合基础版成立

当前建议：
> **更适合在 Phase 2 结束后再定义为“正式版 v1”。**

原因：
- 这样才更贴合产品定位：
  - 不是单纯文件搜索器
  - 也不只是图片/视频素材页
  - 而是统一本地资产工作台

---

## 9. 当前推荐的推进方式

### 推荐口径
当前建议将后续推进定义为：

1. 当前版本 = **Phase 1 收口版 / MVP v1 基线**
2. 下一阶段 = **Phase 2：统一库体验增强**
3. 正式版 v1 = **Phase 2 收口后形成**

### 推荐的 Phase 2 子阶段拆法
为了避免一次性做太宽，建议在 Phase 2 内继续拆小阶段，例如：

- Phase 2A：游戏库轻量版
- Phase 2B：电子书库轻量版
- Phase 2C：软件库轻量版
- Phase 2D：智能集合基础版

这样做的好处：
- 每个阶段都能单独验收
- 不会把 Phase 2 做成一个难以收口的大阶段
- 更适合继续用 plan + Codex + 验收 的方式推进

---

## 10. 当前结论

### 最终建议

#### 什么时候可以开始 Phase 2
> **现在就可以开始准备，但建议先完成一次 Phase 1 freeze。**

#### Phase 2 要实现什么
> **统一库体验增强：重点补游戏库、电子书库、软件库、智能集合基础版。**

#### 一共有几个大 phase 可以开发到正式版
> **建议维持 4 个大阶段：Phase 1 ~ Phase 4。**

#### 什么时候更适合叫正式版
> **建议在 Phase 2 结束后，将产品定义为更完整的正式版 v1。**

---

## 11. 下一步建议

如果继续推进，最推荐的下一步不是直接写代码，而是：

1. 做一份 `Phase 1 freeze / MVP 收口结论`
2. 明确 `Phase 2A` 先做哪一个库
3. 用独立 phase plan 的形式拆成可执行任务
4. 再交给 Codex 做 plan 与实现

当前最适合作为下一个可执行阶段的是：

> **Phase 2A：游戏库轻量版**

或

> **Phase 2A：智能集合基础版**

具体取决于你更偏向：
- 先补“内容库统一体验”
- 还是先补“组织层闭环”
