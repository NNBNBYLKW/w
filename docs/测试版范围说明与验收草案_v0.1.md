# 测试版范围说明与验收草案 v0.1

## 1. 文档目的

本文档用于定义当前项目进入 **测试版（Beta）** 之前的范围冻结口径、验收口径与反馈采集口径。

本文件不是长期产品愿景文档，也不是具体开发任务拆分文档。  
它的作用是：

- 说明 **什么能力必须进入测试版**
- 说明 **什么能力可以明确后置**
- 说明 **测试版的验收标准是什么**
- 说明 **测试版对外验证的重点问题是什么**
- 为后续稳定性 / 性能 / 正式版收口阶段提供边界

---

## 2. 当前测试版定位

当前测试版的目标不是把产品一次性做成“完整成熟版”，也不是把所有历史草案能力全部塞进第一轮对外验证。

测试版的定位应明确为：

> 一个围绕本地文件系统、以 indexed files 为底座、已经具备多垂类库入口与横向组织能力的  
> **Windows local-first 本地资产工作台测试版**

测试版需要验证的不是“功能数量是否足够多”，而是以下问题：

1. 用户是否真的会把它当作本地资产入口来使用
2. 垂类库（媒体 / 游戏 / 电子书 / 软件）是否已经形成可感知差异
3. 标签、颜色标签、集合、最近链路是否足以支撑持续整理
4. shared details + open actions 这条复用链是否足够顺手
5. 当前方法论是否成立：  
   `files + subset surfaces + shared details + tags/color tags/collections`

---

## 3. 测试版前置阶段回顾

在进入测试版前，默认已经完成以下几条后续开发线的最小收口：

1. 媒体库增强版
2. 游戏库首版
3. 电子书库增强版
4. 软件库增强版
5. 组织层与库体验补齐（至少第一轮）

若上述任一阶段仍停留在“占位页 / 仅有命名入口 / 基本不可感知”的状态，则不建议宣布进入测试版。

---

## 4. 测试版核心目标

测试版必须同时满足两类目标：

### 4.1 产品目标
- 用户能明确感知“这不是普通文件列表工具”
- 用户能理解这是一个“本地资产工作台”
- 用户可以依靠垂类库入口进行内容浏览，而不是只靠路径和文件名
- 用户可以在浏览、标记、再找回之间形成闭环

### 4.2 验证目标
- 验证哪些垂类入口最有价值
- 验证用户是否会自然使用标签、颜色标签、集合、最近链路
- 验证 shared details 是否足够承担跨 surface 的统一整理中心角色
- 验证哪些能力值得进入正式版收口优先级，哪些能力应延后

---

## 5. 测试版必须包含的能力（Must Have）

## 5.1 基础底座能力
以下能力必须是稳定可用的：

- source setup / source management
- scan / index 主链
- 基于 indexed files 的统一事实层
- Search
- Files
- shared details
- open file / open containing folder
- tags
- color tags
- collections
- recent imports baseline

这些不是测试版卖点本身，但它们是测试版成立的基础。

---

## 5.2 垂类库能力

### 5.2.1 媒体库
测试版中的媒体库至少应具备：

- 明显的卡片流 / 缩略图浏览体验
- 图片 / 视频内容的高可见度浏览方式
- 标签、颜色标签、基础筛选和排序入口
- 与 shared details 的自然协同
- 从浏览到标记到再找回的闭环

### 5.2.2 游戏库
测试版中的游戏库至少应具备：

- 独立的游戏库入口
- 封面 / 图标 / 标题优先浏览
- 最小启动或打开入口
- 最小状态型组织方式（例如在玩 / 通关 / 搁置）
- 与 Search / tags / collections / shared details 的回流关系

### 5.2.3 电子书库
测试版中的电子书库至少应具备：

- 独立的书库入口
- 封面 / 标题 / 作者 / 书籍基础信息的轻量书库浏览体验
- 标签与集合入口
- 外部打开动作
- 与 shared details 和 Search 的回流关系

### 5.2.4 软件库
测试版中的软件库至少应具备：

- 独立的软件库入口
- 图标 / 标题 / 基础操作优先展示
- 快速打开或启动入口
- 标签 / 状态 / 集合的基础组织方式
- 与 shared details 和 Search 的回流关系

---

## 5.3 横向组织层能力
测试版必须具备第一轮可感知的横向组织层，而不是只有零散标签能力：

- 普通标签
- 颜色标签
- 批量标签（至少一轮）
- 批量颜色标签（至少一轮）
- collections / 保存型检索入口
- recent family 中至少包含：
  - recently indexed / recently imported
  - 与整理动作有关的基础回流入口

若收藏 / 状态 / 评分已进入，则可以纳入测试版；若尚未达到可感知程度，不应为了“功能更全”而硬塞。

---

## 5.4 首页与入口组织
测试版首页不要求成为复杂 dashboard，但至少应满足：

- 能作为总入口
- 能引导用户进入各核心库页
- 能展示 recent / source status / quick links 等最低限摘要
- 不让用户第一次进入时迷失在一堆平级页面中

---

## 5.5 关键页面状态统一
测试版前必须统一以下关键状态体验：

- 空状态
- 加载状态
- 无结果状态
- 基础错误提示
- 详情侧栏选中 / 未选中状态
- 关键操作反馈（如打标签、设置颜色标签、打开失败）

测试版不要求设计系统完全成熟，但这些状态不能再显得明显未收边。

---

## 6. 测试版明确后置的能力（Deferred）

以下能力不应作为测试版进入门槛：

### 6.1 明确后置
- 阅读器化
- 启动器 / 商店平台化
- 安装管理器 / 卸载中心
- 包管理器化
- AI 自动标签
- OCR / 字幕 / 语义识别
- 图像相似检索
- 视频内容理解
- 平台账户同步
- 游戏时长 / 成就系统
- 复杂自动化规则引擎
- 重外部 metadata 平台
- 统一重对象模型中台

### 6.2 可后置但允许进入正式版候选池
- 更强的收藏 / 状态 / 评分体系
- 更强的智能集合
- 更完整的 recent family
- 更丰富的元数据显示
- 更成熟的封面 / 缩略图缓存体系

---

## 7. 测试版关键用户路径

测试版至少要能支撑以下几条连续路径：

### 路径 A：首次使用
1. 添加 source
2. 触发 scan
3. 看到入库结果
4. 从首页进入某个核心 surface

### 路径 B：搜索找回
1. 输入关键词
2. 在 Search 中缩小范围
3. 进入 details
4. 打开文件 / 打开目录

### 路径 C：媒体整理闭环
1. 进入 Media
2. 靠缩略图浏览内容
3. 给素材打标签 / 颜色标签
4. 通过标签或集合再次找到内容

### 路径 D：垂类库使用
1. 进入 Games / Books / Software 任一库
2. 浏览条目
3. 查看详情
4. 进行打开 / 启动 / 外部打开动作
5. 通过组织层再次找到内容

### 路径 E：最近链路整理
1. 进入 Recent
2. 对最近进入系统的一批内容做整理
3. 再通过库页 / Tags / Collections 找回

若这些路径中的两条以上需要依赖隐藏操作、非常规路径或明显未完成的页面，测试版不应冻结。

---

## 8. 测试版验收口径

## 8.1 产品验收口径
以下问题大多数都能回答“是”，才应进入测试版：

1. 用户是否能在 10 分钟内完成 source setup 并看到内容进入系统？
2. 用户是否能明显快于传统目录记忆方式找回目标文件？
3. 媒体库是否已经具备明显的视觉浏览优势？
4. 游戏库是否已经是一个真实可用入口，而不是占位页？
5. 电子书库和软件库是否已经超出纯扩展名子集列表的感知？
6. 用户是否能顺手完成标签 / 颜色标签等整理动作？
7. details 是否仍然是统一、顺手、可复用的整理中心？
8. recent / tags / collections 是否已经不是“附属页”，而是真能回流的入口？

---

## 8.2 体验验收口径
测试版不要求完美，但以下问题不能过于突出：

- 首屏明显卡顿
- 页面层级混乱
- 多个库入口视觉和交互语言割裂
- 详情侧栏频繁失效或状态错乱
- 卡片流与缩略图大量异常
- 打开动作高频失败
- 标签/颜色标签操作反馈不清晰
- 空状态和加载状态像未完成占位

---

## 8.3 边界验收口径
测试版必须仍然保持以下边界：

- 没有滑向阅读平台
- 没有滑向 launcher 平台
- 没有滑向安装管理器
- 没有滑向 AI / OCR / 语义平台
- 没有为了局部垂类引入重中台对象层
- 没有分叉 shared details 体系

只要这些边界被明显破坏，就不应视为“测试版更完整”，而应视为偏航。

---

## 9. 测试版对外验证重点问题

对外验证时，建议重点问以下问题，而不是泛泛问“你喜不喜欢”：

### 9.1 入口与定位
- 你会把它理解成什么？
- 你会不会愿意把它作为本地资产入口之一？

### 9.2 垂类库价值
- 哪个库最有价值：媒体、游戏、电子书还是软件？
- 哪个库目前仍显得太薄？

### 9.3 组织层价值
- 你是否真的会使用标签 / 颜色标签？
- 你会不会使用 collections / recent 作为回流入口？

### 9.4 details 与动作价值
- 你是否能自然理解 details 是统一整理中心？
- 打开文件 / 打开目录是否足够顺手？

### 9.5 正式版优先级
- 如果只能优先做 2~3 项增强，你最希望先补什么？

---

## 10. 测试版反馈收集建议

建议至少按以下维度收集反馈：

- 用户背景（素材用户 / 收藏型用户 / 效率型用户）
- 使用的主要 surface
- 最常使用的回流路径
- 最常见的卡点
- 最常要求补强的垂类库
- 对标签 / 颜色标签 / 集合 / recent 的实际使用情况
- 是否愿意继续长期使用

反馈不应只按“Bug / 非 Bug”二分，而应区分：

1. 路径性问题
2. 组织层问题
3. 库页价值问题
4. 稳定性问题
5. 性能问题
6. 偏航需求

---

## 11. 测试版发布前最小检查清单

### 11.1 功能检查
- Source setup 正常
- Scan / index 正常
- Search 正常
- Files 正常
- Media 正常
- Games 正常
- Books 正常
- Software 正常
- Tags / color tags 正常
- Collections 正常
- Recent 正常
- Details / open actions 正常

### 11.2 体验检查
- 首页入口清晰
- 关键页面无明显残缺空洞
- 状态组件统一
- 主要路径可连续完成

### 11.3 边界检查
- 未引入测试版外高风险功能
- 未为局部需求破坏整体方法论

---

## 12. 测试版退出条件

当满足以下条件时，可认为测试版阶段成立并可进入下一阶段：

1. 测试版范围已冻结
2. 核心路径均可完成
3. 垂类库能力已具备可感知差异
4. 横向组织层已具备第一轮可用性
5. shared details 与 open actions 已形成稳定复用中心
6. 已经具备对外验证材料与问题清单
7. 已有真实用户反馈可进入后续优先级决策

---

## 13. 测试版后的自然下一步

测试版阶段之后，后续主线应进入：

> **稳定性 / 性能 / 剩余草案功能收口阶段**

但前提不是“马上大重构”，而是：

- 基于测试版真实反馈
- 压高优先级稳定性问题
- 压明显性能问题
- 把真正被验证有价值的能力推进到正式版优先级

---

## 14. 当前待拍板项

本草案仍需进一步冻结以下事项：

1. 测试版是否必须包含收藏 / 状态 / 评分中的哪几项
2. 游戏库首版的状态体系是否进入测试版必须项
3. 首页摘要要做到什么信息密度
4. recent family 在测试版中具体保留哪些入口
5. 对外测试版是否只面向 Windows 本地用户
6. 测试版期间是否允许继续新增功能，还是仅允许修正和收边

---
