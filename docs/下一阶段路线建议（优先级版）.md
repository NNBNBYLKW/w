# 下一阶段路线建议（优先级版）

## 1. 目的

本文档用于在当前 MVP 主链已经可用的前提下，为后续增量开发提供：

- 候选方向的优先级排序
- 每个方向的用户价值判断
- 每个方向的实现风险判断
- 推荐的推进顺序
- 不建议当前优先启动的方向说明

本文档不直接定义实现细节，而是作为下一阶段立项前的路线参考。

---

## 2. 当前阶段基线

当前版本已经具备以下已验收通过的主链能力：

- source onboarding
- 首扫 / 重扫 delete-sync
- indexed search
- minimal details panel
- normal tags
- per-file color tags
- flat files listing
- source / exact-directory browse
- media listing
- recent imports listing
- shared details-panel open actions

因此，后续路线应优先考虑：

- 在当前主链上叠加用户感知明显的增量价值
- 避免一次性引入新的复杂平台层
- 避免把浏览、搜索、媒体、动作系统混成一个超大阶段

---

## 3. 优先级建议总览

### 第一优先级
1. **Double-click to open**
2. **FilesPage 更强浏览能力**

### 第二优先级
3. **Tag / Color-tag filtering**
4. **Media thumbnails / previews**

### 第三优先级
5. **Recent Imports 增强**
6. **Files / Media / Recent 的统一交互打磨**

### 当前不建议优先启动
- AI / semantic search
- Dashboard 大改
- Explorer 替代级目录树
- batch action 框架
- 深度 shell integration
- 插件化 / 扩展系统

---

## 4. 第一优先级候选项

### 4.1 Double-click to open

#### 目标
在 Search / Files / Media / Recent 的列表或卡片中，为当前已选文件增加统一的双击打开行为。

#### 用户价值
高。

原因：
- 当前已经有 `Open file` / `Open containing folder`
- 用户下一步最自然的直觉操作就是“双击打开”
- 这是桌面文件工具非常高频的交互

#### 实现风险
中低。

#### 原因
- 桥接能力已存在
- 共享详情面板动作已验证通过
- 主要新增的是页面交互层规则，而不是后端或数据库能力

#### 推荐范围
- Search：双击结果打开文件
- Files：双击行打开文件
- Media：双击卡片打开文件
- Recent：双击行打开文件
- 单击仍保持“只选中，不打开”的当前语义
- browser mode 继续 graceful degrade

#### 当前不建议顺手做
- 双击打开目录
- 双击行为自定义设置
- 右键菜单
- batch open

#### 推荐优先级
**最高**

---

### 4.2 FilesPage 更强浏览能力

#### 目标
把当前的 flat list + exact-directory browse，逐步推进为更接近“文件浏览”的体验，但仍保持克制。

#### 用户价值
中高。

#### 原因
- 当前 FilesPage 虽然已成立，但和用户对“按目录浏览文件”的直觉仍有差距
- 这是当前用户体感中最明显的“能力缺口之一”

#### 实现风险
中。

#### 可考虑的后续子方向
- child-directory discovery
- 更清晰的 source-root 下路径导航
- 更自然的 browse affordance
- 更明确区分“全量列表”和“目录浏览视图”

#### 当前不建议直接做
- 目录树
- Explorer 完整替代
- 多栏 Finder 式浏览
- 复杂 breadcrumb 历史导航系统

#### 推荐优先级
**高**

---

## 5. 第二优先级候选项

### 5.1 Tag / Color-tag filtering

#### 目标
把现有普通 tag 与 color tag 接入查询和浏览页面。

#### 用户价值
中高。

#### 原因
- 当前 tag 体系已经建立
- 用户已经可以整理文件，但还不能在搜索/浏览里消费这些组织信息
- 一旦接入过滤，组织能力才真正形成闭环

#### 实现风险
中。

#### 推荐切法
- 先接 Search
- 再接 Files
- 再考虑 Media
- color tag 过滤可以比普通 tag 过滤更晚一点

#### 当前不建议直接做
- 高级多条件过滤器
- tag autocomplete 重设计
- 独立 Tags 管理页大扩张

#### 推荐优先级
**中高**

---

### 5.2 Media thumbnails / previews

#### 目标
让 MediaLibrary 从“真实媒体列表”升级为更有媒体工具感的页面。

#### 用户价值
高。

#### 原因
- 当前 Media 页面已经有真数据
- 但媒体价值尚未完全释放
- 缩略图/预览能显著提升素材浏览效率

#### 实现风险
中高。

#### 风险来源
- 缩略图生成策略
- 缓存
- 失败态
- 列表性能
- 图片/视频差异处理

#### 推荐切法
- 先做图片静态缩略图
- 再做视频封面
- 再看 preview/hover

#### 当前不建议直接做
- 视频悬停播放
- 复杂媒体元数据分析
- 一次性重做整个 Media 页面密度系统

#### 推荐优先级
**中**

---

## 6. 第三优先级候选项

### 6.1 Recent Imports 增强

#### 目标
让 Recent 从“时间窗口列表”进一步变成更强的整理入口。

#### 用户价值
中。

#### 可考虑方向
- 更丰富的时间分组
- 更清晰的 recently indexed context
- 与 tag / color tag 的更顺手联动

#### 风险
中低。

#### 推荐优先级
**中**

---

### 6.2 页面交互一致性打磨

#### 目标
统一 Search / Files / Media / Recent 的交互细节和状态表现。

#### 价值
中。

#### 包括但不限于
- 单击 / 双击语义统一
- loading / empty / error 文案统一
- row / card 选择态统一
- DetailsPanel 信息层级统一

#### 风险
中低。

#### 推荐优先级
**中**

---

## 7. 当前不建议优先启动的方向

### 7.1 AI / semantic search
原因：
- 当前索引与组织主链仍可通过更低风险功能释放更多价值
- 语义搜索会显著提高复杂度和不确定性

### 7.2 Dashboard 大改
原因：
- 当前真正的使用闭环还在完善中
- 过早大改 dashboard 容易分散精力

### 7.3 Explorer 替代级目录树
原因：
- 风险高
- 范围大
- 很容易把项目推向复杂文件管理器，而不是克制的资产工作台

### 7.4 Batch action framework
原因：
- 当前单文件主链还在不断完善
- 批量系统很容易把 UI、动作、状态管理复杂度迅速拉高

### 7.5 深度 shell integration
原因：
- 当前 minimal open actions 已够支撑 MVP
- 更深 shell 集成应在更明确需求后再立项

---

## 8. 推荐推进顺序

### 推荐路线 A（更偏桌面工具手感）
1. Double-click to open
2. FilesPage 更强浏览能力
3. Tag / Color-tag filtering
4. Media thumbnails / previews

适合偏好：
- 更快提升“像桌面文件工具”的自然手感
- 更快减少操作摩擦

---

### 推荐路线 B（更偏资产整理效率）
1. Tag / Color-tag filtering
2. FilesPage 更强浏览能力
3. Recent Imports 增强
4. Media thumbnails / previews

适合偏好：
- 更快提升“整理、归档、找回”效率
- 更快形成组织层闭环

---

## 9. 当前建议结论

如果只选一个最值得先做的方向，建议：

> **Double-click to open**

原因：
- 用户感知最直接
- 风险最低之一
- 与当前已有 open actions 高度复用
- 能快速提升 Search / Files / Media / Recent 四个页面的桌面工具体验

如果你更在意“像资源管理器一样浏览”的方向，则建议优先：

> **FilesPage 更强浏览能力**

---

## 10. 使用建议

后续每次进入下一阶段前，建议按以下顺序推进：

1. 先从本路线文档中选一个候选项
2. 形成单独 phase plan
3. 明确“做什么 / 不做什么”
4. 实现
5. 用《MVP 验收清单打勾版》回归
6. 把结果更新回《MVP 验收总结 v1》与本路线文档

这样可以持续维持：
- 小步迭代
- 阶段可验收
- 范围可控
- 文档与代码一致
