# Windows 本地文件管理与素材库产品文档（草案）

## 1. 产品概述

### 1.1 产品名称（暂定）
- 中文名：待定
- 英文名：待定

### 1.2 产品一句话定位
一款面向 **大量本地文件管理** 的 Windows 桌面应用，将 **高速索引、标签化组织、媒体素材检索、库式浏览、现代化 UI** 融合在同一套本地优先（local-first）的体验中。

### 1.3 产品目标
解决 Windows 原生资源管理器在以下场景中的核心不足：
- 大量文件下索引慢、搜索慢、筛选弱
- 缺乏可视化标签体系（尤其缺乏类似 macOS Finder 的颜色标签）
- 无法像流媒体网站一样基于标签、分类、筛选条件浏览图片和视频素材
- 对游戏、电子书、免安装软件缺乏“库式管理”能力
- 专业文件管理工具常见 UI 偏传统、偏工具化，缺乏现代感和内容浏览体验

---

## 2. 背景与机会

### 2.1 用户背景
目标用户通常拥有以下特点：
- 本地硬盘/移动硬盘中存有大量文件
- 文件类型复杂：图片、视频、游戏、电子书、压缩包、软件、项目文件混杂
- 不满足于简单树状目录，期望“按内容/按用途/按标签”管理
- 既需要效率型管理，也需要内容型浏览
- 对 UI 审美和现代交互有要求

### 2.2 当前痛点
1. **Windows 搜索与索引体验弱**
   - 初始索引慢
   - 非索引目录搜索性能差
   - 对大体量机械盘/多盘符/移动盘支持体验不稳定
2. **文件组织方式过于依赖目录结构**
   - 一个文件只能放在一个物理目录中，难以表达多重属性
   - 无颜色标签、评分、状态等轻量元数据体系
3. **媒体素材管理能力不足**
   - 图片/视频只能按文件名和文件夹找
   - 缺乏标签、筛选器、封面墙、预览流式浏览
4. **游戏/电子书/软件缺乏统一库视图**
   - 游戏散落多个盘符和启动器
   - 电子书缺乏统一封面浏览、作者/系列/标签管理
   - 免安装软件没有类似 Steam 库的统一入口和元数据管理
5. **现有工具审美与交互偏朴素**
   - 重功能轻体验
   - 不适合长期管理自己的“数字资产仓库”

### 2.3 产品机会
本产品可切入为：
- **文件管理器**：替代或增强资源管理器
- **本地资产库**：像媒体库/Steam 库一样管理内容
- **标签化元数据层**：在原始文件系统之上增加一层可检索、可浏览、可运营的“数据层”

## 3. 产品定位

### 3.1 产品本质
不是单纯“更好看的资源管理器”，而是：

> **建立在 Windows 文件系统之上的本地文件资产管理平台**。

### 3.2 核心价值主张
- **比资源管理器更快地找到文件**
- **比目录树更灵活地组织文件**
- **比传统文件管理器更适合浏览内容**
- **比单独的游戏库/电子书库更统一**

### 3.3 关键词
- Local-first
- 高性能索引
- 标签化管理
- 多视图浏览
- 现代 UI
- 素材库
- 库式管理

### 3.4 非目标（阶段性）
- 不做云盘同步平台
- 不做多人协作 DAM（数字资产管理）系统
- 不做完整视频编辑/图片编辑器
- 不做游戏下载平台
- 不做电子书内容制作平台

---

## 4. 目标用户

### 4.1 核心用户画像

#### A. 素材重度用户
- 设计师、视频剪辑、自媒体作者、画师、3D 制作者
- 拥有大量图片、视频、音频、参考资料
- 强依赖缩略图、预览、标签、筛选

#### B. 本地收藏型用户
- 收藏游戏、电子书、软件、资源包、教程资料
- 希望像“数字藏品库”一样管理自己的本地内容

#### C. 效率型重度文件用户
- 大量项目文件、文档、压缩包、多盘符资料管理
- 需要高速搜索、批量整理、分类标记

### 4.2 次级用户画像
- 普通 Windows 用户中对系统原生体验不满的人
- 希望拥有“更现代、更好看、更顺手”的文件入口的人

## 5. 用户核心需求

### 5.1 功能需求
1. 高速索引与搜索
2. 文件标签系统（含颜色标签）
3. 类流媒体网站的素材浏览与筛选
4. 游戏 / 电子书 / 软件库视图
5. 多类型文件统一管理
6. 批量标记、批量整理、批量归档
7. 缩略图、封面、预览、详情信息
8. 自定义分类、智能集合、保存筛选条件

### 5.2 体验需求
1. 启动快、滚动快、搜索响应快
2. 界面现代化，接近内容平台而非陈旧工具
3. 适合长时间浏览和整理
4. 视觉上能区分“文件浏览”和“内容库浏览”两种模式

### 5.3 心理需求
- “这是我的数字仓库”
- “我可以掌控混乱”
- “找东西不用靠记忆目录位置”
- “我的收藏终于有一个像样的展示空间”

## 6. 核心问题定义

### 6.1 需要解决的首要问题
**如何在不改变用户原有目录结构的前提下，建立一套高性能、本地可控、适合浏览和检索的元数据层。**

### 6.2 产品的关键矛盾
1. **文件系统逻辑 vs 内容平台逻辑**
   - 文件系统强调路径、层级、真实位置
   - 内容平台强调标签、推荐、集合、封面、关联
2. **高性能 vs 富交互**
   - 大量缩略图、预览、标签过滤会带来性能压力
3. **通用性 vs 垂直体验**
   - 一套系统要管理图片、视频、游戏、电子书、软件，抽象层需要设计得足够稳

## 8. 核心功能设计

### 8.1 索引系统

#### 目标
建立比 Windows 原生更快、更可控的索引机制。

#### 索引内容
- 文件路径
- 文件名
- 扩展名
- 文件大小
- 修改时间 / 创建时间
- 所属目录
- 自定义标签
- 颜色标签
- 评分 / 收藏状态
- 文件类型识别（图片/视频/游戏/电子书/软件/压缩包等）
- 媒体元数据（分辨率、时长、编码、封面）
- 电子书元数据（标题、作者、系列、封面）
- 游戏/软件启动信息

#### 索引方式建议
- 首次全盘扫描 + 增量监听
- 用户可选择扫描目录
- 支持忽略目录/文件类型
- 支持后台低优先级构建索引
- 支持手动重建索引

### 8.2 标签系统

#### 标签类型
1. 普通标签（如：教程、待整理、参考、已看）
2. 颜色标签（如：红=紧急，黄=待处理，绿=完成，蓝=收藏）
3. 分类标签（如：游戏、电影感、赛博朋克、建模参考）
4. 状态标签（如：未看、已整理、待归档）

#### 标签能力
- 单文件多标签
- 批量打标签
- 标签组合筛选
- 排除标签筛选
- 保存标签筛选条件

### 8.3 搜索与筛选

#### 搜索方式
- 全局搜索
- 当前库内搜索
- 标签搜索
- 组合条件搜索

#### 可筛选条件
- 文件类型
- 标签
- 颜色标签
- 文件大小范围
- 时间范围
- 分辨率 / 时长（媒体）
- 作者 / 系列（电子书）
- 是否已收藏 / 已评分
- 所在磁盘 / 路径

#### 高级搜索表达式（中后期）
例如：
- `type:image tag:赛博朋克 color:red`
- `library:game installed:true`
- `type:video duration>10m tag:素材`

### 8.4 素材库（图片 / 视频）

#### 浏览方式
- 瀑布流
- 网格卡片
- 时间线
- 标签分组
- 相册/集合

#### 详情信息
- 大图预览 / 视频预览
- 路径
- 标签
- 分辨率 / 时长 / 大小
- 来源目录
- 相关文件

#### 重点体验
- 像流媒体网站一样通过标签快速缩小范围
- 像本地媒体库一样通过缩略图浏览

### 8.5 游戏库

#### 目标
为本地游戏建立类似 Steam 库的浏览页。

#### 功能
- 封面墙显示
- 游戏详情页
- 启动入口
- 自定义封面/背景图
- 平台分类（Steam / 模拟器 / 独立本地游戏 / 压缩游戏）
- 状态管理（已玩 / 在玩 / 通关 / 搁置）
- 标签（合作 / 单机 / RPG / FPS 等）

### 8.6 电子书库

#### 功能
- 封面墙浏览
- 书籍详情页
- 标题 / 作者 / 系列 / 标签管理
- 阅读状态（未读 / 在读 / 已读）
- 分类（技术 / 小说 / 艺术 / 历史等）
- 调用外部阅读器打开

### 8.7 软件库（免安装软件）

#### 功能
- 图标/卡片展示
- 软件说明、版本、来源路径
- 快速启动
- 分类（设计 / 开发 / 工具 / 娱乐）
- 状态（常用 / 测试 / 归档）

### 8.8 现代 UI 设计目标

#### 视觉方向
- 深色模式优先
- 大量卡片式布局
- 统一圆角、间距、动效
- 内容优先，弱化传统“表格工具感”

#### 交互原则
- 浏览优先
- 搜索随时可达
- 筛选器固定可见
- 详情面板可快速展开/收起
- 文件模式与库模式清晰区分

## 10. MVP 范围建议

### 10.1 MVP 核心目标
先验证三件事：
1. 海量文件下索引和搜索是否足够快
2. 标签 + 库视图是否足够有价值
3. 用户是否愿意长期把它当作“数字仓库入口”

### 10.2 MVP 功能
- 指定目录扫描与索引
- 全局搜索
- 标签系统（普通标签 + 颜色标签）
- 图片/视频基础素材库
- 游戏/电子书/软件的基础库卡片页
- 文件详情页
- 深色现代 UI

### 10.3 暂不进入 MVP 的功能
- AI 自动打标签
- OCR / 视频内容识别
- 自动元数据抓取网络信息
- 云同步
- 多端协同
- 推荐系统

## 11. 信息架构与数据模型（简化）

### 11.1 核心对象
1. 文件（FileItem）
2. 标签（Tag）
3. 文件标签关系（FileTagRelation）
4. 库项目（LibraryItem）
5. 收藏/评分/状态（UserMeta）
6. 索引任务（IndexTask）

### 11.2 关键设计思想
- 原始文件仍属于文件系统
- 产品维护的是一套本地数据库中的“元数据层”
- 删除/移动/重命名文件时需要与元数据同步

## 12. 关键技术方向（产品视角）

### 12.1 技术原则
- Local-first
- 不依赖 Windows Search 作为唯一基础能力
- 元数据独立持久化
- 可增量扫描
- UI 与索引异步解耦

### 12.2 性能重点
- 大列表虚拟滚动
- 缩略图缓存
- 分页加载
- 后台索引任务限速
- 大目录懒加载

## 13. 风险与挑战

### 13.1 技术风险
- 海量文件扫描速度与资源占用
- 文件变更监听的稳定性
- 缩略图缓存体积控制
- 多盘符 / 移动硬盘 / 网络盘兼容

### 13.2 产品风险
- 范围过大：文件管理、素材管理、游戏库、书库都想做，容易失焦
- 一开始做得太复杂，反而影响启动速度和可用性

### 13.3 解决思路
- 第一阶段优先统一元数据层 + 标签搜索 + 基础库视图
- 垂类体验逐步增强，不要一开始做太满

## 16. 当前建议的产品定义结论

### 最推荐的定义方式
不要把产品定义为“另一个资源管理器”，而要定义为：

> **面向 Windows 海量本地文件的现代化资产管理工作台**

它有三层能力：
1. **文件层**：浏览、搜索、定位、批量操作
2. **元数据层**：标签、颜色、状态、收藏、评分、集合
3. **库层**：图片/视频素材库、游戏库、电子书库、软件库

这是比“文件管理器”更准确、也更有产品空间的方向。

## 17. 方案 B：收敛版结论（当前建议冻结）

### 17.1 当前推荐的产品主轴
当前建议将产品主轴明确冻结为：

> **以“本地资产库 / 本地数字内容工作台”为核心，以“文件管理增强”作为底层支撑。**

这意味着：
- 产品的第一身份不是“资源管理器替代品”
- 产品的第一身份也不是单一的图片管理器、游戏启动器或电子书库
- 它的核心是：**在 Windows 原生文件系统之上建立一层统一的本地元数据与库式浏览系统**

### 17.2 产品定位（收敛版）

#### 我们是什么
- 面向 Windows 海量本地文件的现代化资产管理工作台
- 一个本地优先（local-first）的文件元数据平台
- 一个统一管理图片、视频、游戏、电子书、免安装软件的库式入口
- 一个兼顾“效率搜索”和“内容浏览”的桌面产品

#### 我们不是什么
- 不优先做完整资源管理器替代品
- 不做云盘/网盘产品
- 不做多人协作型企业 DAM
- 不做内容编辑器（图片编辑 / 视频编辑 / 电子书编辑）
- 不做游戏下载平台或电子书商店

#### 核心价值主张
1. 比 Windows 资源管理器更快地找到本地文件
2. 比目录树更灵活地组织本地内容
3. 比传统文件工具更适合长期浏览和管理个人数字资产
4. 用统一界面管理原本分散的图片、视频、游戏、电子书和软件

### 17.3 当前阶段的关键产品原则
1. **文件系统是真实事实层，产品数据库是元数据层**
2. **不强迫用户改变既有目录结构**
3. **搜索不是唯一入口，浏览同样重要**
4. **标签、状态、集合优先于死板目录记忆**
5. **内容库体验优先于传统工具感**
6. **性能优先于功能堆叠**
7. **UI 现代化是核心竞争力的一部分，不是后期装饰**

### 17.4 信息架构（收敛版）

#### 顶层能力分层

##### 第一层：文件层
职责：
- 识别文件真实位置
- 浏览目录
- 打开文件 / 打开所在目录
- 进行基础批量操作

这一层解决“文件在哪”。

##### 第二层：元数据层
职责：
- 标签
- 颜色标签
- 收藏
- 状态
- 评分
- 自定义分类
- 保存筛选条件
- 智能集合

这一层解决“文件是什么、怎么被组织”。

##### 第三层：库层
职责：
- 素材库（图片 / 视频）
- 游戏库
- 电子书库
- 软件库

这一层解决“如何像内容平台一样浏览本地资产”。

##### 第四层：体验层
职责：
- 现代化 UI
- 高响应搜索
- 封面墙 / 卡片流
- 快速筛选
- 大体量内容下的连续浏览

这一层解决“为什么用户愿意长期把它当作主入口”。

#### 顶部导航建议（当前版本）
1. 首页（Dashboard）
2. 全部文件
3. 素材库
4. 游戏库
5. 电子书库
6. 软件库
7. 标签 / 集合
8. 最近项目
9. 设置

#### 两种工作模式

##### 模式 A：文件模式
适合：
- 找路径
- 看目录结构
- 批量文件操作
- 精确排序与筛选

表现形式：
- 侧边目录树
- 列表/表格视图
- 详情面板

##### 模式 B：库模式
适合：
- 浏览图片/视频/游戏/书籍/软件
- 依靠封面、缩略图、标签、状态查找内容
- 长时间整理和消费内容

表现形式：
- 卡片墙
- 封面流
- 筛选器栏
- 详情页

### 17.5 统一对象模型（产品层抽象）

#### 底层统一对象：FileItem
所有真实文件先统一抽象为 FileItem，至少包含：
- 路径
- 文件名
- 扩展名
- 大小
- 时间信息
- 所属扫描源
- 基础类型识别

#### 上层派生对象：LibraryItem
在 FileItem 基础上，根据规则和元数据，将文件投射为不同库项目：
- MediaLibraryItem
- GameLibraryItem
- BookLibraryItem
- AppLibraryItem

这样设计的好处：
- 底层统一，避免系统碎裂
- 上层可做垂直体验，不被“通用文件表”限制
- 后续能逐步增强每个库，而不破坏总架构

### 17.6 MVP 边界（当前建议冻结）

#### MVP 目标
先验证三件事：
1. 在海量本地文件场景下，是否真的能比 Windows 原生更快地找到文件
2. 标签 + 库视图是否能显著改善管理体验
3. 用户是否愿意把它作为个人数字资产入口长期使用

#### MVP 必做
1. 指定目录扫描
2. 本地索引数据库
3. 增量更新机制（至少支持重扫 + 基础变更同步）
4. 全局搜索
5. 普通标签系统
6. 颜色标签系统
7. 图片 / 视频素材库基础浏览
8. 游戏 / 电子书 / 软件基础库页（哪怕先做轻量版）
9. 文件 / 库项目详情页
10. 深色现代 UI
11. 打开原文件 / 打开所在目录
12. 最近使用 / 最近导入

#### MVP 可做但不必第一批做满
1. 批量标签
2. 保存搜索条件
3. 收藏 / 评分 / 状态
4. 自定义封面
5. 轻量 Dashboard
6. 拖拽整理体验

#### MVP 明确不做
1. 不做完整资源管理器替代
2. 不做云同步
3. 不做多人协作
4. 不做 AI 自动标签
5. 不做 OCR / 视频语义识别
6. 不做在线元数据抓取重依赖功能
7. 不做复杂自动化规则引擎
8. 不做下载器 / 启动平台 / 商店化系统

### 17.7 当前最小闭环（建议作为第一版验收口径）
用户可以完成如下完整路径：
1. 添加若干扫描目录
2. 软件完成索引
3. 在统一搜索中找到目标文件
4. 为文件添加标签 / 颜色标签
5. 在素材库或其他库中通过标签再次找到它
6. 查看详情并打开原文件或打开所在目录

只要这个闭环足够顺畅，产品就已经具备第一版成立的基础。

### 17.8 当前最值得警惕的范围膨胀点
1. 一开始就想完全替代资源管理器
2. 一开始把游戏库、书库、软件库都做成重垂类系统
3. 一开始加入 AI、OCR、自动分类、在线抓取
4. 为了炫技牺牲首屏速度、滚动性能和搜索响应

### 17.9 当前建议的开发优先顺序
1. 索引与数据底座
2. 搜索与筛选
3. 标签系统
4. 素材库体验
5. 其他库模块轻量接入
6. UI 统一与细节打磨
7. 后续再逐步增强游戏 / 电子书 / 软件垂类能力

## 18. 用户场景与用户故事（用于下一步原型与 PRD）

### 18.1 为什么这一节重要
前文解决的是“产品是什么、边界在哪里”。
这一节解决的是：
- 用户到底在什么时刻打开这个产品
- 他们是为了完成什么任务而来
- 哪些任务必须在第一版做到足够顺畅

这一节会直接影响：
- 页面信息架构
- 功能优先级
- 首屏设计
- 核心闭环定义

### 18.2 核心使用场景清单

#### 场景 A：找某张图片或某段视频素材
用户记得素材的大致主题、风格、颜色、用途，但不记得路径。

典型描述：
- “我记得有一张蓝紫色的赛博朋克参考图，但忘了在哪个盘里。”
- “我有一段可以做转场的烟雾视频素材，但只记得它大概是几秒钟。”

用户需要：
- 从全局快速定位素材
- 用标签 / 类型 / 时长 / 分辨率 / 颜色标签缩小范围
- 用缩略图或预览快速确认目标

这说明第一版必须具备：
- 全局搜索
- 标签筛选
- 素材缩略图浏览
- 详情预览

#### 场景 B：整理一批新导入的素材
用户刚下载、拷贝或整理了一批图片/视频，希望快速打标签和归类，以便以后好找。

典型描述：
- “我今天新存了几十张参考图，先统一打上角色设计、盔甲、暗色调之类的标签。”
- “一批视频素材需要先标记为待整理或可用。”

用户需要：
- 快速看到最近导入内容
- 批量打标签
- 批量设置颜色标签或状态
- 之后可通过标签再次找到它们

这说明第一版很值得考虑：
- 最近导入视图
- 批量标签能力
- 轻量状态标记

#### 场景 C：把本地游戏当作一个统一库来浏览和启动
用户的游戏来源复杂，分散在不同盘符、启动器、模拟器目录或压缩包中，希望统一浏览。

典型描述：
- “我记得装过这个游戏，但忘了放在哪个目录里。”
- “我想像看 Steam 库一样看我所有本地游戏。”

用户需要：
- 统一的游戏库页
- 封面 / 图标 / 标题浏览
- 一键打开启动文件或所在目录
- 用标签或状态区分‘在玩 / 已通关 / 搁置’

这说明游戏库第一版至少要有：
- 基础识别与入库
- 卡片浏览
- 启动入口
- 状态标签

#### 场景 D：管理电子书收藏
用户拥有大量 PDF / EPUB / MOBI / AZW3 等文件，希望按封面、作者、系列、主题浏览，而不是只看文件名。

典型描述：
- “我记得这本书是某个作者写的，但文件名不一定规范。”
- “我想把技术书和艺术书分开看，还想记住哪些已读哪些未读。”

用户需要：
- 书籍库视图
- 封面浏览
- 作者 / 标题 / 标签管理
- 阅读状态管理
- 外部阅读器打开入口

#### 场景 E：把免安装软件做成自己的工具库
用户有很多绿色版/便携版工具，希望像一个个人 App Library 一样管理。

典型描述：
- “我知道我存了这个工具，但每次都得进多层目录找 exe。”
- “我想把自己常用的建模工具、压缩工具、开发工具统一放在一个界面里。”

用户需要：
- 软件库视图
- 快速启动
- 图标 / 说明 / 分类 / 常用状态管理

#### 场景 F：我知道路径，但希望更高效地浏览目录
并不是所有用户都想抛弃目录结构。有时他们只是想拥有更好看的、更快的目录浏览与筛选体验。

典型描述：
- “我知道东西就在这个项目目录里，但文件很多，不想用系统资源管理器慢慢翻。”

用户需要：
- 快速目录浏览
- 更强的筛选器
- 列表 / 网格切换
- 详情侧栏

这说明：
- 文件模式不能完全缺席
- 但它不是第一产品主轴

### 18.3 用户任务流（按优先级）

#### 任务流 1：添加扫描目录并完成首次入库
1. 打开产品
2. 添加一个或多个目录/盘符作为扫描源
3. 产品建立索引
4. 用户进入首页或库页面看到已入库内容

这条链路是所有价值产生的前提。

#### 任务流 2：全局搜索一个目标文件
1. 用户在搜索框输入关键词
2. 用户看到混合搜索结果
3. 使用筛选器进一步缩小范围
4. 打开详情确认
5. 打开原文件或所在目录

这条链路是“比资源管理器更快找到文件”的核心验证路径。

#### 任务流 3：给文件打标签并再次通过标签找到它
1. 用户定位到目标文件
2. 添加普通标签 / 颜色标签 / 状态
3. 回到标签页或库页
4. 使用标签筛选重新定位

这条链路是“元数据层成立”的核心验证路径。

#### 任务流 4：从库视图中纯浏览发现内容
1. 进入素材库 / 游戏库 / 电子书库 / 软件库
2. 使用卡片流浏览
3. 借助标签、状态、排序和筛选快速定位内容
4. 查看详情并执行打开操作

这条链路是“不是工具，而是资产工作台”的核心验证路径。

#### 任务流 5：处理最近导入内容
1. 进入最近导入
2. 看到最近新增文件
3. 批量标记标签或状态
4. 将它们纳入后续长期组织体系

这条链路决定产品是否能真正进入“持续整理”的场景，而不是只在找不到文件时偶尔使用。

### 18.4 核心用户故事（建议作为 PRD 输入）

#### 搜索与定位类
- 作为一个拥有大量本地文件的用户，我希望能在一个全局搜索框中快速找到跨盘符文件，这样我不必记住它具体放在哪个目录。
- 作为一个素材重度用户，我希望能按文件类型、标签、时间、大小等条件缩小搜索结果，这样我能更快定位目标内容。
- 作为一个视频素材用户，我希望能看到缩略图、时长和基础信息，这样我能在打开前确认是否是我要的文件。

#### 标签与组织类
- 作为一个经常收藏资料的用户，我希望能给文件添加多个标签，这样同一个文件可以被不同主题同时归类。
- 作为一个习惯轻量标记的用户，我希望支持颜色标签，这样我可以快速表达待整理、已处理、收藏等状态。
- 作为一个面对大量新素材的用户，我希望能批量打标签，这样整理成本不会过高。

#### 素材库类
- 作为一个图片/视频素材用户，我希望能以缩略图墙的方式浏览内容，这样我可以像浏览内容平台一样找素材，而不是只看文件名。
- 作为一个整理参考图的用户，我希望能按风格、用途、状态筛选素材，这样我以后能重复利用自己的资源库。

#### 游戏库类
- 作为一个拥有大量本地游戏的用户，我希望能在统一的游戏库中浏览和启动游戏，这样我不需要回忆每个游戏的安装路径。
- 作为一个收藏型玩家，我希望能给游戏标记“在玩 / 通关 / 搁置”等状态，这样我的库不仅是启动入口，也是记录界面。

#### 电子书库类
- 作为一个电子书收藏用户，我希望能通过封面、标题、作者和标签浏览书籍，这样我不需要依赖混乱的文件名。
- 作为一个长期阅读者，我希望能标记未读、在读、已读状态，这样书库能形成持续管理习惯。

#### 软件库类
- 作为一个常用绿色版软件的用户，我希望能把便携工具集中展示和启动，这样它们就不会继续散落在多个目录中。

#### 文件模式类
- 作为一个仍然需要目录结构的用户，我希望在需要时也能按路径浏览文件，这样产品不会强迫我完全放弃原有管理习惯。

### 18.5 第一版必须重点服务的用户故事
当前建议优先服务以下几条：
1. 全局快速找到文件
2. 给文件打标签/颜色标签
3. 在素材库中通过标签再次找到素材
4. 在库页中浏览图片/视频内容
5. 打开原文件或所在目录

原因：
- 这些故事共同构成最小可验证闭环
- 它们最能体现产品相对 Windows 的差异化
- 它们也是最容易形成使用习惯的入口

### 18.6 第一版暂时可以弱化的用户故事
1. 游戏库深度管理（例如平台自动识别、成就、游玩时长）
2. 电子书深度阅读生态
3. 软件库复杂版本管理
4. 自动智能分类
5. 与在线信息源深度联动

这些可以作为第二阶段增强，而不是第一阶段成立前提。

### 18.7 从用户场景推导出的页面优先级

#### P0 页面
1. 首次引导 / 添加扫描源
2. 首页或入库完成页
3. 全局搜索结果页
4. 素材库页
5. 详情页
6. 标签管理/筛选入口
7. 设置页（扫描源、索引、缓存）

#### P1 页面
1. 游戏库页
2. 电子书库页
3. 软件库页
4. 最近导入页
5. 最近使用页

#### P2 页面
1. Dashboard 强化版
2. 智能集合页
3. 批量整理工作台

### 18.8 当前结论
在当前阶段，最值得优先优化的不是“所有类型文件都支持得很深”，而是：

> **让用户从“找不到文件”过渡到“用标签和库视图长期管理自己的本地资产”。**

这才是第一版最关键的产品跃迁。
