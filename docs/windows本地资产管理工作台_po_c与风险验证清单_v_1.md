# Windows 本地资产管理工作台 PoC 与风险验证清单 v1

## 1. 文档目的

本文件用于在正式大规模实现之前，优先验证本产品最关键、最容易失控、最可能决定项目成败的技术风险。

当前目标不是补更多功能，而是回答：
- 哪些点必须先做小规模验证（PoC）
- 每个验证项要验证什么
- 验证通过的标准是什么
- 如果验证结果不理想，应如何降级或调整方案

本文件的核心原则是：

> **先验证底层成立性，再扩大实现范围。**

对于本产品而言，真正高风险的部分不是页面数量，而是：
- 文件扫描性能
- SQLite 在大量文件下的查询表现
- 缩略图生成与缓存策略
- 视频封面提取成本
- 前端在大量结果下的列表 / 网格滚动表现

---

## 2. 为什么必须做 PoC

### 2.1 当前产品的真正风险来源
这个产品看上去像“文件管理 + 现代 UI”，但真正决定成败的，不是页面画得多好，而是：

1. 文件量大时，扫描是否足够快
2. 搜索和筛选是否足够快
3. 缩略图是否能稳定生成且不会把磁盘打爆
4. 视频封面提取是否会过慢
5. 前端滚动几千上万项时是否仍然流畅

### 2.2 不先做 PoC 的风险
如果不提前做验证，后面很容易出现：
- 页面和交互都做了，但实际扫描非常慢
- 数据都入库了，但查询慢到无法用
- 素材库看起来成立，但缩略图生成太重
- 产品 Demo 好看，但真实大盘数据一接入就崩

所以 PoC 不是“可有可无的前期研究”，而是这个产品第一阶段的必要组成部分。

---

## 3. 风险分级

### 3.1 P0 风险（必须优先验证）
1. 文件扫描性能
2. SQLite 查询性能
3. 图片缩略图生成与缓存
4. 视频封面提取
5. 前端大结果集渲染性能

### 3.2 P1 风险（建议尽早验证）
1. 手动重扫的更新效率
2. 标签筛选在大量数据下的表现
3. 素材库分页 / 分块加载策略
4. 详情侧栏频繁切换时的数据读取成本

### 3.3 P2 风险（可后置）
1. 游戏 / 电子书 / 软件库识别策略
2. 更复杂元数据提取
3. 更复杂批量操作体验

## 4. PoC 验证总览

当前建议至少做以下 8 个 PoC：

1. PoC-01：大目录扫描性能验证
2. PoC-02：数据库写入与重扫更新验证
3. PoC-03：SQLite 搜索与筛选性能验证
4. PoC-04：标签 / 颜色标签查询性能验证
5. PoC-05：图片缩略图生成与缓存验证
6. PoC-06：视频封面提取可行性验证
7. PoC-07：前端列表 / 网格虚拟滚动验证
8. PoC-08：详情侧栏高频切换响应验证

---

## 5. PoC-01：大目录扫描性能验证

### 5.1 目标
验证在 Windows 本地真实目录下，对大量文件进行首次扫描时的速度、资源占用和稳定性。

### 5.2 为什么重要
这是整个产品主链的起点。
扫描如果太慢，用户第一次接入系统就会丧失耐心。

### 5.3 建议测试数据规模
至少准备 3 档：
- 小规模：1 万文件
- 中规模：5 万文件
- 大规模：10 万文件以上

最好覆盖：
- 图片目录
- 视频目录
- 混合目录
- 深层嵌套目录

### 5.4 需要记录的指标
- 扫描总耗时
- 每秒处理文件数
- CPU 占用
- 内存占用
- 是否出现明显卡顿或阻塞
- 是否能正确跳过无权限/异常文件

### 5.5 通过标准（建议）
- 小规模目录扫描体验应足够顺滑
- 中规模目录扫描在可接受时间内完成
- 大规模目录扫描即使耗时较长，也不应卡死整个应用
- 扫描错误不会中断整次任务

### 5.6 若结果不理想的应对
1. 把首次扫描拆分为批次写入
2. 优先记录基础元数据，延后扩展元数据提取
3. 降低扫描阶段的同步计算量
4. 强化扫描状态提示，避免用户误以为程序卡死

---

## 6. PoC-02：数据库写入与重扫更新验证

### 6.1 目标
验证 SQLite 在大量文件记录写入、重扫更新、新增/删除同步时的可行性。

### 6.2 为什么重要
即使扫描本身快，若数据库写入和更新策略不合理，整体仍会变慢或混乱。

### 6.3 需要验证的场景
1. 首次批量插入大量 files 记录
2. 第二次重扫时：
   - 新增文件入库
   - 已删除文件标记 `is_deleted`
   - 已存在文件更新 `last_seen_at`
3. 重复扫描同一目录不会写出重复记录

### 6.4 需要记录的指标
- 首次写入耗时
- 重扫耗时
- 每批写入大小与表现
- 数据一致性
- 索引对写入速度的影响

### 6.5 通过标准（建议）
- 首次写入可控
- 重扫开销明显低于首次全扫
- 不出现 path 重复记录
- 文件删除与新增都能正确同步

### 6.6 若结果不理想的应对
1. 调整批量写入大小
2. 分离首次扫描与后续扩展字段写入
3. 优化主键 / 唯一索引策略
4. 减少重扫时的非必要字段更新

---

## 7. PoC-03：SQLite 搜索与筛选性能验证

### 7.1 目标
验证在 files、tags、file_user_meta 等表达到真实规模后，搜索页与列表页核心查询是否足够快。

### 7.2 为什么重要
这决定“比 Windows 更快找到文件”是否真的成立。

### 7.3 需要验证的查询类型
1. 按名称模糊搜索
2. 按文件类型筛选
3. 按扫描源筛选
4. 按修改时间排序
5. 搜索 + 类型筛选组合
6. 搜索 + 分页

### 7.4 数据规模建议
至少在：
- 1 万文件
- 5 万文件
- 10 万文件
以上规模测试

### 7.5 需要记录的指标
- 查询耗时（平均 / p95）
- 分页首屏耗时
- 排序后的返回速度
- 不同索引策略下的差异

### 7.6 通过标准（建议）
- 常见查询首屏返回应足够快
- 组合筛选不会出现明显级别的性能崩坏
- 搜索页输入后结果刷新能达到“可交互”水平

### 7.7 若结果不理想的应对
1. 增加关键索引
2. 改进搜索字段归一化策略
3. 限制首屏返回字段
4. 对大页结果做分页 / 游标优化
5. 必要时把部分搜索逻辑简化为“前缀/包含”的务实实现

---

## 8. PoC-04：标签 / 颜色标签查询性能验证

### 8.1 目标
验证标签系统在大量文件下是否仍足够轻快。

### 8.2 为什么重要
你这个产品的差异化不只是搜索，而是“标签 + 再找回”。
若标签查询慢，整个元数据层价值会被削弱。

### 8.3 需要验证的查询类型
1. 单标签查询
2. 多标签筛选（AND / OR 的第一阶段策略）
3. 颜色标签筛选
4. 标签 + 文件类型组合筛选
5. 标签结果排序与分页

### 8.4 当前建议的第一阶段策略
第一阶段尽量保持克制：
- 单标签
- 多标签基础组合
- 颜色标签单独过滤
- 不要一开始做过于复杂的布尔筛选编辑器

### 8.5 通过标准（建议）
- 单标签结果返回足够快
- 常见组合筛选可接受
- 标签页切换标签时响应明显快于用户耐心阈值

### 8.6 若结果不理想的应对
1. 先限制复杂组合筛选能力
2. 通过物化结果或查询优化减轻 join 成本
3. 优先优化最常见的单标签与单颜色标签路径

---

## 9. PoC-05：图片缩略图生成与缓存验证

### 9.1 目标
验证图片缩略图是否能稳定生成、缓存，并在素材库中快速被消费。

### 9.2 为什么重要
素材库成立与否，首先看图片缩略图策略是否稳。

### 9.3 需要验证的场景
1. 不同尺寸图片缩略图生成
2. 不同格式图片缩略图生成
3. 首次生成耗时
4. 已缓存缩略图读取速度
5. 缩略图缓存体积增长趋势

### 9.4 需要记录的指标
- 单张缩略图生成耗时
- 批量生成吞吐量
- 缓存命中时读取耗时
- 缓存占用磁盘空间
- 错误图片的失败率与降级效果

### 9.5 通过标准（建议）
- 常见图片格式可稳定生成缩略图
- 已生成缩略图的读取速度足够快
- 缓存体积增长可控
- 失败图片不影响整体体验

### 9.6 若结果不理想的应对
1. 统一缩略图尺寸与质量策略
2. 限制第一阶段支持的图片格式优先级
3. 缓存按需生成，不强制一次性全量生成
4. 缩略图失败直接降级为占位图

---

## 10. PoC-06：视频封面提取可行性验证

### 10.1 目标
验证视频首帧 / 封面图提取是否足够可行，不会把第一阶段拖入重型媒体处理泥潭。

### 10.2 为什么重要
视频浏览体验对素材库很重要，但视频处理天然比图片重。

### 10.3 需要验证的场景
1. 常见视频格式的封面提取
2. 大视频文件与小视频文件的差异
3. 批量封面提取的吞吐量
4. 失败视频的降级策略

### 10.4 需要记录的指标
- 单视频封面提取耗时
- 批量提取耗时
- CPU 占用
- 失败率
- 失败后的用户可见表现

### 10.5 通过标准（建议）
- 常见视频文件可以在可接受成本下生成封面
- 失败视频可以优雅降级为默认图标 / 占位图
- 不要求第一阶段做到非常精美的视频预览

### 10.6 若结果不理想的应对
1. 第一阶段只做非常基础的封面提取
2. 控制支持格式范围
3. 将视频缩略图生成优先级放低于图片
4. 对超大视频文件延后或按需生成

---

## 11. PoC-07：前端大列表 / 大网格渲染验证

### 11.1 目标
验证前端在大量结果下的滚动、筛选、切换视图是否仍然流畅。

### 11.2 为什么重要
即使后端很快，如果前端一渲染 1000+ 条就掉帧，用户仍然会觉得产品“很慢”。

### 11.3 需要验证的页面场景
1. 搜索结果页（列表模式）
2. 搜索结果页（网格模式）
3. 素材库页（大网格）
4. 最近导入页
5. 全部文件页

### 11.4 需要验证的内容
- 虚拟滚动或分块加载可行性
- 列表与网格在不同密度下的表现
- 选中项高亮与详情侧栏联动是否带来额外抖动
- 缩略图加载对滚动的影响

### 11.5 通过标准（建议）
- 常见数据量下滚动流畅
- 切换选中项不会引发整页重渲染卡顿
- 切换列表 / 网格时表现可接受

### 11.6 若结果不理想的应对
1. 必须引入虚拟滚动 / 分块加载
2. 降低单卡片复杂度
3. 减少一次性渲染字段和组件层级
4. 详情侧栏与主列表解耦更新

---

## 12. PoC-08：详情侧栏高频切换响应验证

### 12.1 目标
验证用户连续单击不同文件时，详情侧栏是否能快速刷新且不产生明显闪烁与阻塞。

### 12.2 为什么重要
你的产品大量依赖“单击看详情 → 顺手打标签”的工作流。
如果详情层切换很慢，整个交互节奏会被破坏。

### 12.3 需要验证的场景
1. 搜索页连续点击结果
2. 素材库页连续点击卡片
3. 最近导入页连续整理
4. 已有缩略图与无缩略图两种情况

### 12.4 需要记录的指标
- 单次详情读取耗时
- 连续切换时 UI 是否闪烁
- 标签更新后详情刷新成本
- 详情侧栏更新是否拖慢主列表

### 12.5 通过标准（建议）
- 连续切换体验应顺滑
- 详情更新不应导致整页明显重排
- 标签与颜色标签更新应局部刷新

### 12.6 若结果不理想的应对
1. 详情数据读取结果缓存
2. 详情侧栏局部 loading，而非整页 loading
3. 列表选中状态与详情刷新解耦
4. 减少详情首屏必需字段

---

## 13. 建议测试数据集

### 13.1 测试数据建议分类
至少准备以下目录：
1. 图片目录（大量 jpg/png/webp）
2. 视频目录（mp4/mkv/mov 等）
3. 混合目录（图/视/文档混合）
4. 深层嵌套目录
5. 文件名混乱、格式混乱目录

### 13.2 规模建议
- 轻量：1 万文件
- 中量：5 万文件
- 重量：10 万文件以上

### 13.3 为什么要用真实数据而不是纯模拟数据
因为：
- 实际文件格式不干净
- 目录层级不整齐
- 文件大小差异极大
- 视频与图片格式兼容问题很多

真实数据更能暴露真实风险。

---

## 14. 建议验证顺序

### 第一步：扫描与数据库
先做：
1. PoC-01 扫描性能
2. PoC-02 写入与重扫
3. PoC-03 SQLite 查询性能

### 第二步：媒体能力
再做：
4. PoC-05 图片缩略图
5. PoC-06 视频封面提取

### 第三步：前端承载能力
最后做：
6. PoC-07 大列表 / 大网格渲染
7. PoC-08 详情侧栏高频切换
8. PoC-04 标签查询性能（可穿插）

### 原因
这是最符合产品主链与技术依赖的顺序：
- 先验证后端底座能不能站住
- 再验证素材库能不能成立
- 最后验证 UI 能不能承载真实数据量

## 15. PoC 输出建议格式

每个 PoC 最终都建议形成一页小结，包含：

### 15.1 建议模板
- PoC 名称
- 验证目标
- 测试环境
- 数据规模
- 测试方法
- 关键指标
- 结果结论
- 是否通过
- 若不通过，建议调整方案

### 15.2 通过判定建议
- **Pass**：可直接进入正式实现
- **Pass with constraints**：可实现，但需带约束进入
- **Fail**：需调整方案后再继续

---

## 16. 当前阶段降级策略原则

PoC 的目标不是证明“所有想法都完美成立”，而是尽早识别哪些点需要降级。

### 当前降级原则
1. **优先保证主链成立，不强求每个体验都做到最好**
2. **图片优先于视频**
3. **基础筛选优先于复杂组合搜索**
4. **手动重扫优先于复杂实时监听**
5. **详情侧栏稳定优先于炫技交互**
6. **素材库成立优先于游戏/书库深做**

---

## 17. 当前最关键的通过门槛

如果以下 5 条大体成立，则本项目第一阶段风险基本可控：

1. 扫描 1~5 万文件时，整体体验可接受
2. SQLite 搜索与筛选在目标规模下响应足够快
3. 图片缩略图生成与缓存策略成立
4. 视频封面提取至少能基础可用
5. 前端在大列表 / 大网格下仍能保持可交互流畅度

只要这 5 条成立，你前面的文档链就不是“纸上产品”，而是具备真实落地基础。

---

## 18. 当前结论

当前 PoC / 风险验证清单的核心结论可以浓缩为一句话：

> **这个项目第一阶段最值得先验证的，不是功能是否够多，而是底层是否足够稳，能够支撑“找回、标记、再找回、可浏览”这条主链。**

只要扫描、查询、缩略图、视频封面、前端滚动这几件事成立，后面的产品实现就会进入一个非常健康的区间。

---

## 19. 下一步建议

在 PoC / 风险验证清单之后，最适合继续推进的方向有两个：
1. **前端工程目录与状态管理草案**
2. **PoC 执行记录模板 / 实验表格**

如果准备开始真正动手验证，最推荐下一步先做：

> **PoC 执行记录模板 v1**

因为一旦开始测试扫描、查询、缩略图等能力，你会很快需要一套统一格式来记录结果，否则后面很难比较与复盘。
