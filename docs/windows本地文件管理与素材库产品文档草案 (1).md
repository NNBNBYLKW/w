# Windows 本地文件管理与素材库产品文档（草案）

## 1. 产品概述

### 1.1 产品名称（暂定）
- 中文名：待定
- 英文名：待定

### 1.2 产品一句话定位
一款面向 **大量本地文件管理** 的 Windows 桌面应用，将 **高速索引、标签化组织、媒体素材检索、库式浏览、现代化 UI** 融合在同一套本地优先（local-first）的体验中。

### 1.3 产品目标
解决 Windows 原生资源管理器在以下场景中的核心不足：
- 大量文件下索引慢、搜索慢、筛选弱
- 缺乏可视化标签体系（尤其缺乏类似 macOS Finder 的颜色标签）
- 无法像流媒体网站一样基于标签、分类、筛选条件浏览图片和视频素材
- 对游戏、电子书、免安装软件缺乏“库式管理”能力
- 专业文件管理工具常见 UI 偏传统、偏工具化，缺乏现代感和内容浏览体验

---

## 2. 背景与机会

### 2.1 用户背景
目标用户通常拥有以下特点：
- 本地硬盘/移动硬盘中存有大量文件
- 文件类型复杂：图片、视频、游戏、电子书、压缩包、软件、项目文件混杂
- 不满足于简单树状目录，期望“按内容/按用途/按标签”管理
- 既需要效率型管理，也需要内容型浏览
- 对 UI 审美和现代交互有要求

### 2.2 当前痛点
1. **Windows 搜索与索引体验弱**
   - 初始索引慢
   - 非索引目录搜索性能差
   - 对大体量机械盘/多盘符/移动盘支持体验不稳定
2. **文件组织方式过于依赖目录结构**
   - 一个文件只能放在一个物理目录中，难以表达多重属性
   - 无颜色标签、评分、状态等轻量元数据体系
3. **媒体素材管理能力不足**
   - 图片/视频只能按文件名和文件夹找
   - 缺乏标签、筛选器、封面墙、预览流式浏览
4. **游戏/电子书/软件缺乏统一库视图**
   - 游戏散落多个盘符和启动器
   - 电子书缺乏统一封面浏览、作者/系列/标签管理
   - 免安装软件没有类似 Steam 库的统一入口和元数据管理
5. **现有工具审美与交互偏朴素**
   - 重功能轻体验
   - 不适合长期管理自己的“数字资产仓库”

### 2.3 产品机会
本产品可切入为：
- **文件管理器**：替代或增强资源管理器
- **本地资产库**：像媒体库/Steam 库一样管理内容
- **标签化元数据层**：在原始文件系统之上增加一层可检索、可浏览、可运营的“数据层”

## 3. 产品定位

### 3.1 产品本质
不是单纯“更好看的资源管理器”，而是：

> **建立在 Windows 文件系统之上的本地文件资产管理平台**。

### 3.2 核心价值主张
- **比资源管理器更快地找到文件**
- **比目录树更灵活地组织文件**
- **比传统文件管理器更适合浏览内容**
- **比单独的游戏库/电子书库更统一**

### 3.3 关键词
- Local-first
- 高性能索引
- 标签化管理
- 多视图浏览
- 现代 UI
- 素材库
- 库式管理

### 3.4 非目标（阶段性）
- 不做云盘同步平台
- 不做多人协作 DAM（数字资产管理）系统
- 不做完整视频编辑/图片编辑器
- 不做游戏下载平台
- 不做电子书内容制作平台

---

## 4. 目标用户

### 4.1 核心用户画像

#### A. 素材重度用户
- 设计师、视频剪辑、自媒体作者、画师、3D 制作者
- 拥有大量图片、视频、音频、参考资料
- 强依赖缩略图、预览、标签、筛选

#### B. 本地收藏型用户
- 收藏游戏、电子书、软件、资源包、教程资料
- 希望像“数字藏品库”一样管理自己的本地内容

#### C. 效率型重度文件用户
- 大量项目文件、文档、压缩包、多盘符资料管理
- 需要高速搜索、批量整理、分类标记

### 4.2 次级用户画像
- 普通 Windows 用户中对系统原生体验不满的人
- 希望拥有“更现代、更好看、更顺手”的文件入口的人

## 5. 用户核心需求

### 5.1 功能需求
1. 高速索引与搜索
2. 文件标签系统（含颜色标签）
3. 类流媒体网站的素材浏览与筛选
4. 游戏 / 电子书 / 软件库视图
5. 多类型文件统一管理
6. 批量标记、批量整理、批量归档
7. 缩略图、封面、预览、详情信息
8. 自定义分类、智能集合、保存筛选条件

### 5.2 体验需求
1. 启动快、滚动快、搜索响应快
2. 界面现代化，接近内容平台而非陈旧工具
3. 适合长时间浏览和整理
4. 视觉上能区分“文件浏览”和“内容库浏览”两种模式

### 5.3 心理需求
- “这是我的数字仓库”
- “我可以掌控混乱”
- “找东西不用靠记忆目录位置”
- “我的收藏终于有一个像样的展示空间”

## 6. 核心问题定义

### 6.1 需要解决的首要问题
**如何在不改变用户原有目录结构的前提下，建立一套高性能、本地可控、适合浏览和检索的元数据层。**

### 6.2 产品的关键矛盾
1. **文件系统逻辑 vs 内容平台逻辑**
   - 文件系统强调路径、层级、真实位置
   - 内容平台强调标签、推荐、集合、封面、关联
2. **高性能 vs 富交互**
   - 大量缩略图、预览、标签过滤会带来性能压力
3. **通用性 vs 垂直体验**
   - 一套系统要管理图片、视频、游戏、电子书、软件，抽象层需要设计得足够稳

## 8. 核心功能设计

### 8.1 索引系统

#### 目标
建立比 Windows 原生更快、更可控的索引机制。

#### 索引内容
- 文件路径
- 文件名
- 扩展名
- 文件大小
- 修改时间 / 创建时间
- 所属目录
- 自定义标签
- 颜色标签
- 评分 / 收藏状态
- 文件类型识别（图片/视频/游戏/电子书/软件/压缩包等）
- 媒体元数据（分辨率、时长、编码、封面）
- 电子书元数据（标题、作者、系列、封面）
- 游戏/软件启动信息

#### 索引方式建议
- 首次全盘扫描 + 增量监听
- 用户可选择扫描目录
- 支持忽略目录/文件类型
- 支持后台低优先级构建索引
- 支持手动重建索引

### 8.2 标签系统

#### 标签类型
1. 普通标签（如：教程、待整理、参考、已看）
2. 颜色标签（如：红=紧急，黄=待处理，绿=完成，蓝=收藏）
3. 分类标签（如：游戏、电影感、赛博朋克、建模参考）
4. 状态标签（如：未看、已整理、待归档）

#### 标签能力
- 单文件多标签
- 批量打标签
- 标签组合筛选
- 排除标签筛选
- 保存标签筛选条件

### 8.3 搜索与筛选

#### 搜索方式
- 全局搜索
- 当前库内搜索
- 标签搜索
- 组合条件搜索

#### 可筛选条件
- 文件类型
- 标签
- 颜色标签
- 文件大小范围
- 时间范围
- 分辨率 / 时长（媒体）
- 作者 / 系列（电子书）
- 是否已收藏 / 已评分
- 所在磁盘 / 路径

#### 高级搜索表达式（中后期）
例如：
- `type:image tag:赛博朋克 color:red`
- `library:game installed:true`
- `type:video duration>10m tag:素材`

### 8.4 素材库（图片 / 视频）

#### 浏览方式
- 瀑布流
- 网格卡片
- 时间线
- 标签分组
- 相册/集合

#### 详情信息
- 大图预览 / 视频预览
- 路径
- 标签
- 分辨率 / 时长 / 大小
- 来源目录
- 相关文件

#### 重点体验
- 像流媒体网站一样通过标签快速缩小范围
- 像本地媒体库一样通过缩略图浏览

### 8.5 游戏库

#### 目标
为本地游戏建立类似 Steam 库的浏览页。

#### 功能
- 封面墙显示
- 游戏详情页
- 启动入口
- 自定义封面/背景图
- 平台分类（Steam / 模拟器 / 独立本地游戏 / 压缩游戏）
- 状态管理（已玩 / 在玩 / 通关 / 搁置）
- 标签（合作 / 单机 / RPG / FPS 等）

### 8.6 电子书库

#### 功能
- 封面墙浏览
- 书籍详情页
- 标题 / 作者 / 系列 / 标签管理
- 阅读状态（未读 / 在读 / 已读）
- 分类（技术 / 小说 / 艺术 / 历史等）
- 调用外部阅读器打开

### 8.7 软件库（免安装软件）

#### 功能
- 图标/卡片展示
- 软件说明、版本、来源路径
- 快速启动
- 分类（设计 / 开发 / 工具 / 娱乐）
- 状态（常用 / 测试 / 归档）

### 8.8 现代 UI 设计目标

#### 视觉方向
- 深色模式优先
- 大量卡片式布局
- 统一圆角、间距、动效
- 内容优先，弱化传统“表格工具感”

#### 交互原则
- 浏览优先
- 搜索随时可达
- 筛选器固定可见
- 详情面板可快速展开/收起
- 文件模式与库模式清晰区分

## 10. MVP 范围建议

### 10.1 MVP 核心目标
先验证三件事：
1. 海量文件下索引和搜索是否足够快
2. 标签 + 库视图是否足够有价值
3. 用户是否愿意长期把它当作“数字仓库入口”

### 10.2 MVP 功能
- 指定目录扫描与索引
- 全局搜索
- 标签系统（普通标签 + 颜色标签）
- 图片/视频基础素材库
- 游戏/电子书/软件的基础库卡片页
- 文件详情页
- 深色现代 UI

### 10.3 暂不进入 MVP 的功能
- AI 自动打标签
- OCR / 视频内容识别
- 自动元数据抓取网络信息
- 云同步
- 多端协同
- 推荐系统

## 11. 信息架构与数据模型（简化）

### 11.1 核心对象
1. 文件（FileItem）
2. 标签（Tag）
3. 文件标签关系（FileTagRelation）
4. 库项目（LibraryItem）
5. 收藏/评分/状态（UserMeta）
6. 索引任务（IndexTask）

### 11.2 关键设计思想
- 原始文件仍属于文件系统
- 产品维护的是一套本地数据库中的“元数据层”
- 删除/移动/重命名文件时需要与元数据同步

## 12. 关键技术方向（产品视角）

### 12.1 技术原则
- Local-first
- 不依赖 Windows Search 作为唯一基础能力
- 元数据独立持久化
- 可增量扫描
- UI 与索引异步解耦

### 12.2 性能重点
- 大列表虚拟滚动
- 缩略图缓存
- 分页加载
- 后台索引任务限速
- 大目录懒加载

## 13. 风险与挑战

### 13.1 技术风险
- 海量文件扫描速度与资源占用
- 文件变更监听的稳定性
- 缩略图缓存体积控制
- 多盘符 / 移动硬盘 / 网络盘兼容

### 13.2 产品风险
- 范围过大：文件管理、素材管理、游戏库、书库都想做，容易失焦
- 一开始做得太复杂，反而影响启动速度和可用性

### 13.3 解决思路
- 第一阶段优先统一元数据层 + 标签搜索 + 基础库视图
- 垂类体验逐步增强，不要一开始做太满

## 16. 当前建议的产品定义结论

### 最推荐的定义方式
不要把产品定义为“另一个资源管理器”，而要定义为：

> **面向 Windows 海量本地文件的现代化资产管理工作台**

它有三层能力：
1. **文件层**：浏览、搜索、定位、批量操作
2. **元数据层**：标签、颜色、状态、收藏、评分、集合
3. **库层**：图片/视频素材库、游戏库、电子书库、软件库

这是比“文件管理器”更准确、也更有产品空间的方向。

## 17. 方案 B：收敛版结论（当前建议冻结）

### 17.1 当前推荐的产品主轴
当前建议将产品主轴明确冻结为：

> **以“本地资产库 / 本地数字内容工作台”为核心，以“文件管理增强”作为底层支撑。**

这意味着：
- 产品的第一身份不是“资源管理器替代品”
- 产品的第一身份也不是单一的图片管理器、游戏启动器或电子书库
- 它的核心是：**在 Windows 原生文件系统之上建立一层统一的本地元数据与库式浏览系统**

### 17.2 产品定位（收敛版）

#### 我们是什么
- 面向 Windows 海量本地文件的现代化资产管理工作台
- 一个本地优先（local-first）的文件元数据平台
- 一个统一管理图片、视频、游戏、电子书、免安装软件的库式入口
- 一个兼顾“效率搜索”和“内容浏览”的桌面产品

#### 我们不是什么
- 不优先做完整资源管理器替代品
- 不做云盘/网盘产品
- 不做多人协作型企业 DAM
- 不做内容编辑器（图片编辑 / 视频编辑 / 电子书编辑）
- 不做游戏下载平台或电子书商店

#### 核心价值主张
1. 比 Windows 资源管理器更快地找到本地文件
2. 比目录树更灵活地组织本地内容
3. 比传统文件工具更适合长期浏览和管理个人数字资产
4. 用统一界面管理原本分散的图片、视频、游戏、电子书和软件

### 17.3 当前阶段的关键产品原则
1. **文件系统是真实事实层，产品数据库是元数据层**
2. **不强迫用户改变既有目录结构**
3. **搜索不是唯一入口，浏览同样重要**
4. **标签、状态、集合优先于死板目录记忆**
5. **内容库体验优先于传统工具感**
6. **性能优先于功能堆叠**
7. **UI 现代化是核心竞争力的一部分，不是后期装饰**

### 17.4 信息架构（收敛版）

#### 顶层能力分层

##### 第一层：文件层
职责：
- 识别文件真实位置
- 浏览目录
- 打开文件 / 打开所在目录
- 进行基础批量操作

这一层解决“文件在哪”。

##### 第二层：元数据层
职责：
- 标签
- 颜色标签
- 收藏
- 状态
- 评分
- 自定义分类
- 保存筛选条件
- 智能集合

这一层解决“文件是什么、怎么被组织”。

##### 第三层：库层
职责：
- 素材库（图片 / 视频）
- 游戏库
- 电子书库
- 软件库

这一层解决“如何像内容平台一样浏览本地资产”。

##### 第四层：体验层
职责：
- 现代化 UI
- 高响应搜索
- 封面墙 / 卡片流
- 快速筛选
- 大体量内容下的连续浏览

这一层解决“为什么用户愿意长期把它当作主入口”。

#### 顶部导航建议（当前版本）
1. 首页（Dashboard）
2. 全部文件
3. 素材库
4. 游戏库
5. 电子书库
6. 软件库
7. 标签 / 集合
8. 最近项目
9. 设置

#### 两种工作模式

##### 模式 A：文件模式
适合：
- 找路径
- 看目录结构
- 批量文件操作
- 精确排序与筛选

表现形式：
- 侧边目录树
- 列表/表格视图
- 详情面板

##### 模式 B：库模式
适合：
- 浏览图片/视频/游戏/书籍/软件
- 依靠封面、缩略图、标签、状态查找内容
- 长时间整理和消费内容

表现形式：
- 卡片墙
- 封面流
- 筛选器栏
- 详情页

### 17.5 统一对象模型（产品层抽象）

#### 底层统一对象：FileItem
所有真实文件先统一抽象为 FileItem，至少包含：
- 路径
- 文件名
- 扩展名
- 大小
- 时间信息
- 所属扫描源
- 基础类型识别

#### 上层派生对象：LibraryItem
在 FileItem 基础上，根据规则和元数据，将文件投射为不同库项目：
- MediaLibraryItem
- GameLibraryItem
- BookLibraryItem
- AppLibraryItem

这样设计的好处：
- 底层统一，避免系统碎裂
- 上层可做垂直体验，不被“通用文件表”限制
- 后续能逐步增强每个库，而不破坏总架构

### 17.6 MVP 边界（当前建议冻结）

#### MVP 目标
先验证三件事：
1. 在海量本地文件场景下，是否真的能比 Windows 原生更快地找到文件
2. 标签 + 库视图是否能显著改善管理体验
3. 用户是否愿意把它作为个人数字资产入口长期使用

#### MVP 必做
1. 指定目录扫描
2. 本地索引数据库
3. 增量更新机制（至少支持重扫 + 基础变更同步）
4. 全局搜索
5. 普通标签系统
6. 颜色标签系统
7. 图片 / 视频素材库基础浏览
8. 游戏 / 电子书 / 软件基础库页（哪怕先做轻量版）
9. 文件 / 库项目详情页
10. 深色现代 UI
11. 打开原文件 / 打开所在目录
12. 最近使用 / 最近导入

#### MVP 可做但不必第一批做满
1. 批量标签
2. 保存搜索条件
3. 收藏 / 评分 / 状态
4. 自定义封面
5. 轻量 Dashboard
6. 拖拽整理体验

#### MVP 明确不做
1. 不做完整资源管理器替代
2. 不做云同步
3. 不做多人协作
4. 不做 AI 自动标签
5. 不做 OCR / 视频语义识别
6. 不做在线元数据抓取重依赖功能
7. 不做复杂自动化规则引擎
8. 不做下载器 / 启动平台 / 商店化系统

### 17.7 当前最小闭环（建议作为第一版验收口径）
用户可以完成如下完整路径：
1. 添加若干扫描目录
2. 软件完成索引
3. 在统一搜索中找到目标文件
4. 为文件添加标签 / 颜色标签
5. 在素材库或其他库中通过标签再次找到它
6. 查看详情并打开原文件或打开所在目录

只要这个闭环足够顺畅，产品就已经具备第一版成立的基础。

### 17.8 当前最值得警惕的范围膨胀点
1. 一开始就想完全替代资源管理器
2. 一开始把游戏库、书库、软件库都做成重垂类系统
3. 一开始加入 AI、OCR、自动分类、在线抓取
4. 为了炫技牺牲首屏速度、滚动性能和搜索响应

### 17.9 当前建议的开发优先顺序
1. 索引与数据底座
2. 搜索与筛选
3. 标签系统
4. 素材库体验
5. 其他库模块轻量接入
6. UI 统一与细节打磨
7. 后续再逐步增强游戏 / 电子书 / 软件垂类能力

## 18. 用户场景与用户故事（用于下一步原型与 PRD）

### 18.1 为什么这一节重要
前文解决的是“产品是什么、边界在哪里”。
这一节解决的是：
- 用户到底在什么时刻打开这个产品
- 他们是为了完成什么任务而来
- 哪些任务必须在第一版做到足够顺畅

这一节会直接影响：
- 页面信息架构
- 功能优先级
- 首屏设计
- 核心闭环定义

### 18.2 核心使用场景清单

#### 场景 A：找某张图片或某段视频素材
用户记得素材的大致主题、风格、颜色、用途，但不记得路径。

典型描述：
- “我记得有一张蓝紫色的赛博朋克参考图，但忘了在哪个盘里。”
- “我有一段可以做转场的烟雾视频素材，但只记得它大概是几秒钟。”

用户需要：
- 从全局快速定位素材
- 用标签 / 类型 / 时长 / 分辨率 / 颜色标签缩小范围
- 用缩略图或预览快速确认目标

这说明第一版必须具备：
- 全局搜索
- 标签筛选
- 素材缩略图浏览
- 详情预览

#### 场景 B：整理一批新导入的素材
用户刚下载、拷贝或整理了一批图片/视频，希望快速打标签和归类，以便以后好找。

典型描述：
- “我今天新存了几十张参考图，先统一打上角色设计、盔甲、暗色调之类的标签。”
- “一批视频素材需要先标记为待整理或可用。”

用户需要：
- 快速看到最近导入内容
- 批量打标签
- 批量设置颜色标签或状态
- 之后可通过标签再次找到它们

这说明第一版很值得考虑：
- 最近导入视图
- 批量标签能力
- 轻量状态标记

#### 场景 C：把本地游戏当作一个统一库来浏览和启动
用户的游戏来源复杂，分散在不同盘符、启动器、模拟器目录或压缩包中，希望统一浏览。

典型描述：
- “我记得装过这个游戏，但忘了放在哪个目录里。”
- “我想像看 Steam 库一样看我所有本地游戏。”

用户需要：
- 统一的游戏库页
- 封面 / 图标 / 标题浏览
- 一键打开启动文件或所在目录
- 用标签或状态区分‘在玩 / 已通关 / 搁置’

这说明游戏库第一版至少要有：
- 基础识别与入库
- 卡片浏览
- 启动入口
- 状态标签

#### 场景 D：管理电子书收藏
用户拥有大量 PDF / EPUB / MOBI / AZW3 等文件，希望按封面、作者、系列、主题浏览，而不是只看文件名。

典型描述：
- “我记得这本书是某个作者写的，但文件名不一定规范。”
- “我想把技术书和艺术书分开看，还想记住哪些已读哪些未读。”

用户需要：
- 书籍库视图
- 封面浏览
- 作者 / 标题 / 标签管理
- 阅读状态管理
- 外部阅读器打开入口

#### 场景 E：把免安装软件做成自己的工具库
用户有很多绿色版/便携版工具，希望像一个个人 App Library 一样管理。

典型描述：
- “我知道我存了这个工具，但每次都得进多层目录找 exe。”
- “我想把自己常用的建模工具、压缩工具、开发工具统一放在一个界面里。”

用户需要：
- 软件库视图
- 快速启动
- 图标 / 说明 / 分类 / 常用状态管理

#### 场景 F：我知道路径，但希望更高效地浏览目录
并不是所有用户都想抛弃目录结构。有时他们只是想拥有更好看的、更快的目录浏览与筛选体验。

典型描述：
- “我知道东西就在这个项目目录里，但文件很多，不想用系统资源管理器慢慢翻。”

用户需要：
- 快速目录浏览
- 更强的筛选器
- 列表 / 网格切换
- 详情侧栏

这说明：
- 文件模式不能完全缺席
- 但它不是第一产品主轴

### 18.3 用户任务流（按优先级）

#### 任务流 1：添加扫描目录并完成首次入库
1. 打开产品
2. 添加一个或多个目录/盘符作为扫描源
3. 产品建立索引
4. 用户进入首页或库页面看到已入库内容

这条链路是所有价值产生的前提。

#### 任务流 2：全局搜索一个目标文件
1. 用户在搜索框输入关键词
2. 用户看到混合搜索结果
3. 使用筛选器进一步缩小范围
4. 打开详情确认
5. 打开原文件或所在目录

这条链路是“比资源管理器更快找到文件”的核心验证路径。

#### 任务流 3：给文件打标签并再次通过标签找到它
1. 用户定位到目标文件
2. 添加普通标签 / 颜色标签 / 状态
3. 回到标签页或库页
4. 使用标签筛选重新定位

这条链路是“元数据层成立”的核心验证路径。

#### 任务流 4：从库视图中纯浏览发现内容
1. 进入素材库 / 游戏库 / 电子书库 / 软件库
2. 使用卡片流浏览
3. 借助标签、状态、排序和筛选快速定位内容
4. 查看详情并执行打开操作

这条链路是“不是工具，而是资产工作台”的核心验证路径。

#### 任务流 5：处理最近导入内容
1. 进入最近导入
2. 看到最近新增文件
3. 批量标记标签或状态
4. 将它们纳入后续长期组织体系

这条链路决定产品是否能真正进入“持续整理”的场景，而不是只在找不到文件时偶尔使用。

### 18.4 核心用户故事（建议作为 PRD 输入）

#### 搜索与定位类
- 作为一个拥有大量本地文件的用户，我希望能在一个全局搜索框中快速找到跨盘符文件，这样我不必记住它具体放在哪个目录。
- 作为一个素材重度用户，我希望能按文件类型、标签、时间、大小等条件缩小搜索结果，这样我能更快定位目标内容。
- 作为一个视频素材用户，我希望能看到缩略图、时长和基础信息，这样我能在打开前确认是否是我要的文件。

#### 标签与组织类
- 作为一个经常收藏资料的用户，我希望能给文件添加多个标签，这样同一个文件可以被不同主题同时归类。
- 作为一个习惯轻量标记的用户，我希望支持颜色标签，这样我可以快速表达待整理、已处理、收藏等状态。
- 作为一个面对大量新素材的用户，我希望能批量打标签，这样整理成本不会过高。

#### 素材库类
- 作为一个图片/视频素材用户，我希望能以缩略图墙的方式浏览内容，这样我可以像浏览内容平台一样找素材，而不是只看文件名。
- 作为一个整理参考图的用户，我希望能按风格、用途、状态筛选素材，这样我以后能重复利用自己的资源库。

#### 游戏库类
- 作为一个拥有大量本地游戏的用户，我希望能在统一的游戏库中浏览和启动游戏，这样我不需要回忆每个游戏的安装路径。
- 作为一个收藏型玩家，我希望能给游戏标记“在玩 / 通关 / 搁置”等状态，这样我的库不仅是启动入口，也是记录界面。

#### 电子书库类
- 作为一个电子书收藏用户，我希望能通过封面、标题、作者和标签浏览书籍，这样我不需要依赖混乱的文件名。
- 作为一个长期阅读者，我希望能标记未读、在读、已读状态，这样书库能形成持续管理习惯。

#### 软件库类
- 作为一个常用绿色版软件的用户，我希望能把便携工具集中展示和启动，这样它们就不会继续散落在多个目录中。

#### 文件模式类
- 作为一个仍然需要目录结构的用户，我希望在需要时也能按路径浏览文件，这样产品不会强迫我完全放弃原有管理习惯。

### 18.5 第一版必须重点服务的用户故事
当前建议优先服务以下几条：
1. 全局快速找到文件
2. 给文件打标签/颜色标签
3. 在素材库中通过标签再次找到素材
4. 在库页中浏览图片/视频内容
5. 打开原文件或所在目录

原因：
- 这些故事共同构成最小可验证闭环
- 它们最能体现产品相对 Windows 的差异化
- 它们也是最容易形成使用习惯的入口

### 18.6 第一版暂时可以弱化的用户故事
1. 游戏库深度管理（例如平台自动识别、成就、游玩时长）
2. 电子书深度阅读生态
3. 软件库复杂版本管理
4. 自动智能分类
5. 与在线信息源深度联动

这些可以作为第二阶段增强，而不是第一阶段成立前提。

### 18.7 从用户场景推导出的页面优先级

#### P0 页面
1. 首次引导 / 添加扫描源
2. 首页或入库完成页
3. 全局搜索结果页
4. 素材库页
5. 详情页
6. 标签管理/筛选入口
7. 设置页（扫描源、索引、缓存）

#### P1 页面
1. 游戏库页
2. 电子书库页
3. 软件库页
4. 最近导入页
5. 最近使用页

#### P2 页面
1. Dashboard 强化版
2. 智能集合页
3. 批量整理工作台

### 18.8 当前结论
在当前阶段，最值得优先优化的不是“所有类型文件都支持得很深”，而是：

> **让用户从“找不到文件”过渡到“用标签和库视图长期管理自己的本地资产”。**

这才是第一版最关键的产品跃迁。

## 19. MVP 功能优先级文档（用于设计与开发排期）

### 19.1 这一节的目标
这一节不是继续罗列想法，而是明确：
- 第一版必须做什么
- 哪些功能值得做但不应阻塞上线
- 哪些功能后面再做
- 哪些功能现在明确不做

其核心原则是：

> **优先保证“索引 → 搜索 → 标签 → 库浏览 → 打开文件”的最小闭环成立。**

---

### 19.2 P0：第一版必须完成的能力

#### P0-1 扫描源管理
**目标**：让用户把真实文件系统接入产品。

**必须包含**：
- 添加扫描目录 / 盘符
- 删除扫描源
- 启用 / 禁用某个扫描源
- 忽略指定目录（至少基础版）

**为什么必须做**：
没有扫描源管理，产品无法建立自己的数据底座。

**验收口径**：
- 用户能添加至少一个目录作为扫描源
- 系统能记住扫描源配置
- 用户能移除或暂停某个扫描源

#### P0-2 本地索引数据库
**目标**：建立不依赖 Windows Search 的元数据底层。

**必须包含**：
- 文件基础元数据入库
- 首次扫描
- 手动重扫
- 基础去重逻辑（至少按路径唯一）
- 文件消失后的状态同步（至少重扫后正确）

**为什么必须做**：
这是搜索、标签、库视图的共同基础。

**验收口径**：
- 指定目录内文件可被稳定发现并写入索引
- 重扫后新增/删除文件可反映到结果中
- 数据不会因重复扫描无限重复写入

#### P0-3 全局搜索
**目标**：让用户比 Windows 资源管理器更快找到内容。

**必须包含**：
- 文件名搜索
- 基础模糊匹配
- 按文件类型筛选
- 按扫描源/路径筛选（至少基础版）
- 搜索结果列表可直接进入详情

**为什么必须做**：
这是用户最容易感知价值的入口。

**验收口径**：
- 用户可通过一个统一入口检索跨目录内容
- 结果能按至少“全部 / 图片 / 视频 / 书籍 / 软件 / 其他”分类或筛选
- 搜索到目标后可继续查看详情并打开原文件

#### P0-4 标签系统（普通标签）
**目标**：建立目录之外的第二组织方式。

**必须包含**：
- 新建标签
- 给单个文件添加/移除标签
- 在详情页展示已有标签
- 通过标签筛选结果

**为什么必须做**：
没有标签，产品只能是“另一种搜索器”，无法形成元数据层。

**验收口径**：
- 用户能为文件添加多个标签
- 标签能稳定保存
- 标签能参与搜索/筛选

#### P0-5 颜色标签系统
**目标**：提供轻量、直觉化的快速标记能力。

**必须包含**：
- 至少一组固定颜色标签
- 单文件设置颜色标签
- 按颜色标签筛选

**为什么必须做**：
这是你相对 Windows 原生体验的重要差异点。

**验收口径**：
- 用户可以明确看见颜色标签状态
- 颜色标签可用于筛选
- 颜色标签与普通标签不冲突

#### P0-6 素材库（图片 / 视频）基础版
**目标**：把标签化浏览价值最强的场景先做出来。

**必须包含**：
- 图片 / 视频的库视图入口
- 网格卡片浏览
- 基础缩略图
- 视频基础信息（至少时长或可识别为视频）
- 标签/类型筛选
- 打开详情

**为什么必须做**：
素材库是第一版最能体现产品差异化与现代 UI 的核心模块。

**验收口径**：
- 用户能在独立素材库里浏览图片与视频
- 用户可依靠缩略图而非文件名找内容
- 标签和筛选器能在素材库中起效

#### P0-7 统一详情页 / 详情面板
**目标**：承接搜索与浏览，成为所有操作的汇聚点。

**必须包含**：
- 文件基础信息
- 路径
- 所属标签
- 颜色标签
- 打开原文件
- 打开所在目录

**为什么必须做**：
没有详情层，用户无法完成确认、整理和跳转动作。

**验收口径**：
- 从搜索结果和库视图都能进入详情
- 详情页可完成基础标记和打开操作

#### P0-8 基础文件模式
**目标**：保留最必要的路径导向浏览能力。

**必须包含**：
- 基于扫描源的目录浏览
- 文件列表视图
- 列表与网格至少二选一，但建议列表必须有
- 基础排序（名称 / 修改时间 / 大小 至少其一部分）

**为什么必须做**：
有些用户仍然需要路径心智；没有这层会削弱产品可信度。

**验收口径**：
- 用户可按目录进入文件
- 可从目录结果跳转详情
- 不强迫所有使用场景都走搜索或库视图

#### P0-9 最近导入 / 最近更新
**目标**：提供持续整理入口，而不是只在找不到文件时使用。

**必须包含**：
- 基于时间排序的最近文件列表
- 能从此列表进入详情和打标签

**为什么必须做**：
这有助于产品形成高频使用习惯。

**验收口径**：
- 用户能看见最近进入索引的文件
- 能直接在这里完成整理动作

#### P0-10 现代深色 UI 基线
**目标**：从第一版开始就体现产品审美与结构优势。

**必须包含**：
- 深色模式优先
- 卡片与列表的统一视觉体系
- 清晰的侧边导航
- 搜索框始终易达
- 筛选器位置稳定

**为什么必须做**：
你的产品差异化包含体验感，不能把 UI 完全放到后期。

**验收口径**：
- 主要页面视觉风格统一
- 浏览密度与可读性平衡
- 不呈现传统老式文件工具的混乱感

### 19.5 当前阶段明确不做

#### 不做 1：完整资源管理器替代
不以复刻 Windows Explorer 全部能力为目标。

#### 不做 2：云同步 / 多端同步
当前专注 local-first，不引入同步复杂度。

#### 不做 3：多人协作 / 企业权限模型
当前明确是个人桌面产品。

#### 不做 4：AI 自动标签 / OCR / 视频语义识别
这些都可能很有价值，但会显著抬高复杂度并模糊第一版目标。

#### 不做 5：重在线元数据依赖
例如自动联网抓取大量游戏、书籍、软件资料，不应成为第一版成立前提。

#### 不做 6：商店化 / 下载器 / 启动平台生态
产品是用户本地资产入口，不是分发平台。

#### 不做 7：复杂自动化规则引擎
例如自动搬运、自动命名、自动多条件归档等，后期再考虑。

### 19.7 MVP 上线验收标准（建议）
当以下问题的答案大多为“能”，即可认为 MVP 成立：

1. 用户能否在 10 分钟内完成扫描源设置并看到入库结果？
2. 用户能否比使用 Windows Explorer 更快找到目标文件？
3. 用户能否给文件加标签并通过标签再次找到它？
4. 用户能否在素材库中主要依靠缩略图而非文件名找素材？
5. 用户能否顺畅完成“查看详情 → 打开文件 / 打开目录”？
6. 用户是否愿意把该产品作为自己本地数字资产的常用入口之一？

如果前 5 条成立，而第 6 条在体验测试中开始出现明显正反馈，那么第一版就具备继续投入的价值。

---

### 19.8 当前结论
当前 MVP 不应该追求“什么都能做一点”，而应该追求：

> **先把本地文件资产的“找回、标记、再找回、可浏览”这条主链做透。**

这条主链一旦成立，后面的游戏库、书库、软件库、智能集合、自动化能力才有真正生长的土壤。

### 对应任务流

P0

P0

### 20.6 页面关系与跳转主链

#### 主链 A：首次使用

任意页面顶部搜索 → 搜索结果页 → 详情页 → 打开文件 / 打开目录

#### 主链 C：整理与再找回
最近导入 / 搜索结果 / 素材库 → 详情页 → 添加标签 → 标签页 / 素材库再次筛选

#### 主链 D：内容浏览
首页 / 素材库 → 卡片流浏览 → 详情页 → 打开文件

这些主链应成为原型设计时优先验证的路径。

---

### 20.7 页面设计优先原则

#### 原则 1：搜索入口必须全局常驻
因为搜索是最强感知价值之一。

#### 原则 2：详情能力必须统一
不要让不同页面拥有割裂的标记与跳转体验。

#### 原则 3：筛选器位置要稳定
特别是素材库与搜索结果页，否则用户难以建立操作习惯。

#### 原则 4：文件模式与库模式要有明显区分
前者强调结构，后者强调内容。

#### 原则 5：最近导入不能只是附属列表
它应被视为“整理入口”而不是“日志页面”。

### 20.8 当前页面级结论
当前最值得优先做透的页面不是最多，而是以下 5 个：
1. 扫描源配置页
2. 全局搜索结果页
3. 素材库页
4. 统一详情页
5. 最近导入页

原因：
- 它们共同构成第一版最关键的产品闭环
- 它们最直接体现相对 Windows 原生的差异化
- 它们也是最容易在原型阶段验证用户反馈的页面

### 21.3 页面 1：扫描源配置页（P0）

### 模块 C：添加扫描源操作区
内容：

优先级：高

### 模块 D：扫描操作区
内容：

- 当前扫描状态

### 模块 E：空状态引导区

优先级：高（仅空状态）

### 初始空状态布局

- 中部主卡片：添加扫描源按钮
- 下方辅助说明：支持管理图片、视频、游戏、电子书、软件等本地内容

### 已添加扫描源后的布局
- 顶部：标题 + 扫描按钮
- 主体：扫描源列表
- 侧边或下方：忽略目录 / 基础说明

## 必须考虑的页面状态
- 无扫描源
- 有扫描源但未扫描
- 扫描中
- 扫描完成
- 扫描失败（基础提示即可）

> **P0 页面交互细节文档（包括空状态、加载状态、批量选择、标签交互方式、侧栏开合逻辑）**

## 22. P0 页面交互细节文档（用于可点击原型与前端交互约束）

### 22.1 这一节的目标
这一节不再回答“有哪些页面”，而是回答：
- 页面里的操作如何发生
- 哪些动作是一击完成，哪些需要二次确认
- 空状态、加载状态、无结果状态如何表现
- 详情侧栏如何开合
- 标签与颜色标签如何高频操作
- 批量选择是否进入单独模式

这一节的产出目标是：

> **形成第一版统一交互语言**

这样后续原型与开发不会在细节上各自理解。

---

### 22.2 全局交互基线

#### 原则 1：单击用于“查看 / 选中”，双击用于“打开”
当前建议统一为：
- **单击文件 / 卡片**：选中，并在右侧展开或刷新详情侧栏
- **双击文件 / 卡片**：直接打开原文件

原因：
- 单击查看、双击打开，符合桌面文件场景习惯
- 有利于保留详情侧栏作为高频整理入口

#### 原则 2：详情侧栏默认按选中项展开
当前建议：
- 首次进入页面时，详情侧栏默认收起或显示占位提示
- 选中任意文件后，详情侧栏展开并展示该项信息
- 再次点击其他项时，侧栏刷新而不关闭
- 用户可手动关闭侧栏；关闭后再次选中文件时自动展开

#### 原则 3：搜索与筛选尽量“即时生效”
当前建议：
- 搜索输入在短暂 debounce 后更新结果
- 标签、颜色标签、类型筛选点击后立即生效
- 排序切换立即生效
- 不采用“设置完一堆条件再点应用”的重流程

#### 原则 4：高频操作尽量就地完成
当前建议：
- 打标签、改颜色标签、打开文件、打开目录这些动作，尽量在当前页就完成
- 减少跳出式弹窗
- 详情侧栏承担高频整理职责

#### 原则 5：批量操作进入“明确的选择状态”
当前建议：
- 不默认一直处于复杂批量模式
- 用户主动进入多选状态后，页面出现批量操作工具栏
- 退出多选后恢复普通浏览模式

---

### 22.3 全局状态设计基线

## 空状态
适用场景：
- 尚未添加扫描源
- 某页当前没有内容
- 某个标签下没有文件
- 最近导入为空

统一设计建议：
- 一句明确说明“为什么现在是空的”
- 一个明确下一步动作按钮
- 可选一行辅助说明

例如：
- “还没有扫描任何目录” → 按钮：添加扫描源
- “当前标签下还没有文件” → 按钮：查看全部内容
- “最近暂无新文件” → 按钮：返回素材库

## 加载状态
适用场景：
- 首次扫描后读取结果
- 搜索结果正在更新
- 缩略图尚未生成
- 页面初次进入时读取数据

统一设计建议：
- 页面骨架屏优先于纯转圈
- 列表页显示行骨架
- 卡片页显示卡片骨架
- 详情侧栏显示局部骨架

## 无结果状态
适用场景：
- 搜索词无结果
- 筛选条件组合过严

统一设计建议：
- 明确告诉用户“当前没有匹配结果”
- 回显当前筛选条件
- 提供快速操作：清空部分筛选 / 返回全部

## 失败状态
适用场景：
- 扫描失败
- 打开文件失败
- 读取缩略图失败

统一设计建议：
- 给出简明错误提示
- 提供重试入口
- 不在第一版堆叠复杂错误解释

---

### 22.4 全局选择与打开交互

#### 单个项目交互
- **单击**：选中项目，刷新详情侧栏
- **双击**：打开原文件
- **右键**（如第一版要做）：展示基础上下文菜单

#### 建议第一版上下文菜单最小项
1. 打开
2. 打开所在目录
3. 添加标签
4. 设置颜色标签
5. 复制路径（可选）

#### 多选交互
当前建议：
- 桌面端支持 `Ctrl` 多选、`Shift` 区间多选（如果实现成本可控）
- 同时提供显式“多选模式”按钮，便于非快捷键用户
- 进入多选模式后：
  - 顶部或底部出现批量操作条
  - 详情侧栏可切换为批量摘要态，或临时收起

#### 批量操作最小集合
1. 批量添加标签
2. 批量移除标签（可后置到 P1）
3. 批量设置颜色标签
4. 批量打开目录（不建议）

当前结论：
- 第一版若实现批量，只优先做“批量标签”和“批量颜色标签”
- 不建议一开始扩展太多批量文件操作

---

### 22.5 标签交互设计（P0 核心）

## 普通标签

### 建议交互方式
- 在详情侧栏的标签区展示已有标签
- 提供“添加标签”输入框或按钮
- 输入时可：
  - 匹配已有标签
  - 支持直接创建新标签
- 选中后立即生效，无需额外提交页

### 当前推荐模式
> **输入框 + 下拉候选 + 支持直接创建**

理由：
- 兼顾已有标签复用与新标签创建
- 适合高频、轻量操作

### 标签展示规则
- 已有标签使用胶囊样式
- 标签过多时可折叠显示“+N”
- Hover 或点击可展示完整列表（若第一版需要）

## 颜色标签

### 建议交互方式
- 在详情侧栏中提供固定颜色点或颜色胶囊选择器
- 点击某个颜色立即设置
- 点击当前颜色再次触发可取消，或提供“清除颜色标签”入口

### 当前推荐模式
> **固定颜色快捷条 + 明确清除入口**

理由：
- 颜色标签本质是快速状态标记
- 不适合藏在深层弹窗中

### 颜色数量建议
第一版建议固定少量颜色，例如：
- 红
- 黄
- 绿
- 蓝
- 紫
- 无颜色

不要第一版开放任意色盘，否则会削弱状态语义。

---

### 22.6 筛选器交互设计

## 筛选器位置
当前建议：
- 搜索结果页和素材库页的筛选器位置固定
- 不要放到深层抽屉里作为唯一入口
- 可折叠，但默认可见

## 筛选器行为
- 点击即生效
- 当前生效的筛选条件需要清楚回显
- 每个筛选条件可单独移除
- 应有“清空全部筛选”入口

## 当前推荐回显方式
- 在页面标题下或结果区上方展示筛选条件胶囊
- 例如：`图片` `蓝色标签` `赛博朋克`
- 每个胶囊都可单独关闭

## 排序交互
- 排序入口应常驻工具栏
- 切换排序后立即刷新结果
- 建议至少支持：
  - 最近修改
  - 名称
  - 大小（在适用页）

---

### 22.7 搜索交互设计

## 搜索输入行为
当前建议：
- 输入后短暂延迟再检索，避免每个字符都强刷新
- 回车可立即触发当前关键词搜索
- 点击清空按钮后恢复默认结果或空搜索态

## 搜索态分类
1. **未输入状态**：展示提示或默认内容
2. **输入中状态**：短暂 debounce
3. **有结果状态**：展示结果与筛选器
4. **无结果状态**：展示无结果提示与放宽建议

## 是否显示搜索建议
第一版建议：
- 可不做复杂搜索建议系统
- 先保证搜索结果本身够快、够清楚

---

### 22.8 详情侧栏交互设计

## 展开逻辑
当前建议：
- 页面初始默认关闭或显示占位态
- 单击项目后展开
- 在桌面宽屏下默认停靠右侧

## 收起逻辑
当前建议：
- 用户点击关闭按钮可收起
- 页面切换后默认回到各页面自己的最近状态
- 再次选中项目时重新展开

## 侧栏内部结构顺序
建议顺序：
1. 文件名与快速动作
2. 预览区
3. 标签区
4. 颜色标签区
5. 元数据区
6. 打开 / 打开目录动作区

原因：
- 标签与颜色标签是高频整理动作，应置于更靠前位置
- 路径等元数据重要，但不应压过整理动作

## 侧栏宽度建议
- 需要足够展示标签与路径
- 但不能吞掉主浏览区太多空间
- 原型阶段可先按固定宽度设计，后续再看是否允许拖拽调整

---

### 22.9 最近导入页交互设计

## 页面定位
这是“快速整理页”，不是单纯时间列表页。

## 关键交互建议
- 单击项目：详情侧栏展开
- 双击项目：打开原文件
- 支持快速连续为多个项目打标签
- 若实现批量：应优先在这里落地

## 当前推荐整理节奏
- 用户从上到下浏览最近内容
- 看到目标项 → 单击 → 侧栏加标签 / 颜色标签
- 处理完后点击下一项继续

这是一种非常适合桌面整理的连续操作节奏。

## 页面辅助交互
- 应有“仅看未打标签内容”的快捷筛选（可作为 P0.5 或 P1）
- 若第一版不做，也建议在文档中预留

---

### 22.10 素材库页交互设计

## 卡片交互
- 单击卡片：选中 + 右侧详情
- 双击卡片：打开原文件
- Hover：显示轻量信息或快速动作（若成本可控）

## 网格密度切换
当前建议：
- 提供 2~3 档密度，而不是无限缩放
- 例如：紧凑 / 标准 / 沉浸

原因：
- 有助于兼顾“高密度找素材”和“舒适浏览”两种需求

## 预览优先级
- 图片：直接缩略图
- 视频：封面图 + 时长角标（如能拿到）
- 其他不适合放入素材库的文件类型，应在入库规则层控制，而不是在页面里大量例外处理

---

### 22.11 全部文件页交互设计

## 目录树交互
- 单击目录：刷新中间文件列表
- 展开/收起目录：局部进行，不影响当前选中文件

## 文件列表交互
- 单击行：刷新详情侧栏
- 双击行：打开原文件
- 点击列头：切换排序

## 当前结论
全部文件页应保持克制，不要在第一版往 Explorer 全替代方向滑。

---

### 22.12 设置页交互设计

## 设置保存策略
当前建议：
- 用户修改设置后立即保存，减少“应用/取消”的传统表单负担
- 对高风险动作（如重建索引）可二次确认

## 高风险动作确认
适用动作：
- 删除扫描源
- 重建索引
- 清空缓存（若提供）

建议：
- 使用轻确认弹窗
- 文案明确说明影响范围

### 22.13 第一版最值得统一的交互组件

以下交互组件建议尽早定义行为规范：
1. 文件/卡片单击与双击行为
2. 详情侧栏开合逻辑
3. 标签输入与创建逻辑
4. 颜色标签选择器
5. 筛选器胶囊回显
6. 批量选择模式条
7. 空状态组件
8. 无结果组件
9. 错误提示组件

这些组件一旦统一，整个产品会更像“一个系统”，而不是一组拼起来的页面。

### 22.14 当前阶段最关键的交互结论

#### 结论 1
**单击看详情，双击打开文件** 应作为全产品统一规则。

#### 结论 2
**详情侧栏是第一版高频整理中心**，而不是辅助信息面板。

#### 结论 3
**标签和颜色标签必须能在少量点击内完成**，否则元数据层很难真正被使用。

#### 结论 4
**搜索与筛选应即时生效**，不要设计成重表单流程。

#### 结论 5
**最近导入页应优先支持连续整理节奏**，因为它最可能成为高频页面。

## 23. 前端信息架构草案（用于页面实现与组件拆分）

### 23.1 这一节的目标
这一节不再讨论“产品做什么”，而是开始回答：
- 前端页面如何组织
- 页面之间如何跳转
- 哪些状态应该全局持有，哪些只属于页面局部
- 哪些 UI 是通用组件，哪些是页面专属组件
- 如何在不把系统做碎的前提下支持文件模式与库模式

这一节的产出目标是：

> **把产品文档翻译成前端实现结构语言**

这不会直接限定技术栈，但会为 React / Vue / Electron / Tauri 等桌面前端方案提供共同的结构依据。

---

### 23.2 前端结构设计原则

#### 原则 1：页面层、领域层、通用组件层分开
前端不应把所有逻辑都塞在页面文件中。

建议分为：
1. 页面层（Page）
2. 领域功能层（Feature / Domain UI）
3. 通用组件层（Shared UI Components）
4. 应用壳层（App Shell）

#### 原则 2：搜索、标签、详情侧栏是跨页面共享能力
这些能力不应在每个页面各写一套。

#### 原则 3：文件模式与库模式共享底层对象，不共享全部界面结构
也就是说：
- 数据底层尽量统一（FileItem / LibraryItem）
- UI 表达可分为列表模式和卡片模式

#### 原则 4：页面状态尽量就近管理，全局状态保持克制
不要一开始把所有筛选器和页面视图都塞进一个超大的全局 store。

#### 原则 5：右侧详情侧栏应视为 App 级共享能力，但其内容由当前页面上下文驱动

---

### 23.3 路由结构建议

#### 一级路由（P0）
当前建议的页面路由骨架：
- `/` → 首页 / 入库概览页
- `/onboarding` → 首次引导 / 扫描源配置页
- `/search` → 全局搜索结果页
- `/files` → 全部文件页
- `/library/media` → 素材库页
- `/tags` → 标签与筛选入口页
- `/recent` → 最近导入页
- `/settings` → 设置页

#### 一级路由（P1）
- `/library/games` → 游戏库页
- `/library/books` → 电子书库页
- `/library/apps` → 软件库页
- `/collections` → 智能集合页

#### 详情页路由策略建议
有两种做法：

##### 做法 A：详情只存在于右侧侧栏
优点：
- 页面切换轻
- 整体更像桌面工作台

缺点：
- 深链接能力较弱

##### 做法 B：侧栏 + 独立详情页共存
例如：
- `/item/:id`
- `/library/media/:id`

优点：
- 更适合深链接和未来扩展
- 更适合小窗口或窄屏

缺点：
- 实现稍复杂

#### 当前建议
第一版建议：
> **以右侧详情侧栏为主，保留未来扩展独立详情页的结构可能，但不必先做重路由化详情页。**

---

### 23.4 应用壳层（App Shell）建议

## App Shell 职责
作为全局布局与跨页面共享交互的承载层。

### 建议包含的固定区域
1. 左侧主导航
2. 顶部全局栏
3. 中央页面容器
4. 右侧详情侧栏容器
5. 全局 Toast / 通知区
6. 全局弹窗层

### App Shell 负责的行为
- 全局搜索框展示与跳转
- 当前页面标题承载
- 详情侧栏的开合容器
- 页面间导航
- 全局快捷键（后期可扩展）

### App Shell 不应过度承担的逻辑
- 不应持有每个页面的全部筛选状态
- 不应直接管理页面专属的数据请求
- 不应塞入具体库页面业务逻辑

---

### 23.5 页面层与领域功能层拆分建议

## 页面层（Pages）
页面层职责：
- 路由入口
- 组合当前页面需要的功能模块
- 决定页面布局
- 与 App Shell 对接

### 建议页面列表
- HomePage
- OnboardingPage
- SearchPage
- FilesPage
- MediaLibraryPage
- TagsPage
- RecentImportsPage
- SettingsPage
- （P1）GamesLibraryPage / BooksLibraryPage / AppsLibraryPage

## 领域功能层（Features / Domain UI）
领域功能层职责：
- 封装与某一业务能力直接相关的 UI + 状态 + 交互
- 被多个页面复用

### 建议拆分的功能模块
1. SearchFeature
2. FileBrowserFeature
3. MediaLibraryFeature
4. TaggingFeature
5. RecentImportsFeature
6. SourceManagementFeature
7. DetailsPanelFeature
8. FilterBarFeature
9. SelectionModeFeature

这样做的好处：
- 页面层更薄
- 共享能力更自然
- 后续新增游戏库、书库、软件库时更容易沿用已有能力

---

### 23.6 共享组件层建议

以下组件建议尽早抽为共享组件：

#### 布局类
- AppSidebar
- TopBar
- PageHeader
- RightDetailsPanel
- PageEmptyState
- PageErrorState
- LoadingSkeleton

#### 展示类
- FileList

### 23.8 状态管理边界建议

## 应放在全局状态的内容
以下状态更适合放在全局 App 级状态中：
1. 当前选中详情项（selectedItemId）
2. 详情侧栏是否展开
3. 全局主题设置
4. 扫描源基础摘要（若多个页面都要展示）
5. 全局通知 / Toast

## 应放在页面局部状态的内容
以下状态更适合由页面自行管理：
1. 当前页面的筛选条件
2. 当前页面的排序方式
3. 当前页面的视图模式（列表 / 网格）
4. 当前页面的多选状态
5. 当前页面的临时搜索上下文

## 介于两者之间的状态
某些状态需要谨慎设计，例如：
- 全局搜索词
- 标签缓存列表
- 当前扫描状态

### 当前建议
- **全局搜索词**：由顶栏输入触发跳转到 `/search`，在 SearchPage 内真正持有和消费
- **标签列表缓存**：可由共享数据层缓存，但不一定要暴露为全局 UI store
- **扫描状态**：由 SourceManagement 领域层持有，App Shell 只消费摘要态

### 23.10 数据流建议

## 搜索页数据流
1. 顶部输入搜索词
2. SearchPage 接收并持有当前 query
3. SearchFeature 发起结果请求/筛选
4. 结果渲染到列表/网格
5. 用户单击某项
6. selectedItemId 更新
7. DetailsPanelFeature 根据 selectedItemId 读取详情并展示

## 标签操作数据流
1. 用户在详情侧栏添加标签
2. TaggingFeature / DetailsPanelFeature 提交更新
3. 当前详情数据刷新
4. 当前列表/卡片中的标签回显更新
5. 若当前页结果依赖该标签筛选，则按规则刷新结果

## 最近导入整理流
1. RecentImportsPage 拉取最近内容
2. 用户逐个选中内容
3. 详情侧栏中连续执行标签/颜色标签设置
4. 列表即时回显已处理状态

### 23.13 当前最应避免的前端实现问题

#### 问题 1：每个页面各自维护一套详情区逻辑
会导致交互与行为迅速割裂。

#### 问题 2：把所有状态都扔到全局 store
会导致状态互相污染，调试困难。

#### 问题 3：搜索、标签、筛选在不同页面使用不同交互组件
会削弱整个产品的一致性。

#### 问题 4：为了“以后可扩展”过早设计复杂状态机和微前端式拆法
当前阶段不需要。

#### 问题 5：文件模式和库模式完全割裂成两套系统
会让产品越来越像几个拼接工具，而不是一个统一工作台。

### 23.14 当前前端信息架构结论

当前最推荐的前端实现方向可以概括为：

1. **App Shell 统一壳层**
2. **Pages 负责布局与路由**
3. **Features 负责业务能力复用**
4. **Shared Components 负责交互与视觉统一**
5. **DetailsPanelFeature 作为跨页面核心能力**
6. **页面筛选与排序状态尽量局部管理**
7. **搜索、标签、筛选、详情侧栏尽早统一组件与行为**

这是一条足够稳、同时又不至于过度设计的前端骨架。

## 24. P0 核心组件规范文档（用于前端实现与设计统一）

### 24.1 这一节的目标

## 24.3 GlobalSearchInput

### 组件目标
作为整个产品最重要的全局输入入口，承载“快速找回内容”的第一动作。

### 使用页面
- App Shell 顶部

- 首页（作为主要入口）

### 第一版必须支持
- 输入关键词
- 清空输入
- 回车立即触发
- 输入后 debounce 触发搜索

### 输入（建议）
- value
- placeholder
- onChange
- onSubmit
- onClear

### 输出（行为）
- 更新当前关键词
- 触发跳转到 `/search`
- 触发搜索结果刷新

### 状态
- 默认态
- 输入中态
- 加载态（可轻提示）
- 有内容态

### 交互规则
- 在非 SearchPage 中使用时，提交后应跳转到 `/search`
- 在 SearchPage 中使用时，提交后应刷新当前结果
- 清空后应恢复默认搜索态或当前页定义的空搜索态

- 历史搜索下拉
- 智能纠错

### 24.20 第一版组件之间的优先关系

#### 第一优先级组件
1. GlobalSearchInput
2. TagPicker
3. ColorTagPicker
4. RightDetailsPanel
5. MediaCard

#### 第二优先级组件

这些组件重要，但不应抢占第一批主功能资源。

### 24.21 当前最应避免的组件设计问题

#### 问题 1：TagPicker 做成重量级管理弹窗
会显著提高打标签成本。

#### 问题 2：MediaCard 和 FileCard 不分
会导致通用组件越来越臃肿。

#### 问题 3：RightDetailsPanel 被页面私有化改造
会迅速破坏一致性。

#### 问题 4：FilterBar 过早走向复杂查询构建器
第一版不需要。

#### 问题 5：为了“通用性”把所有组件抽得过度抽象
第一版组件应该优先为当前产品服务，而不是假想未来所有可能场景。

### 24.22 当前组件规范结论
当前最核心的组件化方向可以总结为：

1. **搜索是第一入口** → GlobalSearchInput 必须稳
2. **标签是元数据核心** → TagPicker 必须轻
3. **颜色标签是差异化能力** → ColorTagPicker 必须快
4. **详情侧栏是跨页面中枢** → RightDetailsPanel 必须统一
5. **素材库是第一视觉战场** → MediaCard 必须好看且清晰
6. **筛选是内容组织关键** → FilterBar 必须稳定易懂

如果这 6 个组件先做对，第一版产品感就会明显成立。
