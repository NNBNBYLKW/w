# Windows 本地资产管理工作台 PRD v1

## 1. 文档信息

### 1.1 文档名称
Windows 本地资产管理工作台 PRD v1

### 1.2 文档目的
用于明确产品第一阶段的定位、目标用户、核心问题、MVP 范围、页面结构、交互原则与开发优先级，作为后续原型设计、技术方案与任务拆解的共同依据。

### 1.3 当前阶段定位
本版本聚焦 **MVP / 第一阶段成立条件**，目标不是做完整资源管理器替代品，而是验证一个可长期成立的本地数字资产入口。

---

## 2. 产品概述

### 2.1 产品一句话定位
一款面向 Windows 海量本地文件的现代化资产管理工作台，通过 **高速索引、标签系统、库式浏览、统一详情层与现代 UI**，帮助用户更快找回、整理和浏览自己的本地数字资产。

### 2.2 产品本质
本产品不是传统意义上的“更好看的资源管理器”，而是：

> **建立在 Windows 文件系统之上的本地元数据与库式浏览系统。**

其核心目标不是替代用户已有目录结构，而是在原始文件系统之上建立一层可搜索、可标记、可筛选、可浏览的组织层。

### 2.3 当前主轴定义
当前产品主轴明确为：

> **以本地资产库 / 本地数字内容工作台为核心，以文件管理增强作为底层支撑。**

---

## 3. 产品背景与问题定义

### 3.1 背景
Windows 原生资源管理器在面对大量本地文件时存在明显短板：
- 索引与搜索体验较弱
- 过度依赖目录路径与文件名记忆
- 缺乏类似 macOS Finder 的颜色标签体系
- 不适合大量图片 / 视频的内容型浏览
- 对游戏、电子书、免安装软件缺乏统一库式体验
- 现有专业文件管理器多数偏传统工具风格，缺乏现代化 UI 与内容浏览体验

### 3.2 核心问题
本产品要解决的首要问题是：

> **如何在不改变用户原有目录结构的前提下，建立一套高性能、本地可控、适合浏览和检索的元数据层。**

### 3.3 当前关键矛盾
1. **文件系统逻辑 vs 内容平台逻辑**
   - 文件系统强调路径、层级、真实位置
   - 内容平台强调标签、筛选、封面、集合、浏览
2. **高性能 vs 富交互**
   - 大量缩略图、元数据、筛选与预览会带来性能压力
3. **通用性 vs 垂直体验**
   - 需要统一管理图片、视频、游戏、电子书、软件，但又不能把所有内容揉成一个无差别列表

## 4. 产品定位与边界

### 4.1 我们是什么
- 面向 Windows 海量本地文件的现代化资产管理工作台
- 一个本地优先（local-first）的文件元数据平台
- 一个统一管理图片、视频、游戏、电子书、免安装软件的库式入口
- 一个兼顾“效率搜索”和“内容浏览”的桌面产品

### 4.2 我们不是什么
- 不优先做完整资源管理器替代品
- 不做云盘 / 网盘产品
- 不做多人协作型企业 DAM
- 不做图片编辑器 / 视频编辑器 / 电子书制作工具
- 不做游戏下载平台、电子书商店或分发平台

### 4.3 当前阶段的核心价值主张
1. 比 Windows 资源管理器更快地找到本地文件
2. 比目录树更灵活地组织本地内容
3. 比传统文件工具更适合长期浏览和管理个人数字资产
4. 用统一界面管理原本分散的图片、视频、游戏、电子书和软件

---

## 5. 产品原则

### 5.1 核心原则
1. **文件系统是真实事实层，产品数据库是元数据层**
2. **不强迫用户改变既有目录结构**
3. **搜索不是唯一入口，浏览同样重要**
4. **标签、状态、集合优先于死板目录记忆**
5. **内容库体验优先于传统工具感**
6. **性能优先于功能堆叠**
7. **现代 UI 是产品竞争力的一部分，不是后期装饰**

### 5.2 当前阶段的工程原则
1. 第一版优先建立统一主链，不追求全类型深度支持
2. 不为“未来可能扩展”过早引入重型抽象
3. 搜索、标签、筛选、详情侧栏必须尽早统一
4. 文件模式与库模式应共享底层对象与主壳层，而非发展为两套系统

## 6. 目标用户

### 6.1 核心用户画像

#### A. 素材重度用户
- 拥有大量图片、视频、参考素材
- 需要缩略图、标签、筛选、快速确认
- 更关心内容浏览而不是路径浏览

#### B. 本地收藏型用户
- 收藏游戏、电子书、绿色软件、教程资源
- 希望把零散内容变成“自己的数字库”
- 需要统一入口与库式体验

#### C. 效率型重度文件用户
- 拥有大量项目资料、文档、压缩包、多盘符内容
- 需要跨目录快速搜索、定位与基础整理

### 6.2 当前阶段优先服务用户
第一阶段最优先服务：
1. 素材重度用户
2. 本地收藏型用户

原因：
- 这两类用户最能感知标签、库视图、缩略图与现代 UI 的价值
- 也最容易形成持续使用习惯

## 7. 典型使用场景

### 7.1 场景 A：找某张图片或某段视频素材
用户记得素材的大致主题、风格、颜色或用途，但不记得路径。

目标：
- 通过搜索 + 标签 + 缩略图快速确认目标

### 7.2 场景 B：整理一批新导入的素材
用户刚下载或拷贝了一批图片 / 视频，希望顺手打标签和标记状态，以便以后好找。

目标：
- 在最近导入中连续整理内容

### 7.3 场景 C：把本地游戏当作统一库浏览
用户的游戏散落在多个盘符、启动器、模拟器目录中，希望像 Steam 库一样浏览与启动。

目标：
- 统一展示与快速定位

### 7.4 场景 D：管理电子书收藏
用户拥有大量 PDF / EPUB / MOBI 等文件，希望按封面、作者、标签而非混乱文件名浏览。

目标：
- 以“书库”方式而非“文件夹”方式看书

### 7.5 场景 E：管理便携软件
用户有大量绿色版工具，希望集中展示、分类并快速启动。

目标：
- 形成自己的工具库

### 7.6 场景 F：保留路径导向浏览
并非所有时候都需要标签与库视图，有时用户只是知道路径，希望更高效地浏览目录内容。

目标：
- 保留必要的文件模式，不强迫用户放弃路径心智

---

## 8. 用户故事（第一版重点）

### 8.1 第一版必须优先服务的用户故事
1. 作为一个拥有大量本地文件的用户，我希望能跨盘符快速找到文件，这样我不必记住它具体放在哪个目录。
2. 作为一个素材重度用户，我希望能给文件添加普通标签和颜色标签，这样我以后可以更轻松地再次找到它。
3. 作为一个图片 / 视频素材用户，我希望能以缩略图墙方式浏览内容，这样我可以像浏览内容平台一样找素材，而不是只看文件名。
4. 作为一个整理型用户，我希望能在最近导入里快速处理一批新内容，这样整理不会变成高负担任务。
5. 作为一个路径导向用户，我希望在需要时仍然可以按目录浏览文件，这样产品不会强迫我完全放弃原有习惯。

## 11. MVP 范围

### 11.1 MVP 目标
先验证三件事：
1. 在海量本地文件场景下，是否真的能比 Windows 原生更快找到内容
2. 标签 + 库视图是否能显著改善管理体验
3. 用户是否愿意把它作为个人数字资产入口长期使用

### 11.2 MVP 必做（P0）
1. 扫描源管理
2. 本地索引数据库
3. 全局搜索
4. 普通标签系统
5. 颜色标签系统
6. 素材库（图片 / 视频）基础版
7. 统一详情页 / 详情侧栏
8. 基础文件模式
9. 最近导入页
10. 现代深色 UI 基线
11. 打开原文件 / 打开所在目录

### 11.3 P1（第一版后尽快补齐）
1. 批量标签
2. 保存搜索条件 / 智能集合基础版
3. 收藏 / 状态 / 评分
4. 游戏库基础增强
5. 电子书库基础增强
6. 软件库基础增强
7. Dashboard 强化版

### 11.4 P2（中期增强）
1. 高级搜索表达式
2. 更完整的媒体元数据
3. 更完整的电子书元数据
4. 更完整的游戏 / 软件库元数据
5. 批量整理工作台
6. 拖拽式集合与组织能力

### 11.5 当前阶段明确不做
1. 不做完整资源管理器替代
2. 不做云同步 / 多端同步
3. 不做多人协作 / 企业权限模型
4. 不做 AI 自动标签 / OCR / 视频语义识别
5. 不做重在线元数据依赖
6. 不做商店化 / 下载器 / 启动平台生态
7. 不做复杂自动化规则引擎

## 14. 核心交互原则

### 14.1 点击规则
- **单击**：选中并展示详情
- **双击**：直接打开原文件

### 14.2 详情侧栏规则
- 默认按选中项展开
- 切换选中项时内容刷新而不重建
- 是第一版高频整理中心，而非附属信息面板

### 14.3 搜索与筛选规则
- 输入后 debounce 生效
- 回车立即搜索
- 筛选点击即生效
- 当前筛选条件用胶囊回显
- 支持逐项移除与一键清空

### 14.4 标签交互规则
#### 普通标签
- 输入框 + 下拉候选 + 支持直接创建
- 选择后立即生效

#### 颜色标签
- 固定颜色快捷条 + 明确清除入口
- 点击即设置

### 14.5 批量操作规则
- 仅在主动进入多选状态后出现
- 第一版若做批量，优先只做：
  - 批量添加标签
  - 批量设置颜色标签

### 14.6 页面状态规则
统一考虑：
- 空状态
- 加载状态
- 无结果状态
- 失败状态

优先使用：
- 骨架屏而非纯转圈
- 明确下一步动作而不是只提示“暂无内容”

## 18. MVP 验收标准

当以下问题的答案大多为“能”，即可认为 MVP 成立：

1. 用户能否在较短时间内完成扫描源设置并看到入库结果？
2. 用户能否比使用 Windows Explorer 更快找到目标文件？
3. 用户能否给文件加标签并通过标签再次找到它？
4. 用户能否在素材库中主要依靠缩略图而非文件名找素材？
5. 用户能否顺畅完成“查看详情 → 打开文件 / 打开目录”？
6. 用户是否愿意把该产品作为自己本地数字资产的常用入口之一？

## 19. 当前版本结论

当前 PRD v1 的核心结论可以浓缩为一句话：

> **第一版不追求做一个什么都能替代的文件管理器，而是先把本地文件资产的“找回、标记、再找回、可浏览”这条主链做透。**

只要这条主链成立，后续的游戏库、书库、软件库、智能集合、批量整理与更强自动化能力，才有真正生长的土壤。
