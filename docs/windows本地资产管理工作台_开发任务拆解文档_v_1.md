# Windows 本地资产管理工作台 开发任务拆解文档 v1

## 1. 文档目的

本文件用于将以下已完成文档正式转化为**可执行开发任务**：
- PRD v1
- 低保真线框文案 v1
- 技术架构初稿
- 数据库 Schema 与 API 草案 v1

当前目标不是继续扩展产品想法，而是明确：
- 第一阶段要按什么顺序开发
- 每个阶段包含哪些任务包
- 每个任务包依赖什么
- 每个任务包完成后怎么验收
- 哪些任务必须先做，哪些可以并行，哪些应后置

本文件的目标产物是：

> **一份可以直接指导实现排期的阶段化任务清单。**

---

## 2. 任务拆解原则

### 2.1 总原则
1. **先搭主链，再补体验，再补垂类**
2. **先搭数据与查询底座，再做页面表现层**
3. **尽量按可验收闭环拆任务，而不是按零散技术点拆碎片任务**
4. **每个阶段完成后都应具备可演示结果**
5. **优先保证扫描 → 搜索 → 标签 → 素材库 → 详情 → 打开文件这条主链成立**

### 2.2 当前阶段不建议的拆法
不建议按如下方式切任务：
- “先把所有页面都搭一层空壳”
- “先把所有数据库表都写完再说”
- “先把所有组件抽完再看业务”
- “先做游戏库、书库、软件库，再回头补主链”

因为这样很容易形成：
- 页面能点但没有主链
- 架构看起来完整但没有真正价值闭环
- 任务多但用户价值出现得很晚

---

## 3. 总体开发阶段建议

当前建议将第一阶段拆为 7 个主阶段：

### Phase 0：工程底座与运行链路
目标：项目能跑起来，桌面壳、前端、本地服务、数据库初始化链路建立。

### Phase 1：扫描源与索引底座
目标：能添加扫描源，能建立基础文件索引，能完成首次扫描与手动重扫。

### Phase 2：查询与详情主链
目标：能搜到文件、看详情、打开文件 / 打开目录。

### Phase 3：标签与颜色标签主链
目标：能为文件添加普通标签和颜色标签，并通过它们再次找回文件。

### Phase 4：素材库主链
目标：能以缩略图和筛选器方式浏览图片 / 视频内容。

### Phase 5：最近导入与整理入口
目标：让产品从“查找工具”变成“持续整理入口”。

### Phase 6：整体收口与 MVP 验收
目标：把页面、组件、状态、交互统一起来，形成可交付 MVP。

---

## 13. 当前阶段不建议进入的任务

1. 不进入 Shell 深度集成（右键菜单 / 默认文件管理器替换）
2. 不进入云同步
3. 不进入 AI 自动标签
4. 不进入复杂批量文件操作
5. 不进入重型垂类库（游戏时长、联网封面抓取、书籍完整元数据生态）
6. 不进入复杂规则引擎

这些任务即使有价值，也不属于第一阶段成立条件。

---

## 14. 当前最关键的验收闭环

在所有任务拆完之后，第一阶段最关键的用户闭环仍然只有这一条：

> **添加扫描源 → 扫描 → 搜索 → 查看详情 → 打标签 / 颜色标签 → 在素材库或标签页再次找到 → 打开文件 / 打开目录**

只要这条闭环顺畅，第一版就具备成立基础。

## 15. 当前结论

当前开发任务拆解的核心不是“列出尽可能多的功能”，而是把产品压缩成若干个**逐步显现价值的阶段**：

1. 先能跑
2. 再能接入文件
3. 再能找回文件
4. 再能组织文件
5. 再能浏览素材
6. 再让用户养成整理习惯
7. 最后统一收口成 MVP

这种拆法的优点是：
- 每个阶段都能验证价值
- 每个阶段都能演示
- 出现偏航时容易及时修正

## 16. 下一步建议

在开发任务拆解文档 v1 之后，最适合继续推进的方向有两个：

1. **前端工程目录与状态管理草案**
2. **PoC / 风险验证清单**

如果准备开始真正实现，最推荐下一步先做：

> **PoC / 风险验证清单 v1**

因为在这类本地资产管理产品里，最值得尽早验证的不是页面，而是：
- 扫描速度
- SQLite 查询速度
- 缩略图生成与缓存可行性
- 视频封面提取成本
- 大量文件下前端滚动与加载体验

这些风险一旦先验证清楚，后面的工程推进会稳很多。
