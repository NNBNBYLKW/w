# 软件库增强版（范围草案）v0.1

## 1. 文档目的

本文档用于定义后续阶段中的 **软件库增强版** 范围，作为当前项目后续开发阶段中的一份范围草案。

本草案的目标不是重定义产品，而是基于当前已经成立的系统骨架，在 **不破坏现有方法论** 的前提下，把当前 `Software` 从基础 subset listing 增强为一个更接近“软件库测试版”的可用入口。

---

## 2. 当前起点

当前项目已经是一个 Windows local-first 本地资产工作台，建立在真实文件系统之上，以 `files` 为统一索引对象，围绕以下主链工作：

`source setup -> scan/index -> search/browse/retrieve -> shared details -> tags/color tags/collections -> open actions`

当前 `Software` 已经作为真实 subset surface 存在，backend 已有 `GET /library/software` 路由；shared details、tags、color tags、collections 与桌面 open actions 也已形成复用链。

当前 `Software` 的语义仍应理解为：

- active indexed `.exe` / `.msi` / `.zip` 等 software-related files 的识别子集
- 基于 indexed files 的受控子集入口
- 继续复用 shared details、tags、color tags、collections、Search 和 open actions 的入口

而不是安装管理器、软件商店、包管理平台或完整 launcher hub。

---

## 3. 本阶段目标

本阶段目标是：

> 把当前 `Software` 从“software-related files subset listing”增强为一个真正可用于找工具、看工具、分组管理、轻量启动的 **软件库测试版**。

本阶段结束后，用户应明显感知到：

- 软件库不再只是通用文件页按扩展名筛出来的轻包装
- 用户可以主要依靠图标 / 标题 / 分类信息，而不是路径记忆，重新定位便携软件与相关文件
- tags、color tags 与基础筛选在软件库中具备真实使用价值
- 从“扫描进入系统”到“查看、标记、再次找到、打开”的链路更自然

---

## 4. 本阶段不做什么

本阶段明确不做以下方向：

- 安装管理器
- 卸载中心
- 包管理器
- 平台化 launcher hub
- 自动联网抓取重依赖软件元数据
- 软件更新器
- 商店化系统
- 独立软件对象中台
- 脱离 indexed files 的独立软件数据库
- 与当前 shared details 分叉的软件专属详情系统

这里的边界要刻意收紧，避免软件库增强过程中把系统推向“软件平台化”或“系统级工具中心”。

---

## 5. 本阶段应坚持的架构前提

本阶段默认继续坚持以下当前已验证的方法论：

1. `files` 仍是底座
2. `Software` 仍是 subset surface，而不是独立平台
3. shared details 仍是共同详情与动作面
4. Search / tags / color tags / collections 仍是软件库的横向复用组织层
5. 桌面 open actions 仍通过现有薄壳能力承接
6. 不引入重对象模型中台，不重写底层事实层

---

## 6. 用户价值定义

本阶段重点服务的不是“企业软件资产管理”，而是个人本地工具重度用户在以下场景中的高频任务：

- 找某个绿色版 / 便携版工具
- 记不住路径，但记得图标、名字或用途
- 想把常用工具集中成一个可浏览入口
- 对工具做轻量标记、分类、再找回
- 从软件库中快速打开软件或打开所在目录

---

## 7. 范围总览

本阶段建议分三批次推进：

### 批次 1：软件库展示密度增强
目标：让软件库首先看起来、用起来更像“个人工具库”，而不是文件列表的变体。

### 批次 2：软件动作与检索增强
目标：让用户能在软件库中更稳定地定位、确认并打开工具，而不是频繁退回 Search / Files。

### 批次 3：软件组织闭环增强
目标：让软件库与 Tags / Collections / Recent / shared details 的回流关系更顺，形成轻量管理闭环。

---

## 8. 批次 1：软件库展示密度增强

### 8.1 目标
强化软件库的“工具库浏览”属性，让用户主要通过图标、标题、分类感知和更合理的信息层级浏览软件相关文件。

### 8.2 包含内容

- 优化 `SoftwarePage` 或相应软件库页面的卡片流/列表表现
- 提供更明确的展示密度层级
- 提高图标、标题与软件类型提示在卡片中的主视觉优先级
- 优化空状态、加载状态、占位态
- 弱化路径在首屏中的主导地位，但不隐藏
- 保持单击查看详情、双击打开文件的统一交互规则，继续依赖 shared details 承接整理动作

### 8.3 不包含内容

- 自动安装/卸载流程
- 运行参数管理中心
- 多启动项复杂配置系统
- 平台账号联动

### 8.4 前端重点

- `Software` 相关页面与 `RightDetailsPanel` 协同增强
- 优先统一卡片/列表展示、选中态、空态、骨架态
- 不单独造“软件专属详情页系统”，仍以共享 details 容器为中心

### 8.5 验收口径

- 软件库首屏明显更接近“工具库”而不是“扩展名筛选结果”
- 用户在软件库中找工具时不再主要依赖路径
- 空状态、加载状态与占位态不再显得像未完成占位

---

## 9. 批次 2：软件动作与检索增强

### 9.1 目标
让软件库中的定位、确认和打开动作稳定、高频、低摩擦。

### 9.2 包含内容

- 强化软件库中的类型筛选
- 强化标签筛选、颜色标签筛选的可见度与操作路径
- 强化排序入口与浏览方式切换
- 提高软件卡片与 shared details 之间的信息协同
- 在 details 中更好展示路径、大小、修改时间、扩展名、来源目录等当前已有基础信息
- 强化 `Open file` 与 `Open containing folder` 在软件场景中的可达性

### 9.3 后端允许的最小补强

本阶段允许做 **少量后端补强**，但必须满足两个条件：

1. 仍然服务于当前 `GET /library/software` 与 `GET /files/{id}` 主链
2. 不引入新的重软件对象层

可接受的补强方向例如：

- 更稳定的软件列表字段补充
- 对现有过滤参数表达能力做小范围增强
- 对图标或展示字段做轻量补强

但不应出现新的软件中心化数据模型。

### 9.4 不包含内容

- 自动识别所有安装状态
- 外部平台软件资料抓取中心
- 更新检测系统
- 软件运行统计系统

### 9.5 验收口径

- 用户能在软件库中直接用标签、颜色标签、基础筛选缩小范围
- 打开软件或打开所在目录的路径更自然
- 用户不需要频繁跳回 Search 才能完成工具定位

---

## 10. 批次 3：软件组织闭环增强

### 10.1 目标
把软件库从“看工具的地方”推进到“能顺手管理个人工具库的地方”。

### 10.2 包含内容

- 强化软件库与 `Recent` 的联动
- 强化软件库与 `Tags` 的回流关系
- 强化软件库与 `Collections` 的回流关系
- 让从“刚扫描进入系统”到“打标签 / 颜色标签 / 再找回”的路径更顺
- 继续由 shared details 承接标签、颜色标签、打开文件、打开所在目录等动作

### 10.3 优先级说明

这一批不要求立刻引入完整的“收藏 / 状态 / 评分”体系，也不要求完整批量整理工作台，因为这些更适合放在后续“组织层与库体验补齐”阶段统一推进。

因此本批次更适合做：

- 轻量整理流程增强
- 页面之间的自然回流
- 标记后回显更新
- 软件库内连续处理节奏优化

### 10.4 不包含内容

- 完整批量整理工作台
- 收藏 / 状态 / 评分全量体系
- 智能集合引擎升级

### 10.5 验收口径

- 用户能从 Recent 进入软件库继续整理
- 用户能从软件库打标签后通过 Tags / Collections 再次找到工具
- 软件库已经具备“轻量工具库入口”的性质，而不是只有浏览功能

---

## 11. 页面与组件影响范围

### 11.1 主要影响页面

- `/software` 或对应软件库页面
- `/recent`（仅限与软件整理链路相关部分）
- `/tags`（仅限软件回流体验相关部分）
- `/collections`（仅限软件回流体验相关部分）

### 11.2 主要影响组件 / feature

- `Software` 对应页面 feature
- `DetailsPanelFeature`
- `FilterBarFeature`
- `TaggingFeature`
- `RecentImportsFeature`
- 软件库卡片 / 列表组件
- `RightDetailsPanel`
- 相关空状态 / 骨架态 / 无结果态组件

### 11.3 默认不新增

- 独立软件详情体系
- 独立软件选择体系
- 脱离 shared details 的软件操作浮层系统

---

## 12. 后端与数据层边界

### 12.1 默认复用

- `files` 统一索引对象
- 现有 software subset route
- 现有 `GET /files/{id}` 详情主链
- 现有 tags / color tags / collections / open actions 基础能力

### 12.2 允许的补强类型

- 对 `GET /library/software` 进行轻量字段增强
- 对软件过滤/排序做小范围增强
- 对 `GET /files/{id}` 的软件展示字段做小范围增强
- 对图标、显示标题等表现层协作做增强

### 12.3 明确禁止

- 新建软件中心表，作为正式主语义替代 `files`
- 在本阶段引入复杂软件运行时管理体系
- 引入独立软件库数据库或外部重元数据平台

---

## 13. 交互原则

本阶段软件库继续沿用当前建议中的统一交互语言：

- **单击**：选中并在右侧 details 中查看
- **双击**：打开原文件
- 筛选即时生效
- 标签和颜色标签尽量在当前页就地完成
- details 侧栏仍是高频整理中心，而不只是辅助信息面板

---

## 14. 进入前提

进入本阶段前，默认以下前提成立：

- 当前 `Software` route 已存在
- 当前 shared details 主链可用
- tags / color tags / collections 可复用
- 桌面 open actions 仍由薄壳承接
- 不需要先重写架构或重建事实层

---

## 15. 退出条件

本阶段结束时，至少应满足以下条件：

1. `Software` 已具备清晰的“软件库测试版”气质
2. 浏览、筛选、详情、整理四条链能够自然连通
3. 用户主要依靠图标 / 标题 / 分类感知，而不是路径记忆找工具
4. 标签、颜色标签与基础筛选在软件库中成为真实高频入口
5. 不依赖新增重对象模型
6. 未滑向安装管理器、商店化系统或平台 launcher 方向

---

## 16. 给 Codex 的执行约束

在编码执行时，应额外遵守以下约束：

- 不改写项目当前产品定位
- 不把 `Software` 做成独立平台
- 不分叉 shared details 体系
- 不新增脱离 `files` 的软件事实层
- 能复用现有 feature / shared components 的，优先复用
- 前端优先做“体验增强与链路打顺”，不是为了未来抽象过度设计

---

## 17. 本阶段后的自然下一步

本阶段完成并验收后，下一步应进入：

> **《组织层与库体验补齐（范围草案）》**

如果仍按“Books / Software 增强并行推进”的节奏，也可在软件库范围草案确认后，再视情况整理对应 Codex 执行版任务文档。

---

## 18. 当前待拍板的 6 个点

这份草案还需要最终拍板的点有：

1. 本阶段是否允许补少量 backend 过滤/字段能力
2. 软件库首版是否以卡片为主，还是卡片/列表双模式并行
3. 当前软件识别范围是否仍只收 `.exe / .msi / .zip`，还是允许扩一小步
4. details 侧栏里的软件信息增强做到什么程度
5. Recent 与 Software 的回流入口是否做成明确 CTA
6. 本阶段是否拆成 3 个独立 Codex 任务执行，还是 1 个整阶段任务执行
