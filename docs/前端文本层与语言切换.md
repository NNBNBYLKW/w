# 前端文本层与语言切换

## 文档目的

这份文档用于说明当前 Workbench 前端中：

- UI 文案为什么从组件里抽离出来
- 当前轻量语言切换是如何实现的
- 后续维护者在新增文案、补齐页面、增加语言时应该改哪里

它服务的是**前端工程维护、beta 收口和后续补齐协作**，不是为了定义一个完整国际化平台。

当前项目仍处于测试版阶段，这套能力的目标是：

- 减少高可见硬编码文案继续散落在 JSX 中
- 为中英文验证准备提供一个可用闭环
- 让后续继续补文案和补语言时不需要重新拆架构

## 当前能力范围

当前前端已经完成的文本层能力包括：

- 高可见 UI 文案已从组件中抽离到独立 locale 资源
- 统一通过 `t(key, params?)` 读取文本
- 当前已提供 `en` 与 `zh-CN` 两套 locale
- `Settings` 中已有显式语言切换入口
- locale 选择会写入本地存储
- 应用初始化时会恢复用户上次选择的 locale

当前这套能力是**轻量前端文本层 + locale 切换闭环**，而不是完整国际化平台。

## 当前目录结构

当前相关目录主要分为两部分：

### locale 资源

- `apps/frontend/src/locales/en/`
- `apps/frontend/src/locales/zh-CN/`

当前两套资源都按相同语义拆分：

- `common.ts`
- `navigation.ts`
- `shell.ts`
- `pages.ts`
- `settings.ts`
- `details.ts`
- `features.ts`
- `index.ts`

这些文件负责：

- 保存静态 UI 文案
- 按语义组织当前前端 copy
- 作为后续新增语言时需要对齐的 shape 模板

一般来说：

- 新增或修改导航文案，优先改 `navigation.ts`
- 壳层、顶部、桌面标题栏相关文案，优先改 `shell.ts`
- 页面 header 与页面级说明，优先改 `pages.ts`
- Settings 页的设置项、说明与字段名，优先改 `settings.ts`
- DetailsPanel 的静态标签和说明，优先改 `details.ts`
- 更具体的功能区块文案，优先改 `features.ts`

### 文本入口与 locale 运行时

- `apps/frontend/src/shared/text/index.ts`
- `apps/frontend/src/shared/text/runtime.ts`
- `apps/frontend/src/shared/text/LocaleProvider.tsx`
- `apps/frontend/src/shared/text/types.ts`

这些文件负责：

- 提供统一的 `t(key, params?)` 入口
- 提供当前 locale 的运行时字典选择
- 提供 `LocaleProvider` 和 `useLocale()`
- 提供 `LocaleCode`、`TextKey` 等轻量类型支持

当前 locale 持久化 key 为：

- `WORKBENCH_LOCALE`

当前已经接入文本层和 locale 切换的高可见区域包括：

- 左侧导航
- 顶部壳层
- 桌面标题栏
- 页面 header

这意味着当前 beta 验证里最容易被看到的前端主路径文案，已经可以跟随 `en / zh-CN` 切换。

## 当前设计原则

当前这套能力有明确边界：

- 这是**轻量前端文本层**，不是重型国际化平台
- 不做后端 locale 协商
- 不做远程语言包
- 不做浏览器语言自动识别
- 不做 Electron 单独存储层
- 不引入重型第三方 i18n 平台
- 当前优先服务 beta 阶段体验统一和真实验证

这也意味着当前实现更关注：

- 主路径是否可维护
- 高可见区域是否能切换
- 后续补齐新语言是否容易继续推进

而不是一次性解决所有国际化复杂问题。

## 后续维护指南

### 新增一个静态文案

推荐顺序：

1. 找到对应语义文件，例如 `pages.ts`、`features.ts` 或 `settings.ts`
2. 在 `en` 和 `zh-CN` 中同时补齐同一 key
3. 在组件中使用 `t(key)` 或 `t(key, params)`
4. 不要直接把新字符串再写回 JSX

### 修改一个已有文案

优先改 locale 文件，不要优先改组件。

如果只是文字调整：

- 先改 `en`
- 再同步改 `zh-CN`
- 保持 key 不变，除非原 key 语义已经明显不合适

### 新增一个页面接入 `t()`

推荐做法：

1. 先识别页面内哪些是高可见静态文案
2. 把这些文案放进对应 locale 文件
3. 在页面组件里改成 `t(key)` 读取
4. 如果有顶层常量在模块加载时就调用了 `t()`，改成在 render 内计算，或改成工厂函数

这是为了避免 locale 切换后页面仍保留旧语言结果。

### 新增一种语言

推荐做法：

1. 在 `apps/frontend/src/locales/` 下新增新的语言目录
2. 复制 `en` 当前结构作为 shape 模板
3. 保持 `index.ts` 聚合方式一致
4. 在 `shared/text/runtime.ts` 中把新 locale 加入字典映射与 `LocaleCode`
5. 再决定是否需要在 `Settings` 中暴露新的选项

新增语言时，不要顺手重做文本层架构。

### 哪些文本当前可以后置

当前可以继续后置的内容包括：

- 低可见动态提示
- 分支很多的复杂 retrieval hints
- API 返回的错误消息
- 纯开发期或调试导向文案

后置这些内容是为了保持这套能力足够轻，不把当前 beta 阶段工程整理扩成完整消息平台。

## 注意事项 / 常见坑

- 不要把 key 直接渲染到页面
- 保持 `en` 和 `zh-CN` 的 shape 尽量一致
- 不要一边继续硬编码，一边局部使用 `t()`
- 不要把这套能力误扩成完整国际化平台
- 文案调整优先改 locale 文件，不要回退到 JSX 内联字符串
- 如果一个常量在模块顶层就执行了 `t()`，locale 切换后可能不会即时更新，应改到 render 或工厂函数内
- 当前唯一显式语言切换入口是 `Settings`，不要在其他位置额外扩散入口，除非需求明确变化

## 当前仍未完全覆盖的部分

当前仍有少量低优先级文案没有完全抽离或没有完全纳入 locale 切换闭环，例如：

- `CollectionsFeature` 中更复杂的集合编辑与兼容结果文案
- `DetailsPanelFeature` 中分支较多的 retrieval hint 文案
- 部分 API 层错误消息
- 低可见或偏开发向提示

这些内容目前不应被写成“已经全部完成”。它们可以在不改变当前文本层结构的前提下，后续继续按同一方式补齐。
