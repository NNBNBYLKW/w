# 测试版验证准备

## 1. 文档目的

本文件用于支持当前项目进入**测试版验证**阶段。

它的目标不是继续新增功能，而是把当前测试版整理成一套：

- 可演示的主路径
- 可验证的核心问题
- 可复用的观察模板
- 可回收的优先级判断方法

## 2. 当前默认前提

当前默认前提如下：

- 测试版范围已经冻结
- 四条垂类库线已经成立
  - Media
  - Games
  - Documents（内部兼容 route / placement 仍为 `books`）
  - Software
- 组织层最小闭环已经成立
  - batch organize
  - tags / color tags
  - favorite / rating
  - collections
  - recent family
- 体验统一已经完成基础收口
- 近期 Documents、分类与手动归类、Games preview、视频预览、语言即时刷新和深色模式修复均已通过验收

这意味着当前阶段重点应是：

- 设计验证路径
- 观察真实使用问题
- 记录反馈
- 给问题分优先级

而不是继续补大功能。

最近这轮前端改动也应按这个阶段理解：主要是在统一三栏工作台、导航 / details 控制、状态提示和整体视觉表达，为外部验证减少理解噪音。

当前 beta 包的验证还需要额外确认一体化运行前提：

- 一体化桌面包应自动启动本地 backend，packaged mode 默认地址为 `http://127.0.0.1:8765`
- 视频缩略图与 details 内 6 帧循环预览应默认使用随包携带的 `ffmpeg.exe`
- 开发模式仍可手动启动 backend，默认地址为 `http://127.0.0.1:8000`

## 3. 本阶段重点验证什么

外部验证建议集中看以下 5 类问题：

### 3.1 用户是否能快速理解产品是什么

- 用户第一次打开时，能否理解这是一个什么产品
- 用户是否会误判成：
  - 通用文件管理器
  - 游戏平台
  - 电子书阅读器
  - 软件安装管理器
  - AI 工具

### 3.2 用户是否能感知垂类库的价值

- Media / Documents / Games / Software 是否真的像独立入口
- 它们是否比“都堆进 Files”更有价值
- 哪个入口最强，哪个最弱

### 3.3 用户是否能完成主路径

- 浏览 -> 查看 details -> 打开 -> 标记 -> 再找回 是否顺畅
- batch organize 是否比逐个处理更有效
- 从 Recent / Tags / Collections / Details 回到 subset 是否自然

### 3.4 用户是否觉得组织层有价值

- tags / color tags / favorite / rating / collections / recent family 是否真的帮助组织内容
- 用户是否会主动用 collections
- favorite / rating 是否有存在感

### 3.5 用户是否感知到系统边界清楚

- 用户是否频繁期待当前不该有的能力
- 哪些页面最容易引发“为什么没有阅读器/启动器/安装器”的预期
- 是否存在表达过强导致误判产品类型的地方

### 3.6 用户是否能自然理解当前工作台界面

- 用户是否能快速理解左侧导航、中间浏览区、右侧 details 的三栏结构
- 左侧导航的展开 / 收起是否自然
- details 的显示 / 隐藏和中间区补位是否容易理解
- 顶部 backend 状态图标和 details toggle 是否比早期文字提示更自然、更不打断主路径

## 4. 建议验证对象

建议优先找以下用户：

- 有较多本地内容的人
  - 图片 / 视频素材
  - 本地游戏
  - 电子书文件
  - 软件安装包或便携软件
- 不参与项目开发的人
- 愿意边操作边说出想法的人

优先做可观察、可追问、可复盘的验证，而不是只发一个链接让对方自己试。

## 5. 建议验证路径

每次验证建议优先走固定主路径，而不是自由漫游。

### 路径 A：Recent -> 整理 -> 再找回

1. 打开 Recent
2. 进入 batch mode
3. 批量加标签 / 颜色标签
4. 退出 batch mode
5. 通过 Tags / Collections / subset 再找到对应内容

### 路径 B：Media 主路径

1. 进入 Media
2. 浏览 / 筛选
3. 查看 details
4. 做整理动作
5. 通过 recent / collections 再找回
6. 若样本中有视频，观察单帧缩略图与 details 内 6 帧循环预览是否降级自然

### 路径 C：Games 主路径

1. 进入 Games
2. 浏览
3. 打开游戏条目
4. 查看 details
5. 调整 status
6. 再按 status / tag 找回

### 路径 D：Documents / Software 主路径

1. 进入 Documents 或 Software
2. 浏览并确认条目语义
3. 查看 details
4. 写入 favorite / rating / tag / collection
5. 通过 recent / collections 再找回

### 路径 E：Library 文件库主路径

1. 进入 Library
2. Overview tab 确认统计摘要可读
3. 切换到 Objects tab，浏览对象列表，打开对象详情
4. 切换到 Pending tab，查看候选列表，执行 Scan Candidates 和 Generate Plan
5. 切换到 Plans tab，查看计划列表，打开计划详情
6. 检查 Plan Detail 中的 before/after path preview、conflict status
7. 对 ready 计划执行 Preflight → 用户确认 → Execute
8. 检查 execution logs 中各 action 状态
9. 观察 completed / completed_with_errors / failed 状态的正确表达

### 路径 F：Tools Video Merge 主路径

1. 进入 Tools
2. 确认 Current Tool tab 显示 video_merge 表单
3. 通过文件拖拽或 indexed file picker 添加视频输入
4. 调整合并顺序、输出文件名和目录
5. 选择 copy 或 reencode 模式
6. 点击 Start，观察 run 进入 In Progress tab
7. 等待 run 完成，确认 Completed tab 中显示输出路径
8. 确认 run log/error 可读

### 路径 G：视图与外观偏好

1. 在 Search / Documents / Software / Media / Games 中确认默认进入 Details 视图
2. 切换到 Icons 视图，确认单击仍更新 shared details，双击仍走 open action
3. 刷新或重启后确认每页视图选择被本地保留
4. 在 Settings 中切换 English / 简体中文，确认 Sidebar、Shell、当前页面和 DetailsPanel 即时刷新
5. 在 Settings 中切换 Light / Dark，确认 shell、Home / Search / Files / Recent、icon grid、DetailsPanel 和表单控件可读

## 6. 单次验证应记录什么

每次外部验证建议固定记录这些观察项：

### 理解类

- 用户第一句话怎么描述这个产品
- 用户有没有理解错定位

### 路径类

- 用户在哪一步停顿最久
- 用户在哪一步最自然
- 用户有没有走错入口
- 用户是否能快速理解三栏结构和 details 补位逻辑

### 组织层类

- 用户是否主动使用 tags / collections / favorite / rating
- 哪些组织动作最难理解
- 哪些组织动作最有价值感

### 垂类差异类

- 哪个 subset surface 最强
- 哪个最弱
- 哪个最不像独立入口

### 预期偏差类

- 用户最常期待哪些当前没有的能力
- 这些期待是功能缺失，还是表达误导

### 当前 UI 一致性类

- 导航展开 / 收起是否容易发现和理解
- details toggle 是否足够自然
- details 手动调整宽度是否自然，隐藏 / 再显示后是否保持可用
- backend 状态提示是否克制而清楚
- 当前界面是否更像统一工作台，而不是松散页面集合
- Details / Icons 视图切换是否容易理解，刷新后是否保持
- Light / Dark 主题切换是否可读且不影响语言切换
- Home / Search / Files / Recent 在深色模式下是否仍有浅色块、不可读文字或不协调 hover / selected 状态

### 发布包与运行环境类

- 一体化 beta 包是否能自动启动 backend
- backend 未 ready 或启动失败时提示是否可理解
- Electron beta 包是否能启动到工作台
- 无 saved source 时是否进入 Onboarding
- 有 saved source 时是否进入 Home
- open file / open containing folder 是否在桌面壳中可用
- 不配置系统 `PATH` 中的 `ffmpeg` 时，随包 FFmpeg 是否仍可支持视频缩略图
- 异常视频时，视频预览是否仅降级而不影响主路径

## 7. 开发模式启动验收

开发模式推荐从仓库根目录启动：

```powershell
.\start-dev.ps1 -StartDesktop
```

或直接双击：

```text
start-dev.bat
```

预期启动内容：

- Backend：`http://127.0.0.1:8000`
- Frontend：`http://127.0.0.1:5173`
- Desktop：`npm run dev`

### 正确停止

关闭 backend 前，应在 `Backend Dev` 窗口按 `Ctrl+C`。如果出现：

```text
Terminate batch job (Y/N)?
```

请输入 `Y`。

不要只关闭 PowerShell 窗口。`uvicorn --reload` 可能同时存在 reloader 父进程和 server 子进程；旧进程残留会导致后续访问 `http://127.0.0.1:8000` 时命中旧 backend。

### 启动前端口检查

```powershell
Get-NetTCPConnection -LocalPort 8000 -ErrorAction SilentlyContinue |
  Select-Object LocalAddress, LocalPort, State, OwningProcess

Get-NetTCPConnection -LocalPort 5173 -ErrorAction SilentlyContinue |
  Select-Object LocalAddress, LocalPort, State, OwningProcess
```

必要时清理旧端口进程：

```powershell
Get-NetTCPConnection -LocalPort 8000 -ErrorAction SilentlyContinue |
  ForEach-Object {
    Stop-Process -Id $_.OwningProcess -Force -ErrorAction SilentlyContinue
  }

Get-NetTCPConnection -LocalPort 5173 -ErrorAction SilentlyContinue |
  ForEach-Object {
    Stop-Process -Id $_.OwningProcess -Force -ErrorAction SilentlyContinue
  }
```

### 启动后后端指纹检查

```powershell
$base = "http://127.0.0.1:8000"

Invoke-RestMethod "$base/debug/runtime" -TimeoutSec 5 | ConvertTo-Json -Depth 8

Invoke-RestMethod "$base/debug/thumbnails/warmup" -TimeoutSec 5 | ConvertTo-Json -Depth 8
```

通过标准：

- `sys_executable` 指向 `G:\Windows\Documents\GitHub\w\.venv\Scripts\python.exe`
- `process_id` 存在
- `process_start_time` 存在
- `/debug/thumbnails/warmup` 中能看到 `pdf_render_mode=subprocess-v1`
- `service_module_path` / `service_module_mtime` / `pdf_render_command_kind` 等实现指纹存在

异常表现：

- `sys_executable` 是 `C:\Python314\python.exe`：说明没有使用 `.venv`，或访问到了旧进程。
- `process_id` / `process_start_time` 缺失：说明当前 backend 可能是旧代码。
- `pdf_render_mode=subprocess-v1` 缺失：说明 thumbnail warmup 当前不是最新实现。

### Python 版本注意

backend 推荐使用 Python 3.12 的项目根 `.venv`。Python 3.14 可能在 app 导入 ORM model 阶段触发 SQLAlchemy 类型解析错误：

```text
TypeError: descriptor '__getitem__' requires a 'typing.Union' object but received a 'tuple'
```

该错误不是 PDF thumbnail 本身的问题。不要手改 `pyvenv.cfg`，也不要复制 `python.exe` 到 venv 中；如需更换 Python 版本，应按项目当前约定准备 `.venv`。

## 8. PDF thumbnail warmup 后端验收

该验收用于确认后端 PDF 第一页缩略图 warmup queue 能在当前会话内生成 cache。它不是前端 UI 验收。

清空 PDF cache：

```powershell
Remove-Item -Recurse -Force "G:\Windows\Documents\GitHub\w\apps\backend\data\thumbnails\pdf" -ErrorAction SilentlyContinue
```

重新查询当前 Documents 兼容 route `/library/books` 第 2 页 ids。不要复用旧 `file_id`：

```powershell
$base = "http://127.0.0.1:8000"
$books = Invoke-RestMethod "$base/library/books?page=2&page_size=50&sort_by=modified_at&sort_order=desc"
$ids = @($books.items | ForEach-Object { [int]$_.id })
$body = @{ file_ids = $ids } | ConvertTo-Json -Compress
```

轮询 warmup：

```powershell
for ($i = 1; $i -le 20; $i++) {
  $warm = Invoke-RestMethod -Method Post `
    "$base/files/thumbnails/warmup" `
    -ContentType "application/json" `
    -Body $body `
    -TimeoutSec 10

  "{0:HH:mm:ss} cached={1} queued={2} in_progress={3} unsupported={4} missing={5} failed={6}" -f `
    (Get-Date),
    $warm.cached.Count,
    $warm.queued.Count,
    $warm.in_progress.Count,
    $warm.unsupported.Count,
    $warm.missing.Count,
    $warm.failed.Count

  Start-Sleep -Seconds 5
}
```

通过标准：

- 第一次可能 `queued≈PDF数量`
- `cached` 应逐步增加
- `missing=0`
- `failed` 不应成批出现
- 对 cached 的 id，`GET /files/{id}/thumbnail` 返回 `200 image/png`

常见误判：

- 旧 `file_id` 返回 `FILE_NOT_FOUND` 或 warmup `missing`，通常说明当前 DB 已变化，需要重新查询当前页 ids。
- `missing` 表示 DB 中没有该 id 或源文件不存在，不等于 PDF 渲染失败。
- `failed` 表示找到文件但生成失败，应继续看 `/debug/thumbnails/warmup/{file_id}`。
- `THUMBNAIL_PENDING` 表示 cache miss 后已进入队列或等待生成。
- `THUMBNAIL_NOT_AVAILABLE` 表示当前实现认为该文件不可生成或生成失败。
- 后端 `cached` 正常但前端仍不显示时，才进入前端 thumbnail 状态同步排查。

## 9. 当前功能验收清单

### 9.1 Documents 验收

- Sidebar 显示 `Documents / 文档`。
- PDF / EPUB / MOBI / AZW3 仍出现在 Documents。
- DOCX / XLSX / CSV / PPTX / TXT / MD 等常见文档格式出现在 Documents。
- Excel 文件不会导致列表、详情、icon 视图崩溃。
- 内部 `/library/books` 和 `books` placement 仍作为兼容命名可用，不代表用户侧仍叫 Books。

### 9.2 Classification / manual placement 验收

- 普通 `.zip` / `.rar` / `.7z` 默认只在 Files，且可通过 Files Archives quick filter 找到。
- 普通 archive 默认不进入 Software，也不进入 Games。
- DetailsPanel 可把单文件设置为 Auto / Documents / Media / Games / Software / Files only。
- Batch organize 可批量设置 placement。
- `Auto` 清空 `manual_placement`；`Files only` 会排除出 Media / Documents / Games / Software smart views。
- scan / backfill / auto classification 不覆盖已有 `manual_placement`。

### 9.3 Games preview 验收

- 手动把 `.exe` 放入 Games 后显示现有 exe icon thumbnail。
- 手动把 `.zip` / `.7z` 放入 Games 后显示 fallback，不白屏，不无限 retry。
- 手动把图片 / 视频 / PDF 放入 Games 后显示真实缩略图或安全 fallback。
- Games icon / list / details 视图均不崩溃。
- DetailsPanel preview 对 Games item 正常。

### 9.4 Video preview 验收

- 长视频多帧预览覆盖前中后时间段，不集中在开头连续帧。
- 当前 DetailsPanel video preview 保持 6 帧，基于 ffprobe duration 做内部均匀采样。
- ffprobe / ffmpeg 输出解码不再出现 Windows GBK `UnicodeDecodeError`。
- 旧视频需要 rescan metadata 或刷新 video preview cache 后，才会获得 duration 驱动的均匀采样。
- 该能力不影响图片 / PDF / exe thumbnail。

### 9.5 i18n 验收

- Settings 中 English -> 中文后，Sidebar 即时刷新。
- 中文 -> English 后，DetailsPanel 即时刷新。
- 当前主面板和 Shell 可见文案即时刷新。
- 不需要切换页面、刷新浏览器或重启桌面端。
- 不破坏 theme / routing / selection / details open state。

### 9.7 Library organize 验收

- Library Objects tab 可浏览对象列表，可过滤 object type / review status
- Object detail 显示 type、root path、metadata source、asset.yaml status、members
- Pending tab 可执行 Scan Candidates 并显示候选列表（confidence、reason、suggested type）
- Plans tab 可 Generate Plan、查看 action 列表、before/after path
- Mark Ready 不执行文件操作
- Preflight 返回 source/target path 检查结果
- Execute 需要二次确认（confirmation modal），执行后显示 per-action 日志
- completed / completed_with_errors / failed 状态正确表达

### 9.8 Tools 验收

- Tools 页面显示 Video Merge 工具
- 支持 indexed_file 和 external_path 输入源
- 视频文件可通过拖拽或 picker 添加，排列顺序即合并顺序
- copy / reencode 模式可选
- 输出不会覆盖已有文件
- In Progress / Completed tab 状态与 run log 可读

### 9.9 Dark mode 验收

- Home / Search / Files / Recent 深色模式无异常浅色块。
- 文字、secondary metadata、status pill 可读。
- hover / selected / border 状态协调。
- light mode 未退化。

## 11. 反馈优先级规则

### P0：阻断型问题

- 用户无法完成主路径
- 页面让用户误判产品是什么
- 核心功能无法理解或无法操作

### P1：高价值收口问题

- 主路径可走但明显卡顿
- subset surface 表达不够成立
- 组织层能力存在但不自然

### P2：后置优化问题

- 更深垂类语义
- 更强视觉表达
- 更顺滑细节体验

### P3：暂不跟进问题

- 明显超出当前测试版边界
- 平台化要求
- AI / 推荐 / 阅读器 / 安装管理器等方向

## 12. 当前完成标志

本阶段完成的标志不是“再做一个页面”，而是：

- 已有固定主路径可用于外部验证
- 已有固定问题清单
- 已有统一反馈模板
- 已有优先级分类方法
- 已准备好做真实用户测试

## 13. 当前结论

测试版验证准备阶段的核心是：

> 把“能跑的测试版”变成“能被验证的测试版”。

因此，本阶段一切工作都应服务于：

- 提高验证质量
- 提高反馈可回收性
- 防止验证结果把项目重新带偏

最近的 UI/UX 收口也应服务于这一点：帮助外部验证更聚焦在主链和边界，而不是被不一致的壳层表达分散注意力。
