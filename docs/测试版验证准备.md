# 测试版验证准备

## 1. 文档目的

本文件用于支持当前项目进入**测试版验证**阶段。

它的目标不是继续新增功能，而是把当前测试版整理成一套：

- 可演示的主路径
- 可验证的核心问题
- 可复用的观察模板
- 可回收的优先级判断方法

## 2. 当前默认前提

当前默认前提如下：

- 测试版范围已经冻结
- 四条垂类库线已经成立
  - Media
  - Games
  - Books
  - Software
- 组织层最小闭环已经成立
  - batch organize
  - tags / color tags
  - favorite / rating
  - collections
  - recent family
- 体验统一已经完成基础收口

这意味着当前阶段重点应是：

- 设计验证路径
- 观察真实使用问题
- 记录反馈
- 给问题分优先级

而不是继续补大功能。

最近这轮前端改动也应按这个阶段理解：主要是在统一三栏工作台、导航 / details 控制、状态提示和整体视觉表达，为外部验证减少理解噪音。

当前 beta 包的验证还需要额外确认一体化运行前提：

- 一体化桌面包应自动启动本地 backend，packaged mode 默认地址为 `http://127.0.0.1:8765`
- 视频缩略图与 details 内 6 帧循环预览应默认使用随包携带的 `ffmpeg.exe`
- 开发模式仍可手动启动 backend，默认地址为 `http://127.0.0.1:8000`

## 3. 本阶段重点验证什么

外部验证建议集中看以下 5 类问题：

### 3.1 用户是否能快速理解产品是什么

- 用户第一次打开时，能否理解这是一个什么产品
- 用户是否会误判成：
  - 通用文件管理器
  - 游戏平台
  - 电子书阅读器
  - 软件安装管理器
  - AI 工具

### 3.2 用户是否能感知垂类库的价值

- Media / Games / Books / Software 是否真的像独立入口
- 它们是否比“都堆进 Files”更有价值
- 哪个入口最强，哪个最弱

### 3.3 用户是否能完成主路径

- 浏览 -> 查看 details -> 打开 -> 标记 -> 再找回 是否顺畅
- batch organize 是否比逐个处理更有效
- 从 Recent / Tags / Collections / Details 回到 subset 是否自然

### 3.4 用户是否觉得组织层有价值

- tags / color tags / favorite / rating / collections / recent family 是否真的帮助组织内容
- 用户是否会主动用 collections
- favorite / rating 是否有存在感

### 3.5 用户是否感知到系统边界清楚

- 用户是否频繁期待当前不该有的能力
- 哪些页面最容易引发“为什么没有阅读器/启动器/安装器”的预期
- 是否存在表达过强导致误判产品类型的地方

### 3.6 用户是否能自然理解当前工作台界面

- 用户是否能快速理解左侧导航、中间浏览区、右侧 details 的三栏结构
- 左侧导航的展开 / 收起是否自然
- details 的显示 / 隐藏和中间区补位是否容易理解
- 顶部 backend 状态图标和 details toggle 是否比早期文字提示更自然、更不打断主路径

## 4. 建议验证对象

建议优先找以下用户：

- 有较多本地内容的人
  - 图片 / 视频素材
  - 本地游戏
  - 电子书文件
  - 软件安装包或便携软件
- 不参与项目开发的人
- 愿意边操作边说出想法的人

优先做可观察、可追问、可复盘的验证，而不是只发一个链接让对方自己试。

## 5. 建议验证路径

每次验证建议优先走固定主路径，而不是自由漫游。

### 路径 A：Recent -> 整理 -> 再找回

1. 打开 Recent
2. 进入 batch mode
3. 批量加标签 / 颜色标签
4. 退出 batch mode
5. 通过 Tags / Collections / subset 再找到对应内容

### 路径 B：Media 主路径

1. 进入 Media
2. 浏览 / 筛选
3. 查看 details
4. 做整理动作
5. 通过 recent / collections 再找回
6. 若样本中有视频，观察单帧缩略图与 details 内 6 帧循环预览是否降级自然

### 路径 C：Games 主路径

1. 进入 Games
2. 浏览
3. 打开游戏条目
4. 查看 details
5. 调整 status
6. 再按 status / tag 找回

### 路径 D：Books / Software 主路径

1. 进入 Books 或 Software
2. 浏览并确认条目语义
3. 查看 details
4. 写入 favorite / rating / tag / collection
5. 通过 recent / collections 再找回

## 6. 单次验证应记录什么

每次外部验证建议固定记录这些观察项：

### 理解类

- 用户第一句话怎么描述这个产品
- 用户有没有理解错定位

### 路径类

- 用户在哪一步停顿最久
- 用户在哪一步最自然
- 用户有没有走错入口
- 用户是否能快速理解三栏结构和 details 补位逻辑

### 组织层类

- 用户是否主动使用 tags / collections / favorite / rating
- 哪些组织动作最难理解
- 哪些组织动作最有价值感

### 垂类差异类

- 哪个 subset surface 最强
- 哪个最弱
- 哪个最不像独立入口

### 预期偏差类

- 用户最常期待哪些当前没有的能力
- 这些期待是功能缺失，还是表达误导

### 当前 UI 一致性类

- 导航展开 / 收起是否容易发现和理解
- details toggle 是否足够自然
- backend 状态提示是否克制而清楚
- 当前界面是否更像统一工作台，而不是松散页面集合

### 发布包与运行环境类

- 一体化 beta 包是否能自动启动 backend
- backend 未 ready 或启动失败时提示是否可理解
- Electron beta 包是否能启动到工作台
- 无 saved source 时是否进入 Onboarding
- 有 saved source 时是否进入 Home
- open file / open containing folder 是否在桌面壳中可用
- 不配置系统 `PATH` 中的 `ffmpeg` 时，随包 FFmpeg 是否仍可支持视频缩略图
- 异常视频时，视频预览是否仅降级而不影响主路径

## 8. 反馈优先级规则

### P0：阻断型问题

- 用户无法完成主路径
- 页面让用户误判产品是什么
- 核心功能无法理解或无法操作

### P1：高价值收口问题

- 主路径可走但明显卡顿
- subset surface 表达不够成立
- 组织层能力存在但不自然

### P2：后置优化问题

- 更深垂类语义
- 更强视觉表达
- 更顺滑细节体验

### P3：暂不跟进问题

- 明显超出当前测试版边界
- 平台化要求
- AI / 推荐 / 阅读器 / 安装管理器等方向

## 9. 当前完成标志

本阶段完成的标志不是“再做一个页面”，而是：

- 已有固定主路径可用于外部验证
- 已有固定问题清单
- 已有统一反馈模板
- 已有优先级分类方法
- 已准备好做真实用户测试

## 10. 当前结论

测试版验证准备阶段的核心是：

> 把“能跑的测试版”变成“能被验证的测试版”。

因此，本阶段一切工作都应服务于：

- 提高验证质量
- 提高反馈可回收性
- 防止验证结果把项目重新带偏

最近的 UI/UX 收口也应服务于这一点：帮助外部验证更聚焦在主链和边界，而不是被不一致的壳层表达分散注意力。
