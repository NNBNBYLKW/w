# 正式版手工验收步骤（最终版）

## 1. 文档目的

本文档用于给当前正式版 v1 提供一套**固定、可重复、可交接**的手工验收路径。

它的目标不是覆盖所有未来能力，而是确保当前已冻结的 v1 边界可以被稳定验证：

- 主链是否成立
- 主要页面是否可用
- 组织层与再找回链路是否成立
- image-first enrichment 是否成立
- source/runtime hardening 是否成立
- shared details panel 与 desktop actions 是否成立

本文档应作为：

> **正式版 v1 release freeze 前的最终手工验收清单。**

它描述的是当前 documented validation steps 中的手工验证部分，不代表自动化 release gate 或 packaging pipeline。

---

## 2. 使用前提

执行本手工验收前，默认满足以下条件：

### 2.1 运行环境
- backend 可正常启动
- frontend 可正常启动
- desktop shell 可正常启动（若本轮需要验证桌面动作）

### 2.2 数据准备
建议准备至少一个本地 source，且 source 下包含以下类型文件：

- 至少 2 张 image 文件
- 至少 1 个 video 文件
- 至少 1 个 document 或其他普通文件
- 至少 1 组多层目录，便于验证 exact-directory browse

### 2.3 推荐样例准备
为了更容易验证，建议准备：

- 一张尺寸明确的 image（例如 1920×1080）
- 一张不同尺寸的 image（例如 800×800）
- 一个后续可修改或替换的 image 文件，用于验证 rescan 后 metadata/thumbnail 行为
- 一个可打 tag 和 color tag 的样本文件

---

## 3. 验收范围

本手工验收只覆盖当前正式版 v1 边界内的能力：

- source onboarding
- scan / delete-sync
- Search
- Files
- Media Library
- Recent Imports
- Tags
- Collections
- shared DetailsPanel
- normal tags / color tags
- metadata baseline
- image thumbnail / preview
- desktop open actions
- source runtime hardening

不覆盖：

- 完整任务平台
- richer media preview
- advanced query builder
- smart rules
- 垂类库（游戏/书籍/软件）
- AI / semantic / OCR

---

## 4. 验收通过标准

只有当以下三层都成立时，才建议判定正式版手工验收通过：

### 4.1 主链正确性通过
- source -> scan -> indexed retrieval -> details -> open action 主链成立

### 4.2 组织层闭环通过
- tag / color tag / Search / Files / Tags / Collections 形成再找回闭环

### 4.3 体验层最低标准通过
- 页面不会明显互相打架
- 局部状态表达清楚
- 关键页面没有“看起来像占位页”的断裂感

---

## 5. 验收步骤（最终版）

## Step 1：启动系统

### 操作
1. 启动 backend。
2. 启动 frontend。
3. 启动 desktop shell（若本轮需要验证桌面动作）。

### 期望结果
- backend 正常启动
- frontend 可访问
- desktop shell 可打开主界面
- 无明显启动级报错阻断主流程

---

## Step 2：进入 Settings，添加 source

### 操作
1. 打开 `/settings`
2. 进入 source management
3. 添加一个有效本地 source 路径

### 期望结果
- source 创建成功
- source 出现在 source 列表中
- source 基本信息可见
- 没有出现错误的 overlap/root 校验阻断

### 记录项
- source create guidance 与路径选择方式是否清楚
- source 创建后的默认状态是否清楚

---

## Step 3：首次执行 scan

### 操作
1. 对刚创建的 source 点击 `Run scan`
2. 等待 scan 完成

### 期望结果
- scan 成功
- source row/card 中能看到最近 scan 状态
- `last_scan_error_message` 为空
- 系统中出现 indexed files

### 补充检查
- 当前 source 的 scan 按钮在请求期间应局部禁用
- 不应误伤其他 source 的可操作性

---

## Step 4：验证 Search 基本检索链

### 操作
1. 打开 `/search`
2. 不输入 query，观察默认结果
3. 输入一个文件名片段并执行搜索
4. 再输入一个路径片段并执行搜索

### 期望结果
- Search 能返回 active indexed files
- 文件名片段可命中结果
- 路径片段可命中结果
- 分页与排序行为正常

---

## Step 5：验证 shared DetailsPanel 基础详情

### 操作
1. 在 Search 结果中点击一个文件
2. 打开/观察右侧共享 details panel

### 期望结果
- details panel 正常显示当前文件详情
- 文件基础字段存在
- source / path / file_type / modified 等信息正常
- 没有出现 page-local detail 替代 shared details 的情况

---

## Step 6：验证 metadata baseline（image）

### 操作
1. 在 Search 或 Files 中选中一张 image
2. 查看 details panel 中的 `Metadata` 区块

### 期望结果
- image 文件显示 `Width` / `Height`
- 值与原图实际尺寸一致或合理一致
- 非 image 文件应显示正常 empty/fallback，而不是报错

### 补充检查
1. 再选中一个非 image 文件
2. 确认 metadata 区块不会让 details panel 进入错误态

---

## Step 7：验证 image thumbnail / preview

### 操作
1. 打开 `/library/media`
2. 观察 image 卡片
3. 点击 image 文件，查看 details panel 中的 `Preview` 区块

### 期望结果
- image 卡片显示真实 thumbnail
- video 卡片仍使用当前占位样式
- image 文件在 details panel 中可显示 preview
- 非 image 文件不会错误显示 image preview

---

## Step 8：验证 normal tag 基本闭环

### 操作
1. 选中一个文件
2. 在 details panel 中添加一个普通 tag
3. 再次确认该 tag 已附着在文件上

### 期望结果
- tag 创建/复用正常
- tag attach 正常
- details panel 中能立即看到该 tag
- 不会出现重复 tag row

---

## Step 9：验证 color tag 基本闭环

### 操作
1. 对同一个文件设置一个 color tag
2. 再清除 color tag
3. 再重新设置一个 color tag

### 期望结果
- color tag 设置成功
- 清除成功
- 重新设置成功
- details panel 中当前 color tag 状态清楚

---

## Step 10：验证 Tags 页面找回

### 操作
1. 打开 `/tags`
2. 选择刚才使用过的普通 tag
3. 查看结果列表

### 期望结果
- Tags 页面能按该 tag 找回文件
- 点击结果行后 shared details panel 正常更新
- Tags 页面仍然是 retrieval 入口，而不是管理中心

---

## Step 11：验证 Search 的 tag/color retrieval loop

### 操作
1. 回到 `/search`
2. 应用刚才的普通 tag filter
3. 再叠加 color tag filter
4. 如有必要，再叠加 `file_type`

### 期望结果
- 过滤语义为纯 `AND`
- 结果进一步缩小
- 不出现重复 row
- 翻页/排序仍正常
- 过滤变化后 page 会回到 `1`

---

## Step 12：验证 Files 的 source/path browse + tag/color 过滤

### 操作
1. 打开 `/files`
2. 选择一个 source
3. 浏览到一个 exact directory
4. 再应用 tag filter
5. 再应用 color tag filter

### 期望结果
- Files 页面仍然是 flat indexed-files browse
- source + exact-directory browse 语义成立
- 再叠加 tag/color 后结果继续按纯 `AND` 缩小
- 变化后 page 回到 `1`
- 点击结果行继续驱动 shared details panel

---

## Step 13：验证 Recent Imports

### 操作
1. 打开 `/recent`
2. 切换 `1d / 7d / 30d`
3. 切换排序方向

### 期望结果
- Recent 列表正常工作
- 不出现与 Search / Files 混淆的过滤逻辑
- 仍然是“recently indexed files”视图，而不是综合 dashboard

---

## Step 14：验证 Collections 基本闭环

### 操作
1. 打开 `/collections`
2. 创建一个 collection，例如：
   - `name = Blue Images`
   - `file_type = image`
   - `color_tag = blue`
3. 确认 collection 出现在列表中
4. 观察结果列表
5. 再创建一个带 `tag_id` 或 `source_id + parent_path` 的 collection

### 期望结果
- collection 创建成功
- 新 collection 自动选中
- 结果按保存条件实时查询
- 结果语义仍是纯 `AND`
- 点击结果行仍由 shared details panel 消费

### 补充检查
1. 删除当前 collection
2. 观察列表与默认选中行为

### 期望结果
- 删除成功
- 若仍有其他 collection，自动选中新的第一项
- 若没有 collection，页面进入正常空态
- shared details panel 不因 collection 删除被异常清空

---

## Step 15：验证桌面打开动作

### 操作
1. 在任意检索面选中一个真实存在的文件
2. 在 details panel 中点击：
   - `Open file`
   - `Open containing folder`

### 期望结果
- `Open file` 能调用系统默认程序打开该文件
- `Open containing folder` 能打开其所在目录
- details panel 本身不被刷新/清空
- 若路径无效，应只出现局部失败提示

---

## Step 16：验证 scan/runtime hardening

### 操作 A：失败反馈
1. 让某个 source 路径进入可控失败状态
2. 再次执行 scan

### 期望结果
- source row 显示失败状态
- `last_scan_error_message` 可见
- Home 的 source overview 最多只显示轻量失败提示，不变成 runtime center

### 操作 B：成功重扫清空失败信息
1. 恢复 source 路径
2. 再次执行 scan

### 期望结果
- source 最近 scan 成功
- `last_scan_error_message` 变回空

### 操作 C：同 source 冲突保护
1. 在同一个 source 上制造重复触发场景
2. 观察第二次请求行为

### 期望结果
- backend 拒绝 same-source active scan
- 错误语义为 `409 SCAN_ALREADY_RUNNING`
- 前端在 source surface 内做局部提示
- 不影响其他页面

---

## Step 17：验证主要主页面与主入口的 release-facing 一致性

### 操作
1. 打开 `/onboarding`
2. 查看 Onboarding 的页面标题与说明
3. 打开 `/`
4. 查看 Home 的 overview、quick links
5. 打开 sidebar
6. 打开 `/settings`
7. 再打开 `/search`、`/files`、`/library/media`、`/recent`、`/tags`、`/collections`

### 期望结果
- Onboarding 明确是开始 source setup 的入口，而不是阶段占位页
- Home 明确是 overview / entry，不是 dashboard center
- Settings 明确是 source/system entry，不是 preferences center
- Sidebar 中各主入口清楚可用
- Collections 已作为正式入口存在
- 主要主页面与主入口的 release-facing wording 保持一致
- 不存在明显“Phase 0”或旧阶段残留文案

## 7. 通过判定规则

### 可判定为“手工验收通过”的条件
- 所有主链步骤通过
- 没有发现新的 v1 blocker
- 即使存在已知问题，也都属于非阻塞问题或范围外事项

### 应判定为“不通过”的条件
- source -> scan -> retrieval 主链断裂
- Search / Files / Details 主检索链断裂
- image metadata / thumbnail / preview 出现明显回退
- Collections 无法创建或结果语义明显错误
- desktop open actions 在桌面壳中失效
- runtime hardening 语义明显失真

---

## 8. 最终说明

这份文档的意义，不是把当前版本包装成“什么都做完了”，而是：

> **给当前正式版 v1 一条固定、可靠、可重复执行的验收主链。**

只要这条主链能够稳定通过，就说明当前版本已经具备被正式冻结为 v1 的基础。
