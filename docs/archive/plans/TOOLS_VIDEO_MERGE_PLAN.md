# Workbench Tools 模块与视频合并工具方案

## 1. 背景

当前 Workbench 已经具备本地资产索引、分类、搜索、文档/媒体/软件/游戏等 smart view 能力。用户在处理本地视频时，经常需要使用 FFmpeg 或传统 `.bat` 脚本完成合并、转码等操作。

传统脚本的主要问题是：

- 需要手动把文件改名为 `1.mp4 / 2.mp4 / 3.mp4` 等固定格式。
- 需要手动修改脚本里的输入文件名和输出文件名。
- 需要把文件移动到特定目录。
- 执行失败后日志和原因不容易追踪。
- 执行结果无法和 Workbench 当前资产工作流衔接。

因此，本方案计划新增一个 `Tools / 工具` 模块，第一版只实现一个内置工具：

- 视频合并 / Video Merge

目标是把常用 FFmpeg 操作从“脚本适配用户”改成“工具表单适配用户输入”。

---

## 2. 产品定位

`Tools / 工具` 模块的定位是：

> 对已索引文件或本地文件执行受控、本地、可追踪的批处理动作。

它不是：

- 任意脚本执行器。
- BAT 脚本管理器。
- 自动化平台。
- 视频剪辑软件。
- FFmpeg GUI 全功能封装。
- Windows 系统管理工具集合。
- 插件平台。

第一版只做一个内置工具：

- 视频合并

后续可以扩展更多工具，但第一版必须保持克制。

---

## 3. 第一版范围

### 3.1 本轮要做

第一版实现：

- 左侧导航新增 `工具 / Tools`。
- 工具页面新增内置工具：`视频合并 / Video Merge`。
- 支持从已索引文件选择视频。
- 支持从 Workbench 内部文件列表拖入视频。
- 支持从 Windows Explorer 外部拖入视频。
- 输入列表支持拖动排序。
- 输入列表支持上移、下移、删除、清空。
- 只接受视频文件。
- 用户可以手动输入最终输出文件名。
- 用户可以选择或输入输出目录。
- 支持两种合并模式：
  - 快速合并：stream copy，不重新编码。
  - 兼容合并：re-encode，重新编码。
- 后端通过后台线程执行任务。
- 前端轮询运行状态。
- 后端记录 tool run。
- 显示运行状态、日志、输出路径。
- 成功后提示用户重新扫描来源以加入索引。
- 不自动把输出文件加入索引。
- 不允许覆盖已有输出文件，采用自动改名或提示。

### 3.2 本轮不做

第一版不做：

- 任意 `.bat` 注册与执行。
- 任意命令输入。
- PowerShell 脚本执行。
- 批量转码工具。
- 视频压缩工具。
- 视频剪辑 / 裁剪 / 加字幕。
- 自动重新索引输出文件。
- 工具插件市场。
- 复杂任务编排。
- 定时任务。
- 多工具流水线。
- 删除 / 移动 / 覆盖用户原始文件的危险操作。
- 联网服务或云处理。

## 7. 视频合并工具 UI

### 7.1 输入文件区

输入文件区包含：

- 拖拽区域
- 已添加文件列表
- 清空按钮
- 文件数量提示

文案示例：

- 将视频文件拖入此处
- 或从媒体/文件页面选择视频加入
- 当前工具只支持视频文件

### 7.2 输入列表

每一行显示：

| 列 | 内容 |
|---|---|
| 拖拽手柄 | 用于拖动排序 |
| 序号 | 当前合并顺序 |
| 文件名 | 视频文件名 |
| 时长 | 如果可用则显示 |
| 大小 | 文件大小 |
| 来源 | 已索引 / 外部文件 |
| 操作 | 上移、下移、删除 |

支持操作：

- 拖动排序
- 上移
- 下移
- 删除
- 清空全部

排序后的列表顺序就是 FFmpeg 合并顺序。

### 7.3 输出设置

输出设置包含：

| 字段 | 说明 |
|---|---|
| 输出文件名 | 用户手动输入最终成品文件名 |
| 输出目录 | 默认第一个输入视频所在目录，可修改 |
| 合并模式 | 快速合并 / 兼容合并 |

输出文件名规则：

- 不能为空。
- 自动去除首尾空格。
- 不允许 Windows 非法字符。
- 如果没有扩展名，自动补 `.mp4`。
- 不允许覆盖已有文件。
- 如果目标已存在，自动生成安全新文件名，或提示用户。

建议第一版采用：

- 自动生成安全新文件名。

示例：

- `final_video.mp4` 已存在
- 自动改为 `final_video_1.mp4`
- 如果仍存在，则 `final_video_2.mp4`

### 7.4 合并模式

#### 快速合并

用户文案：

- 快速合并
- 不重新编码，速度快，适合同一来源、参数一致的视频片段。

技术方式：

- FFmpeg concat demuxer
- `-c copy`

优点：

- 快
- 不重新编码
- 画质无损

缺点：

- 对输入视频参数一致性要求高。
- 不同编码、分辨率、帧率或音频参数的视频可能失败。

失败提示：

- 快速合并失败。可能是视频编码、分辨率、帧率或音频参数不一致。可以尝试兼容合并。

#### 兼容合并

用户文案：

- 兼容合并
- 重新编码，兼容性更高，但速度较慢。

技术方式：

- FFmpeg concat demuxer
- 视频编码：H.264 / libx264
- 音频编码：AAC

优点：

- 兼容性强。
- 更适合不同来源的视频。

缺点：

- 速度较慢。
- 会重新编码。
- 可能有轻微画质损失。

## 8. 输入文件来源

第一版支持三种输入来源。

### 8.1 A：从已索引文件选择

来源：

- 文件页面
- 媒体页面
- 搜索结果
- 其它已有文件列表

用户可以选择视频文件并发送到：

- 工具 > 视频合并

后端输入表示：

| 字段 | 值 |
|---|---|
| source_kind | indexed_file |
| file_id | 对应文件 ID |

特点：

- 最稳定。
- 后端可以通过 file_id 查真实路径。
- 可以复用现有 metadata。

### 8.2 B：从 Workbench 内部拖入

用户从 Workbench 的文件列表、媒体列表中把视频拖到 Tools 的视频合并区域。

本质仍然是已索引文件。

后端输入表示同上：

| 字段 | 值 |
|---|---|
| source_kind | indexed_file |
| file_id | 对应文件 ID |

### 8.3 C：从 Windows Explorer 外部拖入

用户从 Windows Explorer 把视频文件拖入 Tools 区域。

后端输入表示：

| 字段 | 值 |
|---|---|
| source_kind | external_path |
| path | 本地绝对路径 |

注意：

- 外部拖入依赖 Electron 桌面环境能力。
- 普通浏览器环境可能无法拿到真实本地绝对路径。
- 如果当前运行环境不支持外部路径拖入，应显示明确提示。
- 外部路径必须由后端再次验证。

提示示例：

- 当前运行环境不支持外部路径拖入，请从已索引文件中选择视频。
- 当前工具只支持视频文件。

## 10. 后端执行模型

### 10.1 执行方式

后端通过后台线程执行视频合并任务。

流程：

1. 前端提交 run request。
2. 后端创建 tool_run 记录。
3. 后端启动后台线程执行 FFmpeg。
4. API 立即返回 run_id。
5. 前端轮询 run status。
6. 任务完成后状态变为 succeeded 或 failed。
7. 前端显示输出路径或错误信息。

### 10.2 不使用长请求等待

不应让 HTTP 请求一直挂起等待 FFmpeg 完成。

原因：

- 视频合并可能耗时很长。
- 长请求容易超时。
- 用户需要看到状态和日志。
- 后续可以扩展取消、历史记录和重试。

建议新增或扩展以下结构：

| 文件/模块 | 责任 |
|---|---|
| app/api/routes/tools.py | Tools API router |
| app/services/tools/service.py | ToolsService，统一入口 |
| app/services/tools/video_merge.py | VideoMergeToolService |
| app/services/tools/runner.py | Subprocess runner |
| app/db/models/tool_run.py | tool_runs 数据模型 |
| app/repositories/tools.py | tool run 查询与更新 |
| app/schemas/tools.py | request/response schemas |

保持现有架构原则：

- Router 只接收参数、调用 service、返回 response。
- Service 负责业务规则和事务边界。
- Repository 只做 query/add/update/flush。
- 不在 router 写 FFmpeg 或文件系统业务逻辑。

## 13. API 设计草案

### 13.1 获取工具列表

Endpoint:

- GET /tools

返回内容：

- key
- title
- description
- category

第一版只返回 video_merge。

### 13.2 创建视频合并任务

Endpoint:

- POST /tools/video-merge/runs

请求字段：

| 字段 | 说明 |
|---|---|
| inputs | 输入视频数组，顺序即合并顺序 |
| output_name | 输出文件名 |
| output_dir | 输出目录 |
| mode | copy 或 reencode |

inputs 项字段：

| 字段 | 说明 |
|---|---|
| source_kind | indexed_file 或 external_path |
| file_id | indexed_file 时使用 |
| path | external_path 时使用 |

mode：

| 值 | 说明 |
|---|---|
| copy | 快速合并 |
| reencode | 兼容合并 |

响应：

| 字段 | 说明 |
|---|---|
| run_id | 任务 ID |
| status | 初始状态 |

### 13.3 查询任务详情

Endpoint:

- GET /tools/runs/{run_id}

返回：

| 字段 | 说明 |
|---|---|
| id | run id |
| tool_key | 工具标识 |
| status | 当前状态 |
| input | 输入快照 |
| output_path | 成功输出路径 |
| log_tail | 日志尾部 |
| error_message | 错误信息 |
| started_at | 开始时间 |
| finished_at | 结束时间 |

### 13.4 查询运行历史

Endpoint:

- GET /tools/runs

支持分页：

- page
- page_size

第一版可按 created_at desc 排序。

### 13.5 取消任务

第一版暂不实现取消。

可以不提供 API。  
如预留 API，应明确返回当前不支持。

## 14. FFmpeg 命令生成

### 14.1 concat list

后端创建临时工作目录：

- data/tool_runs/{run_id}/

内部生成：

- concat_list.txt
- log.txt

concat list 内容示例：

- file 'G:/Videos/clip01.mp4'
- file 'G:/Videos/clip02.mp4'
- file 'G:/Videos/clip03.mp4'

路径处理要求：

- Windows 反斜杠可转换为正斜杠。
- 单引号需要转义。
- 不复制用户原文件。
- 不要求用户改名。
- 不修改原始输入文件。

### 14.2 快速合并命令

逻辑：

- ffmpeg
- -f concat
- -safe 0
- -i concat_list.txt
- -c copy
- output_path

必须使用 argv list，不能拼接 shell string。

### 14.3 兼容合并命令

逻辑：

- ffmpeg
- -f concat
- -safe 0
- -i concat_list.txt
- -c:v libx264
- -c:a aac
- output_path

可加入常见安全参数，但第一版不要复杂化。

## 15. Subprocess 安全要求

硬性要求：

- 使用 argv list。
- shell=False。
- 不允许任意命令输入。
- 不允许用户编辑命令模板。
- 不允许执行用户提供的 bat。
- 输入路径必须存在。
- 输入文件扩展名必须是允许的视频格式。
- 输出目录必须存在且可写。
- 输出文件名必须通过 Windows 文件名校验。
- 不覆盖已有文件。
- stdout/stderr 使用 bytes capture。
- 解码时使用 UTF-8 errors=replace，避免 Windows GBK 解码错误。
- 日志长度限制，避免数据库或 UI 被大量日志撑爆。

---

## 16. 文件名与路径校验

### 16.1 输出文件名校验

不允许：

- 空字符串
- 全空格
- 包含 `\ / : * ? " < > |`
- 以点号结尾
- Windows 保留名

Windows 保留名包括：

- CON
- PRN
- AUX
- NUL
- COM1 到 COM9
- LPT1 到 LPT9

如果用户没有写扩展名：

- 自动补 `.mp4`

### 16.2 输出目录校验

必须满足：

- 路径存在。
- 是目录。
- 可写。
- 不为空。
- 不指向文件。

### 16.3 输出文件冲突处理

不允许覆盖已有文件。

默认处理：

- 如果 `output.mp4` 已存在，则生成 `output_1.mp4`
- 如果仍存在，则 `output_2.mp4`
- 依次递增

UI 应显示最终实际输出路径。

---

## 17. 外部拖入安全边界

外部路径输入需要后端再次校验：

- path 是否存在。
- path 是否是文件。
- 扩展名是否是允许的视频格式。
- 文件是否可读。
- 不信任前端传入的 displayName、size、duration。

如果验证失败，任务不应启动。

错误提示：

- 输入文件不存在。
- 当前工具只支持视频文件。
- 无法读取该文件。

## 20. 轮询机制

创建任务后：

1. 保存 run_id。
2. 每隔固定时间轮询 GET /tools/runs/{run_id}。
3. 状态为 succeeded / failed 后停止轮询。
4. 显示结果或错误。

建议轮询间隔：

- 1 秒到 2 秒。

如果页面离开：

- 停止当前轮询。
- run 仍在后端继续执行。
- 再进入 Tools 页面可从历史记录查看状态。

## 21. 成功后的用户提示

成功后显示：

- 合并完成。
- 输出文件路径。
- 提示：如需在 Workbench 中看到该文件，请重新扫描对应来源。

可选按钮：

- 复制路径
- 在资源管理器中显示

如果当前项目已有打开资源管理器能力，可以复用。  
如果没有，第一版只做复制路径或纯显示路径即可。

## 22. 错误提示设计

常见错误与提示：

| 错误 | 用户提示 |
|---|---|
| FFmpeg 不存在 | 未找到 FFmpeg，请检查 FFmpeg 是否已配置。 |
| 输入文件不存在 | 输入文件不存在或已被移动。 |
| 非视频文件 | 当前工具只支持视频文件。 |
| 输出文件名非法 | 输出文件名包含 Windows 不允许的字符。 |
| 输出目录不可写 | 当前输出目录不可写，请选择其它目录。 |
| 快速合并失败 | 快速合并失败，可能是视频参数不一致，请尝试兼容合并。 |
| 兼容合并失败 | 兼容合并失败，请查看日志。 |
| 权限不足 | 没有权限读取输入文件或写入输出目录。 |

## 26. 风险点

### 26.1 外部拖入路径能力

Windows Explorer 外部拖入依赖 Electron 桌面能力。  
普通 Web 环境可能无法获得绝对路径。

处理方式：

- Desktop 环境支持。
- 不支持时显示明确提示。
- 不为了外部拖入破坏 Web 安全边界。

### 26.2 FFmpeg 可用性

如果 FFmpeg 未配置，工具无法执行。

处理方式：

- 后端检测 ffmpeg 是否可用。
- 不可用时返回明确错误。
- UI 显示“未找到 FFmpeg”。

### 26.3 视频参数不一致

快速合并可能失败。

处理方式：

- 提供兼容合并模式。
- 不自动悄悄重新编码。
- 让用户明确选择。

### 26.4 长任务状态管理

视频合并可能耗时较长。

处理方式：

- 后端后台线程执行。
- 前端轮询。
- 不使用长 HTTP 请求。

### 26.5 工具模块泛化过度

如果第一版就做任意脚本，会扩大风险。

处理方式：

- 第一版只做内置视频合并。
- 结构预留扩展，不开放任意脚本执行。
