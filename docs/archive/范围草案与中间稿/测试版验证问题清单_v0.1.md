# 测试版验证问题清单 v0.1

## 1. 文档目的

本文档用于配合《测试版范围说明与验收草案 v0.1》，定义测试版对外验证时的核心问题清单、访谈口径、观察重点与反馈归类方式。

本文件不是功能范围文档，也不是缺陷清单。  
它的作用是：

- 明确 **测试版到底要验证什么**
- 避免验证时只收集“喜不喜欢”这类低价值反馈
- 帮助区分 **产品方向问题 / 交互问题 / 稳定性问题 / 性能问题 / 偏航需求**
- 为后续“稳定性 / 性能 / 正式版收口”阶段提供优先级输入

---

## 2. 验证原则

测试版验证的重点不是证明“产品已经完整”，而是验证以下几件事：

1. 用户是否真的理解这是一个 **本地资产工作台**
2. 用户是否会自然使用 **垂类库入口**，而不是只停留在 Search / Files
3. 用户是否会自然使用 **标签 / 颜色标签 / Collections / Recent** 作为组织与回流入口
4. `shared details + open actions` 这条复用链是否真的顺手
5. 当前方法论是否成立：

> `files + subset surfaces + shared details + tags/color tags/collections`

验证时应尽量避免两种失真：

- 只看“能不能点通”，不看用户是否真的愿意长期用
- 被少量高噪音需求带偏，把测试版拉向阅读平台、启动器平台、AI 平台或安装管理器方向

---

## 3. 建议的测试对象分层

建议至少覆盖三类用户，而不是只找一种使用习惯的人：

### 3.1 素材重度用户
典型特征：
- 图片 / 视频 / 参考资料很多
- 对缩略图、筛选、快速再找回敏感
- 更容易验证媒体库价值

### 3.2 收藏 / 库式管理用户
典型特征：
- 有大量游戏、电子书、便携软件、本地资源包
- 更容易验证游戏库 / 书库 / 软件库入口价值

### 3.3 效率型文件重度用户
典型特征：
- 更关注找回效率、组织能力、路径回流
- 更容易暴露 Search / Files / Tags / Recent 的实际可用性

建议不要只找“技术熟悉者”测试，否则容易低估首次理解门槛和入口设计问题。

---

## 4. 测试方式建议

建议采用 **任务驱动 + 过程观察 + 结束访谈** 的组合，而不是只给一个试玩链接后收一句主观评价。

### 4.1 推荐结构

#### 第一步：基础背景确认
先确认：
- 用户平时主要管理什么类型内容
- 他们现在主要依赖什么工具或方式找回本地内容
- 他们是否已经有标签化 / 收藏化 / 集合化习惯

#### 第二步：给定任务
让用户实际完成几条关键路径，而不是自由乱逛。

#### 第三步：过程观察
重点记录：
- 他先点哪里
- 他是否自然理解页面语义
- 他在哪里停顿或迷失
- 他是否自然回到 details / tags / recent / collections

#### 第四步：结束访谈
再问对定位、库页价值、组织层价值、继续使用意愿的判断。

---

## 5. 建议的核心验证任务

## 5.1 任务 A：首次理解与入口判断
### 目标
验证用户第一次进入时，是否能理解这个产品在做什么。

### 建议任务
让用户打开首页，不做任何解释，观察：
- 他如何描述这个产品
- 他第一反应会点哪里
- 他是否把它理解成“另一个资源管理器”
- 他是否能看出这是围绕本地内容组织与浏览的工作台

### 关键观察点
- 首页入口是否清楚
- 各库页命名是否容易理解
- `Recent / Tags / Collections` 是否被看懂

---

## 5.2 任务 B：搜索找回
### 目标
验证 Search 是否真的比目录记忆更高效。

### 建议任务
给用户一个目标内容，让他通过 Search 找到并打开。

### 关键观察点
- 用户是否能自然进入 Search
- 是否会主动使用筛选条件
- 是否会进入 details 确认再打开
- 是否出现“我还是宁愿自己记目录”的反应

### 重点问题
- Search 的结果确认成本高不高
- details 是否真的承担了确认中心角色

---

## 5.3 任务 C：媒体库找素材
### 目标
验证媒体库是否已经具备明显差异化价值。

### 建议任务
让用户从媒体库里找到某张图或某段视频，并进行一次标记。

### 关键观察点
- 用户是否主要依赖缩略图 / 卡片而不是文件名
- 标签 / 颜色标签 / 类型筛选是否被自然使用
- details 与预览是否足够支撑确认动作
- 找到后是否会顺手做整理动作

### 重点问题
- 媒体库是否已经摆脱“文件页轻包装”的感知
- 筛选入口是否足够高频、清楚、稳定

---

## 5.4 任务 D：游戏库 / 书库 / 软件库使用
### 目标
验证垂类入口是否有独立价值，而不是命名不同的同类列表。

### 建议任务
任选一个库：
- 进入库页
- 找到一个目标条目
- 查看详情
- 执行打开 / 启动 / 外部打开动作
- 再通过标签、集合、Search 或 Recent 找回它

### 关键观察点
- 用户是否感知到“这是一个库入口”
- 用户是否仍然把它当作纯扩展名列表
- 各库页是否共享统一交互语言
- 是否出现对平台化能力的强烈期待

### 重点问题
- 哪个库最有价值
- 哪个库最薄、最像占位页

---

## 5.5 任务 E：最近链路整理
### 目标
验证 Recent 是否只是附属列表，还是一个真实整理入口。

### 建议任务
让用户进入 Recent，对最近进入系统的一批内容做轻量整理。

### 关键观察点
- 用户是否理解 Recent 的角色
- 用户是否愿意在这里连续做标记动作
- 标记后是否会自然通过 Tags / Collections / 库页再次找回
- Recent 是否具有“回流入口”价值

---

## 6. 访谈核心问题清单

以下问题建议按模块询问，而不是一次性全问完。

## 6.1 定位理解
1. 你会怎么向别人描述这个产品？
2. 你更觉得它像资源管理器增强，还是本地内容/资产工作台？
3. 你会不会愿意把它当成你本地内容入口之一？为什么？

## 6.2 首页与入口
4. 第一次进入时，你最先想点哪个入口？为什么？
5. 哪些页面名称一眼能懂，哪些不够清楚？
6. 首页是否帮助你理解“从哪里开始用”？

## 6.3 Search 与 Files
7. 你觉得 Search 找内容快不快？
8. 你在什么时候更愿意去 Files，而不是 Search？
9. Search 的筛选、排序、结果确认是否顺手？

## 6.4 媒体库
10. 媒体库有没有让你更容易找图 / 找视频？
11. 你主要依靠什么找媒体：缩略图、标签、文件名还是路径？
12. 媒体库中你最想优先补强的是什么？

## 6.5 游戏库 / 书库 / 软件库
13. 这三个库里，你最可能长期用哪个？
14. 哪个库目前仍然最薄？
15. 你是否会期待更深的平台化能力？
16. 这些期待对你来说是“必须”，还是“以后有更好”？

## 6.6 组织层
17. 你是否真的会使用标签？
18. 颜色标签对你有没有实际帮助？
19. 你会不会使用 Collections / Recent 作为再找回入口？
20. 你会不会愿意做连续整理，而不是只在找不到时偶尔使用？

## 6.7 Details 与动作
21. 你能否自然理解右侧 details 是统一整理中心？
22. 打开文件 / 打开目录 / 外部打开这些动作是否顺手？
23. details 里哪些信息是必要的，哪些是噪音？

## 6.8 继续使用意愿
24. 如果今天开始用，你最可能用它做什么？
25. 哪一个问题会阻止你继续用？
26. 如果正式版只能优先补 2~3 项能力，你最希望补什么？

## 7. 观察重点清单

测试时不要只记用户说了什么，还要记用户做了什么。

建议重点记录以下行为信号：

### 7.1 入口选择信号
- 是否第一时间进入 Search
- 是否会主动进入某个垂类库
- 是否能看懂 Recent / Tags / Collections

### 7.2 停顿与迷失信号
- 是否在首页停很久不知道点哪里
- 是否在 details、筛选器、排序入口前停顿
- 是否频繁退回上一级页面重新找路

### 7.3 组织层真实使用信号
- 是否真的打标签
- 是否会用颜色标签
- 是否会通过 Tags / Collections / Recent 再找回
- 是否愿意连续处理一批内容

### 7.4 价值判断信号
- 是否出现“这比我现在的方法快”
- 是否出现“这让我终于不用记目录了”
- 是否出现“这个库有点意思”
- 是否出现“我还是直接用原系统更快”

## 8. 反馈归类框架

为了避免测试后只得到一堆散乱意见，建议统一按以下类别归类：

## 8.1 路径性问题
定义：
- 用户不知道从哪里开始
- 用户无法自然完成一条关键路径
- 页面跳转与入口组织有问题

典型例子：
- 首页入口混乱
- Recent 不知道有什么用
- 不知道先看库页还是 Search

## 8.2 组织层问题
定义：
- 标签 / 颜色标签 / Collections / Recent 没有形成可感知价值
- 用户不会用，或用了也不回流

典型例子：
- 颜色标签语义不清楚
- Collections 太弱
- Tags 页看起来像附属页

## 8.3 库页价值问题
定义：
- 某个库页没有形成独立价值，只像换皮列表

典型例子：
- 书库看起来只是 pdf/epub 文件页
- 游戏库不像游戏库
- 软件库和 Files 差异太弱

## 8.4 Details / 动作问题
定义：
- details 不是统一整理中心
- 打开、确认、标记动作不顺

典型例子：
- details 信息层级乱
- 打开动作反馈差
- 详情和主列表联动不稳

## 8.5 稳定性问题
定义：
- 功能逻辑正确，但行为不稳定

典型例子：
- 打标签偶发失败
- 详情刷新异常
- 打开文件动作偶发失效

## 8.6 性能问题
定义：
- 用户感知到明显慢、卡、加载不稳

典型例子：
- 首屏太慢
- 缩略图加载拖沓
- 卡片流滚动卡顿

## 8.7 偏航需求
定义：
- 用户提出的需求会把产品拉向当前明确后置或不做的方向

典型例子：
- 阅读器化
- 启动器平台化
- 安装管理器化
- AI 自动标签 / OCR / 语义分析

偏航需求可以记录，但不应直接进入测试版后第一优先级。

## 9. 建议的验证记录模板

建议每位测试用户至少记录以下字段：

- 测试对象编号
- 用户类型（素材 / 收藏 / 效率）
- 当前主要使用系统/工具
- 首次理解描述
- 最常进入的页面
- 完成了哪些任务
- 卡点出现在什么地方
- 是否实际使用标签 / 颜色标签 / Collections / Recent
- 认为最有价值的 surface
- 认为最薄弱的 surface
- 是否愿意继续使用
- 希望优先补强的 2~3 项
- 问题归类（路径 / 组织层 / 库页价值 / details / 稳定性 / 性能 / 偏航）

---

## 10. 成功信号与危险信号

## 10.1 成功信号
如果测试中持续出现以下反馈，说明当前方法论大概率成立：

- 用户能自然理解这是本地资产工作台
- 用户开始依赖媒体库或其他垂类库找内容
- 用户会主动使用标签 / 颜色标签 / Recent / Collections
- 用户认为 details 的统一整理中心角色合理
- 用户愿意继续把它当作入口之一使用

## 10.2 危险信号
如果测试中持续出现以下信号，需要谨慎处理：

- 用户只把它当成“另一个文件列表工具”
- 用户始终不进入垂类库
- 用户始终不使用标签 / 颜色标签 / Collections / Recent
- 用户看不懂 details 的角色
- 用户高频卡在首页与入口组织上
- 用户对平台化能力的期待远高于当前库式入口价值

---

## 11. 测试后的输出要求

每轮测试后，建议至少产出以下三类输出：

### 11.1 问题归类清单
按前述分类整理：
- 路径性问题
- 组织层问题
- 库页价值问题
- details / 动作问题
- 稳定性问题
- 性能问题
- 偏航需求

### 11.2 优先级建议
把问题按：
- 测试版阻断
- 正式版高优先级
- 可后置优化
- 明确不做 / 暂不跟进
进行归并。

### 11.3 方向结论
最终要回答的不只是“有哪些 bug”，而是：

1. 当前产品定位是否被用户正确感知
2. 哪个垂类库最值得继续投入
3. 哪些组织层能力确实形成使用习惯
4. 正式版前应优先收口哪些高价值问题

本清单仍需进一步冻结以下事项：

4. 测试过程是否允许指导用户，还是尽量纯观察
5. 测试后优先级归并由谁拍板
