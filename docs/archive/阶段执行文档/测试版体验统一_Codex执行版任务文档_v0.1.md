# 测试版体验统一_Codex执行版任务文档_v0.1

## 1. 文档目的

本文档用于在“测试版范围冻结”之后，指导当前项目进入 **测试版体验统一** 阶段的实际执行。

本稿不是新增能力需求文档，而是一个 **收口型执行文档**。  
它的目标是：

- 统一当前已完成能力的体验表达
- 收口不同 subset surface 之间的交互差异
- 统一 header / loading / empty / no-result / CTA / details 层级
- 提高测试版对外演示时的整体一致性
- 为后续“测试版验证问题清单”和“外部验证”做准备

---

## 2. 当前阶段前提

当前项目已经完成：

### 2.1 垂类库线
- Media
- Games
- Books
- Software

### 2.2 组织层
- 批量标签 / 批量颜色标签
- favorite / rating 最小闭环
- collections 最小可编辑 saved retrieval
- recent family（imports / tagged / color-tagged）
- shared details 内最小再找回动作

### 2.3 当前阶段转换点
当前不再继续扩单点功能批次，而是进入：

> 范围冻结 → 体验统一 → 验证准备 → 对外验证

因此，本批次默认**不新增新的功能域**，而是收口已有能力的体验。

---

## 3. 本批次目标

本批次的目标是：

> 把当前已经完成的四条垂类库线与组织层能力，统一成一套“看起来属于同一个产品”的测试版体验。

这意味着本批次重点不是“还能不能再做一项功能”，而是：

- 不同页面是否说的是同一种产品语言
- 不同 subset surface 是否有一致的节奏
- shared details 是否有清楚统一的信息层级
- CTA 与回流路径是否对用户可预测
- batch mode 与普通浏览态是否一致易懂

---

## 4. 本批次明确不做

本批次明确不做以下内容：

- 不新增新的垂类库
- 不新增新的组织层能力
- 不新增 AI / 推荐 / OCR / 自动规则
- 不新增安装管理器 / 阅读器 / 游戏平台能力
- 不做测试版整合页以外的“新大页”
- 不重写架构
- 不引入对象中台
- 不做复杂动画或纯视觉重构
- 不做稳定性 / 性能专项优化（除非是体验统一所必需的最小修正）

---

## 5. 本批次核心任务

本批次建议拆成 5 个任务簇。

### 5.1 Header / Description 统一
所有 subset surface 的页面头部应统一以下方面：

- 标题层级
- 副标题长度与口径
- 对自身语义的表达方式
- 避免某页像产品页、某页像技术页、某页像占位页

目标页面包括：

- Media
- Games
- Books
- Software
- Recent
- Tags
- Collections

#### 统一原则
- 标题短
- 描述一句话说清当前页面是什么
- 避免暗示平台化能力
- 避免某些页面显得过强、某些页面显得过弱

---

### 5.2 loading / empty / no-result 统一
当前各页虽然大多已经有 loading / empty / no-result，但测试版阶段需要统一：

- 骨架态密度
- 空状态语义
- 无结果状态语义
- 建议动作文案
- 页面“未崩坏”的可感知性

#### 统一目标
- loading 看起来属于同一产品
- empty 表达“当前没有内容”
- no-result 表达“当前条件下没有结果”
- 不同页不要各说各话

---

### 5.3 CTA 与回流文案统一
当前系统已有大量 CTA：

- Open in Media / Books / Games / Software
- Find in Tags
- Filter in X
- Open matching X
- Save current filters as collection
- Back to X
- Open file / Open software file / Open game entry

本批次需要统一这些 CTA 的：

- 命名风格
- 粒度
- 是否用动词短语
- 是否会误导成平台能力
- 不同页面相同动作是否使用相同 wording

#### 重点
- 同义动作尽量统一
- context-specific wording 可以存在，但必须可解释
- 不要让某些 CTA 看起来像“高级功能”，另一些像“调试按钮”

---

### 5.4 shared details 信息层级统一
shared details 目前承接：

- tags / color tag
- favorite / rating
- game status
- book info
- software info
- open actions
- refind actions

本批次要统一 details 的信息顺序、分区命名、视觉主次和上下文差异处理。

#### 目标
- details 永远是“同一系统的同一右侧面板”
- 不因 subset 不同而像换了另一个产品
- 领域特定区块可以有，但不能压过 shared details 主体

#### 重点检查项
- open actions 区块位置
- tags / color tag / favorite / rating 的相对顺序
- domain-specific info 区块（Book Info / Software Info / Game Status）的视觉轻重
- Re-find actions 的位置与命名
- 批量模式下右侧摘要占位是否仍与单项 details 同属一套系统

---

### 5.5 批量模式与普通浏览模式统一
当前 Recent + 四个主库页已经接入批量模式。  
本批次需要统一：

- 进入批量模式的入口
- 退出批量模式的方式
- 选中态与勾选态的可区分性
- 批量工具条的命名与层级
- 批量成功后的回显与提示

#### 目标
- 用户能明显感知自己进入了另一种交互模式
- 但不会觉得自己进入了另一套系统
- 不同页的批量模式操作感相近

---

## 6. 任务优先级

### P0：必须完成
- Header / description 统一
- loading / empty / no-result 统一
- CTA 命名与回流文案统一
- shared details 信息层级统一
- batch mode 进入 / 退出体验统一

### P1：可以在本批次内顺手完成
- 小范围视觉密度微调
- CTA 位置的轻量统一
- 某些 subset 页明显突兀文案修正

### P2：后置
- 更大范围视觉重设计
- 动效系统
- 品牌级视觉统一
- Dashboard 统计表达

---

## 7. 执行边界

### 7.1 允许的改动
- 前端页面文案
- 前端区块顺序
- 样式统一
- 小范围 CTA 重命名
- 小范围交互收口
- 必要的 query invalidation 或轻量前端逻辑修正（仅当统一体验需要）

### 7.2 尽量避免的改动
- 新 route
- 新 API
- 新数据模型
- 新 domain service
- 大范围后端 contract 调整

### 7.3 禁止项
- 新增新功能线
- 引入平台化能力
- 为体验统一之名重写现有主链
- 把 shared details 分叉成多个系统

---

## 8. 对 Codex 的实现要求

### 8.1 基本要求
- 优先复用现有组件与样式结构
- 不做大而全重构
- 先统一高频主路径，不追求把所有角落一次收完
- 保持最小改动面完成最大体验统一

### 8.2 代码组织要求
- 不新增笨重“体验统一中台”
- 能复用现有 feature / shared components 则优先复用
- 如需抽象，只允许薄抽象

### 8.3 验证要求
实现完成后必须输出：
1. 修改文件清单
2. 每个文件对应的体验统一任务点
3. 是否有接口改动（原则上应为无；如有必须解释）
4. 前端 / 后端检查或测试结果
5. 未完成项与风险点
6. 哪些体验差异仍保留，以及为何保留

---

## 9. 验收口径

本批次验收不以“新增了多少功能”为标准，而看：

### 9.1 产品感是否统一
用户切换不同页时，是否感觉仍在同一个产品里。

### 9.2 主路径是否更顺
用户能否更稳定地完成：
- 浏览
- 详情
- 打开
- 标记
- 再找回

### 9.3 文案与 CTA 是否更可预测
用户是否更容易理解动作含义，而不是依赖猜测。

### 9.4 details 是否更稳
右侧详情面板是否层级更清晰，不像若干 patchwork 拼出来的。

### 9.5 批量模式是否更可懂
批量模式与普通模式边界是否更明确。

---

## 10. 本批次完成后的下一步

本批次完成后，不继续加新功能，而进入：

1. 《测试版验证问题清单》收口
2. 外部验证准备
3. 对外测试版演示与反馈收集

---

## 11. 当前结论

测试版体验统一阶段的本质不是“再做一点功能”，而是：

> 让已经完成的功能，开始像一个统一、可信、可演示的产品。

因此，本执行稿要求的所有改动，都应以“统一体验、收边界、增强主路径可理解性”为第一原则。
